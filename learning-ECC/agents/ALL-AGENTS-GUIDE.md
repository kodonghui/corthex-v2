# ECC 전체 에이전트 가이드 (33명)

> **이 문서 사용법**
>
> 에이전트는 "전문가 직원"이라고 생각하시면 됩니다. 각각 특정 분야를 담당하는 전문가예요.
> 어떤 에이전트는 여러분이 직접 부르고, 어떤 에이전트는 상황에 따라 자동으로 나타납니다.
> 코드를 짤 때 실수를 잡아주는 편집자, 건물 설계를 해주는 건축가, 보안을 점검하는 경비원 등
> 다양한 역할이 있습니다. 아래에서 각 에이전트가 뭘 하는지 쉽게 설명드리겠습니다.

---

# 1. 코드 검수 전문가 (Reviewer 계열)

코드를 작성한 후 "이거 괜찮은지" 검사해주는 전문가들입니다. 글을 쓴 후 편집자가 오탈자와 논리 오류를 잡아주는 것과 같습니다.

---

## code-reviewer

**역할**: 만능 편집자입니다. 글을 쓴 후 오탈자, 논리 오류, 보안 위험을 잡아줍니다.

**언제 나타나나요?**
- 자동: 코드를 수정하거나 새로 작성할 때마다 자동으로 검토합니다.

**뭘 해주나요?**
코드 변경 사항을 전부 읽고 보안 위험(비밀번호 노출 등), 코드 품질(너무 길거나 복잡한 함수), 성능 문제(느린 패턴) 등을 찾아냅니다. 심각도별로 정리해서 "이건 반드시 고쳐야 합니다 / 이건 참고만 하세요" 형태로 보고합니다. 모든 프로그래밍 언어에 적용되지만, 특히 웹 개발에 강합니다.

---

## typescript-reviewer

**역할**: TypeScript/JavaScript 전용 편집자입니다. 웹 개발의 타입 안전성과 비동기 처리를 전문적으로 검토합니다.

**언제 나타나나요?**
- 자동: TypeScript나 JavaScript 파일이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
TypeScript 코드에서 타입 안전성(변수의 종류를 정확히 지정했는지), 비동기 처리(여러 작업을 동시에 할 때 문제 없는지), React 패턴(화면 구성 요소가 올바른지) 등을 검토합니다. PR을 올리기 전에 CI가 통과하는지도 확인하고, 통과 안 되면 먼저 고치라고 알려줍니다.

---

## python-reviewer

**역할**: Python 전용 편집자입니다. Python다운 코드인지, 보안은 괜찮은지 검토합니다.

**언제 나타나나요?**
- 자동: Python 파일(.py)이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
Python 코드가 업계 표준(PEP 8)을 따르는지, 타입 힌트가 제대로 되어 있는지, 보안 취약점은 없는지 검토합니다. Django, FastAPI, Flask 같은 웹 프레임워크별 특수 규칙도 알고 있어서 프레임워크에 맞는 조언을 해줍니다.

---

## go-reviewer

**역할**: Go 언어 전용 편집자입니다. Go만의 독특한 에러 처리와 동시 처리 패턴을 전문적으로 봅니다.

**언제 나타나나요?**
- 자동: Go 파일(.go)이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
Go 코드가 Go다운 방식으로 작성되었는지, 에러를 제대로 처리하는지(무시하면 안 됩니다), 여러 작업을 동시에 할 때 충돌이 없는지 검토합니다. goroutine 누수(작업이 끝나지 않고 쌓이는 것)나 deadlock(서로 기다리느라 멈추는 것) 같은 문제를 잡아냅니다.

---

## rust-reviewer

**역할**: Rust 언어 전용 편집자입니다. 메모리 안전성과 소유권 패턴을 전문적으로 검토합니다.

**언제 나타나나요?**
- 자동: Rust 파일(.rs)이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
Rust의 핵심인 소유권(ownership)과 수명(lifetime) 패턴이 올바른지, 불필요한 복사가 없는지, unsafe 코드가 정당한 이유와 함께 사용되었는지 검토합니다. 동시성 처리에서 blocking(멈추는 것)이나 deadlock 문제도 찾아줍니다.

---

## cpp-reviewer

**역할**: C++ 전용 편집자입니다. 메모리 안전성, 현대적 C++ 패턴, 동시 처리를 검토합니다.

**언제 나타나나요?**
- 자동: C++ 파일이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
메모리를 수동으로 관리하는 부분(new/delete)을 스마트 포인터로 바꾸라고 알려주고, 버퍼 오버플로우(데이터가 넘치는 것), 사용 후 해제(이미 버린 것을 다시 쓰는 것) 같은 위험한 패턴을 잡아냅니다. 현대적인 C++ 방식을 권장합니다.

---

## java-reviewer

**역할**: Java와 Spring Boot 전용 편집자입니다. 계층 구조와 JPA 패턴을 전문적으로 검토합니다.

**언제 나타나나요?**
- 자동: Java 파일이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
Spring Boot의 계층 구조(Controller-Service-Repository)가 제대로 지켜지는지, JPA(데이터베이스 연결)에서 N+1 문제(불필요하게 많은 쿼리를 보내는 것)가 없는지, 트랜잭션이 올바른 위치에 있는지 검토합니다. 보안, 에러 처리, 테스트 패턴도 봅니다.

---

## kotlin-reviewer

**역할**: Kotlin과 Android/KMP 전용 편집자입니다. Coroutine 안전성과 Compose 패턴을 전문적으로 봅니다.

**언제 나타나나요?**
- 자동: Kotlin 파일이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
Kotlin 코드가 Kotlin다운지, Coroutine(비동기 작업)에서 취소가 제대로 전파되는지, Jetpack Compose(안드로이드 UI)에서 불필요한 리컴포지션(화면 다시 그리기)이 없는지 검토합니다. 클린 아키텍처 경계도 확인합니다.

---

## flutter-reviewer

**역할**: Flutter와 Dart 전용 편집자입니다. 위젯 패턴, 상태 관리, 접근성을 검토합니다.

**언제 나타나나요?**
- 자동: Dart 파일이 변경될 때 자동으로 나타납니다.

**뭘 해주나요?**
Flutter 위젯이 효율적으로 구성되었는지, 상태 관리(BLoC, Riverpod 등 프로젝트가 쓰는 방식에 맞춰)가 올바른지, 리소스가 제대로 해제되는지 검토합니다. 접근성(시각 장애인도 사용 가능한지)과 반응형 레이아웃도 확인합니다.

---

## database-reviewer

**역할**: 데이터베이스 전문 검토관입니다. 느린 쿼리를 찾아내고 데이터 구조를 최적화합니다.

**언제 나타나나요?**
- 수동: SQL을 작성하거나, 데이터베이스 구조를 변경하거나, 성능 문제가 있을 때 부릅니다.

**뭘 해주나요?**
데이터베이스 쿼리가 효율적인지(인덱스가 제대로 있는지), 테이블 구조가 올바른지(데이터 타입이 맞는지), 보안이 적용되어 있는지(Row Level Security) 검토합니다. N+1 쿼리, OFFSET 페이지네이션 같은 성능 저하 패턴도 잡아냅니다.

---

# 2. 에러 해결사 (Build Resolver 계열)

빌드(코드를 실행 가능한 프로그램으로 만드는 과정)가 실패했을 때 최소한의 수정으로 빠르게 고쳐주는 전문가들입니다. 소방관처럼 불이 나면 빨리 끄는 게 목표입니다.

---

## build-error-resolver

**역할**: 웹 개발 소방관입니다. TypeScript 빌드가 실패하면 최소한의 수정으로 빠르게 고쳐줍니다.

**언제 나타나나요?**
- 자동: 빌드가 실패하거나 타입 에러가 발생할 때 자동으로 나타납니다.

**뭘 해주나요?**
빌드 에러를 분석하고 가장 적은 코드 변경으로 고칩니다. 타입 에러, 모듈 가져오기 문제, 설정 파일 오류 등을 처리합니다. 코드를 재설계하거나 개선하지 않고, 오직 에러만 고칩니다. "고장 난 것만 고치고 나머지는 건드리지 않는다"가 원칙입니다.

---

## cpp-build-resolver

**역할**: C++ 빌드 소방관입니다. CMake 설정 오류, 링커 에러, 템플릿 에러를 고칩니다.

**언제 나타나나요?**
- 자동: C++ 빌드가 실패할 때 자동으로 나타납니다.

**뭘 해주나요?**
C++ 컴파일 에러, CMake 설정 문제, 링커 에러(함수를 찾을 수 없다는 에러) 등을 최소한의 수정으로 고칩니다. "undefined reference"(정의 안 된 참조)나 "cannot find module"(모듈을 못 찾음) 같은 흔한 에러의 해결 패턴을 알고 있습니다.

---

## go-build-resolver

**역할**: Go 빌드 소방관입니다. Go 빌드 에러와 모듈 의존성 문제를 고칩니다.

**언제 나타나나요?**
- 자동: Go 빌드가 실패할 때 자동으로 나타납니다.

**뭘 해주나요?**
`go build` 에러, `go vet` 경고, 모듈 의존성 충돌 등을 최소한의 수정으로 고칩니다. 타입 불일치, 인터페이스 미구현, 순환 의존성 같은 Go 특유의 문제를 처리합니다.

---

## java-build-resolver

**역할**: Java/Maven/Gradle 빌드 소방관입니다. Java 컴파일 에러와 의존성 문제를 고칩니다.

**언제 나타나나요?**
- 자동: Java 빌드가 실패할 때 자동으로 나타납니다.

**뭘 해주나요?**
Java 컴파일 에러, Maven/Gradle 설정 오류, 의존성 충돌, Lombok/MapStruct 같은 어노테이션 프로세서 문제를 고칩니다. Spring Boot 관련 에러(빈 충돌, 순환 의존성)도 처리합니다.

---

## kotlin-build-resolver

**역할**: Kotlin/Gradle 빌드 소방관입니다. Kotlin 컴파일러 에러와 Gradle 설정 문제를 고칩니다.

**언제 나타나나요?**
- 자동: Kotlin 빌드가 실패할 때 자동으로 나타납니다.

**뭘 해주나요?**
Kotlin 컴파일 에러(스마트 캐스트 불가, 소거된 when 분기 등), Gradle 의존성 충돌, detekt/ktlint 위반 등을 최소한의 수정으로 고칩니다.

---

## rust-build-resolver

**역할**: Rust 빌드 소방관입니다. 소유권/수명 에러와 Cargo 의존성 문제를 고칩니다.

**언제 나타나나요?**
- 자동: Rust 빌드가 실패할 때 자동으로 나타납니다.

**뭘 해주나요?**
Rust의 가장 어려운 부분인 소유권 검사기(borrow checker) 에러를 고쳐줍니다. "cannot borrow as mutable"(이미 빌린 것을 수정할 수 없다), "does not live long enough"(충분히 오래 살지 않는다) 같은 에러를 최소한의 변경으로 해결합니다.

---

## pytorch-build-resolver

**역할**: PyTorch/AI 소방관입니다. 텐서 형태 불일치, GPU 에러, 학습 실패를 고칩니다.

**언제 나타나나요?**
- 자동: PyTorch 학습이나 추론이 실패할 때 자동으로 나타납니다.

**뭘 해주나요?**
AI 학습에서 흔한 "텐서 크기 불일치"(행렬 곱이 안 되는 것), "CUDA out of memory"(GPU 메모리 부족), "같은 장치에 있어야 한다"(CPU와 GPU에 데이터가 분산된 것) 같은 에러를 고칩니다. 그래디언트(학습 방향)가 제대로 흐르는지도 확인합니다.

# 3. 테스트 전문가 (E2E Runner, TDD Guide, Socrates 계열)

코드가 실제로 잘 작동하는지 테스트하는 전문가들입니다. 자동차를 만든 후 시험 주행을 하는 테스트 드라이버와 같습니다.

---

## tdd-guide

**역할**: 테스트 코치입니다. "먼저 시험 문제를 만들고, 그 다음에 답을 쓰라"는 방식으로 개발을 이끕니다.

**언제 나타나나요?**
- 수동: 새 기능을 만들거나, 버그를 고치거나, 코드를 리팩토링할 때 부릅니다.

**뭘 해주나요?**
TDD(테스트 주도 개발) 방식을 가르쳐줍니다. 빨간불-초록불-리팩토링 순서(테스트 실패 -> 최소 구현 -> 개선)를 따르게 하고, 80% 이상의 테스트 커버리지를 목표로 합니다. null 값, 빈 배열, 유니코드, 동시 접근 같은 극단적인 경우도 반드시 테스트하게 합니다.

---

## e2e-runner

**역할**: 실제 사용자 시뮬레이터입니다. 진짜 브라우저에서 클릭, 입력, 이동을 하며 앱이 잘 작동하는지 확인합니다.

**언제 나타나나요?**
- 수동: E2E(처음부터 끝까지) 테스트를 만들거나 실행할 때 부릅니다.

**뭘 해주나요?**
Playwright(브라우저 자동화 도구)를 사용해서 실제 사용자가 하는 것처럼 로그인, 데이터 입력, 페이지 이동 등을 자동으로 수행합니다. 스크린샷, 영상, 추적 파일을 남기고, 불안정한 테스트는 격리시킵니다. 로그인 -> 핵심 기능 -> 결제 같은 중요한 사용자 여정을 우선적으로 테스트합니다.

---

## socrates-functional (Agent A)

**역할**: 기능 검증관입니다. "이 버튼을 누르면 뭐가 나와야 하는가?"를 하나씩 확인합니다.

**언제 나타나나요?**
- 수동: 코드 리뷰 파이프라인에서 소크라테스 QA 팀으로 배치될 때 사용됩니다.

**뭘 해주나요?**
Playwright MCP를 사용해서 실제 브라우저에서 CRUD(생성-읽기-수정-삭제) 사이클, 폼 제출, 네비게이션, 데이터 보존을 테스트합니다. 모든 테스트에서 "기대하는 결과를 먼저 말하고, 그 다음에 확인한다"는 소크라테스 방식을 따릅니다. 버그를 발견하면 스크린샷과 함께 보고합니다.

---

## socrates-visual (Agent B)

**역할**: 디자인 감시관입니다. 화면이 보기 좋은지, 디자인 규칙을 따르는지 확인합니다.

**언제 나타나나요?**
- 수동: 코드 리뷰 파이프라인에서 소크라테스 QA 팀으로 배치될 때 사용됩니다.

**뭘 해주나요?**
데스크톱(1280x800)과 모바일(390x844) 두 가지 화면 크기에서 레이아웃이 올바른지, 디자인 토큰(색상, 글꼴)이 일관되는지, 아이콘이 제대로 표시되는지 확인합니다. 페이지별로 스크린샷을 찍어서 비교하고, 디자인에서 벗어난 부분을 보고합니다.

---

## socrates-edge (Agent C)

**역할**: 극한 상황 테스터입니다. 일반 테스트에서 놓치는 보안 취약점과 극단적인 사용 사례를 찾습니다.

**언제 나타나나요?**
- 수동: 코드 리뷰 파이프라인에서 소크라테스 QA 팀으로 배치될 때 사용됩니다.

**뭘 해주나요?**
일반 사용자가 하지 않는 행동을 시뮬레이션합니다. 예를 들어: 권한 없는 사용자가 관리자 페이지에 접근하면 어떻게 되는지(보안 테스트), 500자를 이름 필드에 넣으면 어떻게 되는지(입력 경계), 버튼을 빠르게 두 번 누르면 어떻게 되는지(빠른 동작), 데이터가 없는 페이지가 빈 화면인지 안내 메시지인지(빈 상태) 등을 테스트합니다.

---

## socrates-regression (Agent D)

**역할**: 영향도 추적관입니다. 코드를 고친 후 다른 곳이 깨지지 않았는지 확인합니다.

**언제 나타나나요?**
- 수동: 코드 리뷰 파이프라인에서 소크라테스 QA 팀으로 배치될 때 사용됩니다.

**뭘 해주나요?**
사이드바의 모든 메뉴를 하나씩 클릭해서 전체 앱이 정상인지 먼저 확인합니다(사이드바 스윕). 그 다음, 변경된 컴포넌트를 사용하는 다른 페이지들이 영향을 받지 않았는지 확인합니다. 테마 일관성(모든 페이지의 색상이 같은지)과 세션 유지(로그인이 풀리지 않는지)도 검증합니다.

---

# 4. 기획/설계 (Architect, Planner)

새로운 기능을 만들기 전에 설계도를 그려주는 전문가들입니다. 집을 짓기 전에 건축 설계를 하는 것과 같습니다.

---

## architect

**역할**: 건축가입니다. 새로운 기능을 만들 때 전체 시스템 구조를 설계하고, 장단점을 분석합니다.

**언제 나타나나요?**
- 수동: 새로운 기능을 기획하거나, 큰 리팩토링을 하거나, 아키텍처 결정이 필요할 때 부릅니다.

**뭘 해주나요?**
현재 시스템 구조를 분석하고, 새 기능에 필요한 컴포넌트, 데이터 흐름, API 계약을 설계합니다. 각 결정에 대해 장점/단점/대안을 문서화하고, 확장성(사용자가 10배 늘어나면?)과 보안도 고려합니다. ADR(Architecture Decision Record)이라는 형식으로 결정 사항을 기록합니다.

---

## planner

**역할**: 프로젝트 매니저입니다. 큰 작업을 작은 단계로 나누고 실행 순서를 정합니다.

**언제 나타나나요?**
- 수동: 복잡한 기능 구현, 큰 리팩토링, 요구사항이 불명확한 작업에서 부릅니다.

**뭘 해주나요?**
요구사항을 분석하고 단계별 구현 계획을 만듭니다. 각 단계에 정확한 파일 경로, 의존성, 위험도, 예상 복잡도를 포함시킵니다. "Phase 1: 최소 동작 -> Phase 2: 핵심 기능 -> Phase 3: 예외 처리 -> Phase 4: 최적화" 순서로 독립적으로 배포 가능한 단위로 나눕니다. 사용자가 "진행해"라고 할 때까지 코드를 건드리지 않습니다.

---

# 5. 보안 (Security Reviewer)

---

## security-reviewer

**역할**: 보안 경비원입니다. 해커가 들어올 수 있는 문을 찾아서 잠급니다.

**언제 나타나나요?**
- 자동: 인증, API 엔드포인트, 사용자 입력 처리, 결제 코드를 작성할 때 자동으로 나타납니다.

**뭘 해주나요?**
OWASP Top 10(가장 흔한 보안 취약점 목록)을 기준으로 코드를 점검합니다. 코드에 비밀번호가 하드코딩되어 있는지, SQL 인젝션(데이터베이스를 조작하는 공격)이 가능한지, XSS(웹페이지에 악성 코드를 심는 공격)가 가능한지, 인증 없이 접근 가능한 페이지가 있는지 등을 확인합니다. 심각한 취약점을 발견하면 즉시 보고하고 해결 방법을 알려줍니다.

---

# 6. 운영/관리

시스템의 원활한 운영과 관리를 담당하는 전문가들입니다.

---

## chief-of-staff

**역할**: 비서실장입니다. 이메일, Slack, LINE, Messenger 등 모든 소통 채널을 통합 관리합니다.

**언제 나타나나요?**
- 수동: 여러 채널의 메시지를 한 번에 정리하거나, 답장을 작성해야 할 때 부릅니다.

**뭘 해주나요?**
모든 채널의 메시지를 4단계로 분류합니다: (1) 자동 보관할 것(봇 알림), (2) 참고만 할 것(CC 메일), (3) 일정 관련(미팅 링크), (4) 답장이 필요한 것(질문). 답장이 필요한 메시지에는 초안을 작성하고, 일정을 확인해서 빈 시간을 제안하고, 보낸 후에는 캘린더와 할 일 목록까지 업데이트합니다.

---

## harness-optimizer

**역할**: 튜닝 전문가입니다. AI 에이전트 시스템 자체의 설정을 최적화합니다.

**언제 나타나나요?**
- 수동: 에이전트 시스템의 성능을 개선하고 싶을 때 부릅니다.

**뭘 해주나요?**
에이전트 시스템(harness)의 현재 상태를 진단하고, hooks(자동 실행 규칙), 평가 기준, 모델 라우팅, 컨텍스트 관리 등에서 개선점 3가지를 찾습니다. 작고 되돌릴 수 있는 변경만 제안하고, 변경 전후의 성능 차이를 측정해서 보고합니다.

---

## loop-operator

**역할**: 자동 운전 감시관입니다. AI가 자율적으로 반복 작업을 할 때 안전하게 관리합니다.

**언제 나타나나요?**
- 수동: 자율 반복 작업(autonomous loop)을 시작하거나 모니터링할 때 부릅니다.

**뭘 해주나요?**
자율 반복 작업의 진행 상황을 추적하고, 정지 조건을 설정합니다. 2회 연속 진전이 없거나, 같은 에러가 반복되거나, 비용이 예산을 초과하거나, 머지 충돌이 발생하면 자동으로 에스컬레이션(상위 보고)합니다. 품질 게이트가 활성화되어 있는지, 롤백 경로가 있는지도 확인합니다.

---

## refactor-cleaner

**역할**: 청소부입니다. 사용하지 않는 코드, 중복 코드, 불필요한 의존성을 찾아서 제거합니다.

**언제 나타나나요?**
- 수동: 코드 정리가 필요할 때 부릅니다.

**뭘 해주나요?**
분석 도구(knip, depcheck, ts-prune)를 사용해서 사용하지 않는 파일, 함수, 패키지를 찾아냅니다. 안전한 것부터 시작해서 한 번에 한 종류씩 제거하고, 매번 테스트를 돌려서 문제가 없는지 확인합니다. "확실하지 않으면 제거하지 않는다"가 원칙입니다.

---

## doc-updater

**역할**: 기록관입니다. 코드가 변경되면 문서도 같이 업데이트합니다.

**언제 나타나나요?**
- 수동: 문서 업데이트나 코드맵 생성이 필요할 때 부릅니다.

**뭘 해주나요?**
코드베이스 구조를 분석해서 아키텍처 맵(codemap)을 생성합니다. 프론트엔드, 백엔드, 데이터베이스, 외부 서비스 연동별로 정리하고, README와 가이드 문서를 코드에 맞게 업데이트합니다. "코드와 일치하지 않는 문서는 없는 것보다 나쁘다"가 원칙입니다.

---

## docs-lookup

**역할**: 사서입니다. 라이브러리나 프레임워크의 최신 공식 문서를 찾아줍니다.

**언제 나타나나요?**
- 수동: 라이브러리 사용법이 궁금하거나, API 사용 예시가 필요할 때 부릅니다.

**뭘 해주나요?**
Context7 MCP를 사용해서 라이브러리의 최신 공식 문서를 검색하고, 코드 예시와 함께 답변합니다. 학습 데이터가 아닌 실제 최신 문서를 기반으로 답변하기 때문에 오래된 정보를 알려줄 위험이 적습니다.

---

## synthesis-master

**역할**: 오케스트라 지휘자입니다. UI/UX 관련 모든 전문가를 조율해서 완성도 높은 결과물을 만듭니다.

**언제 나타나나요?**
- 수동: 종합적인 UI/UX 작업이 필요할 때 부릅니다.

**뭘 해주나요?**
7가지 기둥(의미, 아름다움, 접근성, 보안, 성능, 테스트, 문서화)을 모두 갖춘 프로덕션 레디 UI를 만들기 위해 전문가들을 조율합니다. 접근성 vs 아름다움에서는 항상 접근성이 우선, 보안 vs 편의성에서는 항상 보안이 우선이라는 원칙을 지킵니다. "7가지 중 하나라도 빠지면 출시하지 않는다"가 원칙입니다.
