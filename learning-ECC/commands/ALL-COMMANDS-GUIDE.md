# ECC 전체 명령어 가이드 (114개)

> **이 문서 사용법**
>
> 명령어는 "버튼"이라고 생각하시면 됩니다. Claude Code에서 `/명령어이름`을 입력하면 해당 기능이 실행됩니다.
> 예를 들어 `/save-session`을 입력하면 현재 작업 상태가 저장되고,
> `/verify`를 입력하면 코드에 문제가 없는지 검사합니다.
> 매일 쓸 것, 가끔 쓸 것, 특수 상황에만 쓸 것으로 나누어 설명드리겠습니다.

---

# 1. 매일 쓸 명령어

가장 자주 사용하게 될 핵심 명령어들입니다.

---

## /save-session

**한 줄 요약**: 현재 작업 상태를 파일로 저장합니다. 다음에 이어서 할 수 있도록.

**사용 예시**: "오늘 작업 끝! 내일 이어서 하려면" `/save-session` 입력. 뭘 했는지, 뭐가 성공했는지, 뭐가 실패했는지, 다음에 할 일이 뭔지 전부 기록됩니다.

---

## /resume-session

**한 줄 요약**: 이전에 저장한 작업을 불러와서 이어서 합니다.

**사용 예시**: "어제 하던 작업 이어서 하자" `/resume-session` 입력. 마지막으로 저장한 세션 파일을 읽고 어디까지 했는지 브리핑해줍니다.

---

## /verify

**한 줄 요약**: 코드에 문제가 없는지 전체 점검합니다. 건강검진과 같습니다.

**사용 예시**: "PR 올리기 전에 괜찮은지 확인하자" `/verify` 또는 `/verify pre-pr` 입력. 빌드, 타입, 린트, 테스트, 보안 전부 검사하고 "PR 준비 완료: 예/아니오"로 알려줍니다.

---

## /checkpoint

**한 줄 요약**: 현재 코드 상태에 "표시"를 남겨두고, 나중에 비교할 수 있습니다.

**사용 예시**: "리팩토링 시작 전에 현재 상태 저장하자" `/checkpoint create "리팩토링-전"` 입력. 나중에 `/checkpoint verify "리팩토링-전"`으로 뭐가 바뀌었는지 비교합니다.

---

## /learn

**한 줄 요약**: 이번 세션에서 배운 유용한 패턴을 기록으로 남깁니다.

**사용 예시**: "이 에러 해결 방법 다음에도 쓸 거 같은데" `/learn` 입력. 어떤 문제를 어떻게 해결했는지를 스킬 파일로 저장해서 다음에 같은 상황에서 자동으로 활용됩니다.

---

## /learn-eval

**한 줄 요약**: `/learn`의 고급 버전. 패턴을 저장하기 전에 품질 평가를 합니다.

**사용 예시**: "이 패턴 진짜 저장할 가치가 있을까?" `/learn-eval` 입력. 기존 스킬과 중복되는지, 재사용 가능한지 체크하고 "저장/개선 후 저장/기존에 합치기/버리기" 중 판정합니다.

---

## /plan

**한 줄 요약**: 코드를 쓰기 전에 먼저 계획을 세웁니다. 사용자 확인 전까지 코드를 건드리지 않습니다.

**사용 예시**: "실시간 알림 기능을 추가하고 싶어" `/plan 실시간 알림 기능 추가` 입력. 요구사항 분석, 단계 분할, 위험도 평가를 한 후 "이 계획대로 할까요?"라고 물어봅니다.

---

## /aside

**한 줄 요약**: 작업 중에 잠깐 질문하고 바로 원래 작업으로 돌아옵니다.

**사용 예시**: "이 함수가 뭐 하는 건지 궁금한데 작업 흐름을 끊고 싶지 않아" `/aside 이 함수가 뭐 하는 거야?` 입력. 질문에 답한 후 자동으로 하던 작업을 이어갑니다.

---

## /discuss-mode

**한 줄 요약**: 코드 실행 없이 논의만 합니다. "어떻게 할까?"를 같이 고민합니다.

**사용 예시**: "A 방법이랑 B 방법 중 뭐가 나을까?" `/discuss-mode` 입력 또는 그냥 "논의해보자"라고 말하면 됩니다. 선택지를 비교하고 의견을 내고 추천까지 해줍니다. 코드는 절대 건드리지 않습니다.

---

## /docs

**한 줄 요약**: 라이브러리의 최신 공식 문서를 찾아줍니다.

**사용 예시**: "Next.js 미들웨어 설정법이 궁금해" `/docs Next.js middleware 설정` 입력. Context7 MCP를 통해 최신 공식 문서에서 답과 코드 예시를 찾아줍니다.

---

## /sessions

**한 줄 요약**: 저장된 세션 목록을 보거나, 별명을 붙이거나, 상세 정보를 확인합니다.

**사용 예시**: "이번 주에 저장한 세션들 보고 싶어" `/sessions list` 입력. `/sessions alias abc123 "결제-기능"` 같이 별명을 붙일 수도 있습니다.

---

## /context-budget

**한 줄 요약**: AI가 사용하는 컨텍스트 창(기억 용량)이 얼마나 차 있는지 분석합니다.

**사용 예시**: "요즘 응답이 느린 것 같은데" `/context-budget` 입력. 에이전트, 스킬, MCP 서버 등이 얼마나 메모리를 차지하는지 분석하고 최적화 방법을 알려줍니다.

---

## /prompt-optimize

**한 줄 요약**: 프롬프트(명령문)를 분석해서 최적화된 버전을 만들어줍니다.

**사용 예시**: "이 명령을 더 효과적으로 쓰고 싶어" `/prompt-optimize 실시간 채팅 기능 만들어줘` 입력. 프롬프트를 분석하고, 어떤 스킬/에이전트를 쓰면 좋을지 추천하고, 최적화된 프롬프트를 제안합니다. 실행은 하지 않습니다.

# 2. 코딩 도구

코드를 작성하고, 테스트하고, 검토하는 도구들입니다.

---

## /tdd

**한 줄 요약**: 테스트를 먼저 쓰고 코드를 나중에 쓰는 TDD 방식으로 개발합니다.

**사용 예시**: "이 기능을 TDD로 만들자" `/tdd` 입력. 테스트 먼저 작성 -> 실행해서 실패 확인 -> 최소 코드 작성 -> 테스트 통과 확인 -> 리팩토링 순서로 진행합니다.

---

## /code-review

**한 줄 요약**: 커밋 안 된 변경사항을 보안과 품질 관점에서 검토합니다.

**사용 예시**: "방금 작성한 코드 괜찮은지 봐줘" `/code-review` 입력. git diff를 보고 보안 위험, 코드 품질, 성능 문제를 보고합니다.

---

## /build-fix

**한 줄 요약**: 빌드 에러를 점진적으로 고칩니다.

**사용 예시**: "빌드가 깨졌어" `/build-fix` 입력. 자동으로 빌드 시스템(TypeScript, Go, Rust 등)을 감지하고, 에러를 하나씩 최소한의 수정으로 고칩니다.

---

## /e2e

**한 줄 요약**: Playwright로 E2E 테스트를 생성하고 실행합니다.

**사용 예시**: "로그인 -> 대시보드 -> 설정 흐름을 테스트하자" `/e2e` 입력. Playwright 테스트를 만들고, 실행하고, 스크린샷/영상/추적 파일을 저장합니다.

---

## /test-coverage

**한 줄 요약**: 테스트 커버리지를 분석하고 빠진 부분의 테스트를 생성합니다.

**사용 예시**: "테스트가 부족한 부분이 어딘지 알고 싶어" `/test-coverage` 입력. 현재 커버리지를 분석하고 80% 이상이 되도록 부족한 테스트를 생성합니다.

---

## /refactor-clean

**한 줄 요약**: 사용하지 않는 코드를 안전하게 찾아서 제거합니다.

**사용 예시**: "코드가 지저분해. 안 쓰는 거 정리하자" `/refactor-clean` 입력. 죽은 코드를 찾아서 한 번에 한 종류씩 제거하며, 매번 테스트로 안전을 확인합니다.

---

## /eval

**한 줄 요약**: 평가 기반 개발 워크플로우를 관리합니다.

**사용 예시**: "이 기능의 품질을 수치로 측정하고 싶어" `/eval` 입력. 기능 평가와 회귀 평가를 정의하고, 구현 전후의 점수를 비교합니다.

---

## /quality-gate

**한 줄 요약**: 코드 품질 파이프라인을 수동으로 실행합니다.

**사용 예시**: "이 파일이 품질 기준을 통과하는지 확인하자" `/quality-gate src/` 입력. 포맷, 린트, 타입 체크를 순서대로 실행하고 고쳐야 할 목록을 알려줍니다.

---

## /model-route

**한 줄 요약**: 현재 작업에 가장 적합한 AI 모델을 추천합니다.

**사용 예시**: "이 작업에 Opus를 써야 할까, Sonnet으로 충분할까?" `/model-route 데이터베이스 스키마 설계` 입력. 작업 복잡도와 예산에 따라 haiku(단순)/sonnet(일반)/opus(복잡) 중 추천합니다.

---

## /harness-audit

**한 줄 요약**: 에이전트 시스템의 건강 상태를 점수로 평가합니다.

**사용 예시**: "우리 설정이 잘 되어 있는지 확인하자" `/harness-audit` 입력. hooks, 평가, 라우팅, 컨텍스트, 안전장치 등을 점검하고 점수표를 보여줍니다.

---

---

## /update-codemaps

---

## /setup-pm

# 4. BMAD 파이프라인

BMAD는 프로젝트 기획부터 개발까지의 전체 과정을 체계적으로 관리하는 시스템입니다. "기획서 작성 -> 요구사항 정리 -> 설계 -> 개발 -> 테스트 -> 검토"의 전체 흐름을 자동화합니다.

---

## /kdh-full-auto-pipeline

**한 줄 요약**: BMAD 기반 전체 자동화 파이프라인입니다. 기획부터 개발까지 전부 자동으로.

**사용 예시**: "새 기능을 처음부터 끝까지 만들자" `/kdh-full-auto-pipeline planning` (기획) 또는 `/kdh-full-auto-pipeline story-1` (스토리 개발) 입력. planning 모드에서는 기획서-PRD-아키텍처-UX-에픽을 파티 모드(3명의 크리틱이 검토)로 만들고, story 모드에서는 스토리 생성-개발-테스트-QA-코드리뷰까지 자동 실행합니다.

---

## /kdh-uxui-redesign-full-auto-pipeline

**한 줄 요약**: UXUI 리디자인 전용 자동화 파이프라인입니다. 디자인 시스템부터 구현까지.

**사용 예시**: "앱 디자인을 전면 개편하자" `/kdh-uxui-redesign-full-auto-pipeline` 입력. 6단계(디자인 시스템 DB -> Playwright 시각 리서치 -> Stitch MCP -> 파티 모드 검토)로 진행하며, 여러 테마를 지원하고 하드코딩 색상을 0으로 만듭니다.

---

## /kdh-code-review-full-auto

**한 줄 요약**: 전자동 코드 리뷰 파이프라인입니다. 8단계로 코드를 철저히 검토합니다.

**사용 예시**: "PR을 철저하게 리뷰하자" `/kdh-code-review-full-auto PR-URL` 입력. 정적 분석 -> 시각/E2E -> 리스크 분석 -> 3명 크리틱 파티 -> 판정 -> 자동 수정 -> 재검토 -> 최종 판정 순서로 진행합니다.

---

## /bmad-full-pipeline

**한 줄 요약**: BMAD 기본 파이프라인입니다 (v3). 기획이나 스토리 개발을 체계적으로 진행합니다.

**사용 예시**: "/bmad-full-pipeline planning" (기획 모드) 또는 "/bmad-full-pipeline story-3" (스토리 개발 모드)

---

## /bmad-party-mode

**한 줄 요약**: 설치된 모든 BMAD 에이전트들이 토론을 벌입니다. 다양한 관점에서 검토.

**사용 예시**: "이 PRD를 여러 전문가 관점에서 검토하자" `/bmad-party-mode` 입력. PM, 개발자, QA, UX 디자이너 등이 각자 관점에서 의견을 냅니다.

---

## /bmad-help

**한 줄 요약**: 현재 상태를 분석하고 다음에 뭘 해야 하는지 조언합니다.

**사용 예시**: "지금 뭘 해야 할지 모르겠어" `/bmad-help` 입력.

**한 줄 요약**: 창의적 아이디어 발상 세션을 진행합니다.

**사용 예시**: "새로운 기능 아이디어를 모으자" `/bmad-brainstorming` 입력.
