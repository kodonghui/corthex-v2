이 문서 사용법

ECC(Everything Claude Code) 스킬은 Claude Code에 설치하는 앱이라고 생각하시면 됩니다.

스마트폰에 카메라 앱, 계산기 앱, 지도 앱이 있듯이, Claude Code에도 “코드 리뷰 스킬”, “보안 검사 스킬”, “디자인 스킬” 같은 것들이 있습니다. 각 스킬은 Claude가 특정 작업을 더 잘하도록 전문 지식을 제공합니다.

스킬의 종류는 크게 세 가지입니다:

  1. 자동 스킬 — 별도로 실행하지 않아도 알아서 작동합니다 (백그라운드 앱처럼)
  2. 명령어 스킬 — /명령어를 입력해서 직접 실행합니다 (앱을 터치해서 여는 것처럼)
  3. 참고 스킬 — Claude가 관련 작업을 할 때 자동으로 참조합니다 (사전처럼)

총 142개 스킬이 11개 분류로 나뉘어 있습니다. 아래에서 하나하나 쉽게 설명해 드리겠습니다.

1. 자동으로 돌아가는 것 (자율 학습/관리 시스템)

이 스킬들은 사람이 신경 쓰지 않아도 Claude가 스스로 학습하고 관리하는 시스템입니다. 마치 스마트폰이 밤에 자동으로 업데이트되는 것과 같습니다.

dream

한 줄 요약: Claude가 “잠자면서” 기억을 정리하는 스킬입니다.

비유: 사람이 자는 동안 뇌가 낮에 배운 것을 정리하듯이, Claude도 세션이 끝나면 배운 것을 정리해서 기억 파일에 저장합니다.

언제 쓰나요? - 24시간마다 자동으로 실행됩니다 - Claude가 작업하다가 배운 교정사항, 결정, 선호도 등을 정리할 때

어떻게 작동하나요? 세션이 끝나면 Claude가 대화 내용을 훑어보면서 “아, 이 사용자는 이런 걸 좋아하는구나”, “이 프로젝트에서는 이렇게 하면 안 되는구나” 같은 패턴을 찾아냅니다. 찾아낸 패턴은 MEMORY.md 같은 기억 파일에 자동 저장됩니다. 다음에 대화할 때 이 기억을 읽어서 더 잘 도와줄 수 있습니다.

사용법: 자동으로 실행됨 (Stop hook)

continuous-learning

한 줄 요약: 작업 세션에서 반복되는 패턴을 자동으로 배워서 저장하는 스킬입니다.

비유: 요리사가 매일 요리하면서 자연스럽게 자신만의 노하우를 축적하는 것과 같습니다. Claude도 작업하면서 “이 에러는 이렇게 고치면 되더라”를 자동으로 기록합니다.

언제 쓰나요? - 세션이 끝날 때 자동으로 실행됩니다 - 충분히 긴 세션(메시지 10개 이상)일 때만 작동합니다

어떻게 작동하나요? 세션이 끝나면 Claude가 대화를 분석해서 에러 해결법, 사용자 교정사항, 우회 방법 등의 패턴을 감지합니다. 유용한 패턴은 learned 폴더에 스킬 형태로 저장되어, 나중에 비슷한 상황에서 자동으로 참고됩니다.

사용법: 자동으로 실행됨 (Stop hook)

continuous-learning-v2

한 줄 요약: continuous-learning의 개선판으로, “본능(instinct)” 단위로 더 세밀하게 학습하는 스킬입니다.

비유: v1이 “오늘 배운 것 일기쓰기”라면, v2는 “핵심 한 줄씩 포스트잇에 적어서 분류하는 것”입니다. 더 작은 단위로, 더 정확하게 배웁니다.

언제 쓰나요? - 매 도구 사용 전후에 자동으로 분석합니다 - 프로젝트별로 별도 저장되어, React 프로젝트에서 배운 것이 Python 프로젝트에 섞이지 않습니다

어떻게 작동하나요? Claude가 도구를 사용할 때마다 작은 분석 에이전트가 뒤에서 “이번 작업에서 배울 게 있나?” 확인합니다. 배울 것이 있으면 신뢰도 점수(0.3~0.9)와 함께 저장합니다. 여러 프로젝트에서 같은 패턴이 발견되면 전역 지식으로 승격됩니다.

사용법: 자동으로 실행됨 (PreToolUse/PostToolUse hook)

canary-watch

한 줄 요약: 배포 후 사이트가 잘 되는지 자동으로 감시하는 스킬입니다.

비유: 가게 주인이 CCTV로 가게를 감시하는 것처럼, 웹사이트에 문제가 없는지 자동으로 확인합니다.

언제 쓰나요? - 새 기능을 배포한 직후 - 중요한 코드 변경 후 문제가 없는지 확인하고 싶을 때 - 밤새 사이트를 감시하고 싶을 때

어떻게 작동하나요? 지정한 웹사이트 주소로 접속해서 페이지가 정상인지(200 응답), 에러가 없는지, 성능은 괜찮은지 체크합니다. 정해진 간격으로 반복 체크할 수도 있고, 스테이징 서버와 실서버를 비교할 수도 있습니다.

사용법: /canary-watch https://사이트주소

continuous-agent-loop

한 줄 요약: Claude가 사람 없이 혼자서 반복 작업을 계속하게 만드는 스킬입니다.

비유: 공장의 자동 조립 라인처럼, Claude가 정해진 작업을 사람 개입 없이 반복합니다.

언제 쓰나요? - 사람이 없어도 Claude가 계속 작업하게 하고 싶을 때 - 코드 작성 > 테스트 > 수정을 자동으로 반복하고 싶을 때 - 여러 에이전트가 동시에 작업하고 합치는 방식이 필요할 때

어떻게 작동하나요? Claude가 “작업 > 검증 > 수정” 루프를 자동으로 반복합니다. 단순 순차 실행부터, 여러 에이전트가 동시에 병렬 작업하는 복잡한 방식까지 다양한 패턴을 지원합니다. 품질 검증 단계를 통과해야 다음으로 넘어가는 안전장치가 있습니다.

사용법: 자동으로 실행됨 (autonomous-loops 패턴의 후속)

autonomous-loops

한 줄 요약: Claude가 자율적으로 반복 실행하는 다양한 패턴 모음입니다.

비유: 자동 세탁기의 다양한 코스(빨래, 헹굼, 탈수)처럼, 작업을 자동으로 반복하는 여러 방식이 담겨 있습니다.

언제 쓰나요? - 자동 개발 워크플로우를 만들 때 - 어떤 루프 방식이 내 작업에 맞는지 골라야 할 때

어떻게 작동하나요? 단순한 순차 파이프라인부터, RFC(설계서) 기반 멀티 에이전트 시스템까지 다양한 자동화 패턴을 제공합니다. 프로젝트 성격에 따라 적합한 패턴을 선택해서 사용합니다. continuous-agent-loop이 이 스킬의 후속 버전입니다.

사용법: 프로젝트 설정 시 참고

strategic-compact

한 줄 요약: 대화가 너무 길어지기 전에 적절한 타이밍에 기억을 정리하도록 알려주는 스킬입니다.

비유: 책상이 너무 어질러지기 전에 “지금 정리하면 좋겠다”고 알려주는 비서와 같습니다.

언제 쓰나요? - 긴 세션에서 Claude의 기억 공간이 부족해질 때 - 한 작업이 끝나고 다른 작업으로 넘어갈 때 - 자동으로 도구 사용 횟수(기본 50회)를 추적해서 알림을 줍니다

어떻게 작동하나요? Claude가 도구를 몇 번 사용했는지 추적합니다. 일정 횟수를 넘기면 “지금 기억을 정리하면 좋겠습니다”라고 제안합니다. 작업 단계가 바뀌는 논리적 경계에서 정리하면, 중요한 맥락을 잃지 않고 새 작업을 시작할 수 있습니다.

사용법: 자동으로 실행됨 (PreToolUse hook)

dmux-workflows

한 줄 요약: 여러 AI 에이전트를 동시에 실행해서 병렬로 작업시키는 스킬입니다.

비유: 요리사 여러 명이 동시에 다른 요리를 만들고, 마지막에 한 식탁에 차리는 것처럼, 여러 Claude를 동시에 작업시키고 결과를 합칩니다.

언제 쓰나요? - 여러 작업을 동시에 진행하고 싶을 때 - “이것도 하고 저것도 해야 하는데” 할 때 분담시키기 - 복잡한 작업을 나눠서 병렬 처리하고 싶을 때

어떻게 작동하나요? dmux라는 도구를 사용해서 터미널 창을 여러 개 열고, 각 창에서 별도의 Claude가 작업합니다. 한 창에서는 인증 기능을, 다른 창에서는 테스트를, 또 다른 창에서는 문서를 동시에 작성합니다. 결과를 합칠 때는 m 키를 눌러 메인 세션으로 모읍니다.

사용법: dmux 도구 설치 후 사용

claude-devfleet

한 줄 요약: 여러 Claude 에이전트를 함대처럼 동시에 운영하는 스킬입니다.

비유: 해군 함대가 각 배에 임무를 배정하고 지휘관이 전체를 조율하듯이, 여러 Claude에게 각각 다른 코딩 임무를 줍니다.

언제 쓰나요? - 큰 프로젝트를 여러 Claude에게 동시에 시키고 싶을 때 - “인증 기능은 A가, 테스트는 B가, 문서는 C가” 식으로 분담할 때

어떻게 작동하나요? 작업 계획을 세우면 각 임무가 독립된 작업 공간에서 별도 Claude에 의해 실행됩니다. 하나가 끝나면 자동으로 결과를 합치고, 의존성이 있는 다음 임무를 자동 시작합니다. 최종 보고서로 무엇이 변경되었는지 확인할 수 있습니다.

사용법: Claude DevFleet MCP 서버 설정 후 사용

enterprise-agent-ops

한 줄 요약: 장시간 돌아가는 AI 에이전트를 안전하게 관리하는 운영 스킬입니다.

비유: 24시간 공장을 관리하는 것처럼, 계속 돌아가는 AI 시스템의 시작/중지/모니터링/보안을 관리합니다.

언제 쓰나요? - AI 에이전트를 서버에서 계속 돌릴 때 - 에이전트의 성능, 비용, 에러를 추적할 때 - 문제가 생기면 안전하게 중지시키고 복구할 때

어떻게 작동하나요? 에이전트의 전체 생명주기를 관리합니다. 시작, 일시정지, 중지, 재시작부터, 로그와 지표 모니터링, 권한 관리, 비상 중지 기능까지 포함됩니다. 문제가 발생하면 새 배포를 멈추고, 원인을 파악하고, 최소한의 변경으로 복구합니다.

사용법: 운영 환경 설정 시 참고

learned

한 줄 요약: continuous-learning 스킬이 자동으로 배운 내용을 저장하는 폴더입니다.

비유: 자동으로 채워지는 노트북과 같습니다. Claude가 배울 때마다 여기에 메모가 추가됩니다.

언제 쓰나요? - 자동으로 채워집니다 (SKILL.md 파일은 없음)

어떻게 작동하나요? continuous-learning이나 continuous-learning-v2가 세션에서 유용한 패턴을 발견하면 이 폴더에 저장합니다. 저장된 내용은 이후 세션에서 자동으로 참조됩니다.

사용법: 자동으로 실행됨

configure-ecc

한 줄 요약: ECC 스킬과 규칙을 선택해서 설치하는 마법사(설치 도우미)입니다.

비유: 스마트폰에서 앱스토어를 열고 원하는 앱을 골라 설치하는 것과 같습니다.

언제 쓰나요? - 처음 ECC를 설치할 때 - 새로운 스킬을 추가로 설치하고 싶을 때 - 기존 설치가 제대로 되었는지 확인할 때

어떻게 작동하나요? 단계별로 질문하면서 어떤 스킬을 설치할지 고릅니다. 선택한 스킬을 지정된 폴더에 복사하고, 제대로 설치됐는지 확인합니다. 필요하면 설치된 스킬을 프로젝트에 맞게 최적화할 수도 있습니다.

사용법: “configure ecc” 또는 “install ecc”라고 말하기

context-budget

한 줄 요약: Claude의 기억 공간을 얼마나 쓰고 있는지 분석해주는 스킬입니다.

비유: 스마트폰에서 “저장공간 관리”를 열어서 어떤 앱이 용량을 많이 차지하는지 보는 것과 같습니다.

언제 쓰나요? - Claude가 느려지거나 답변 품질이 떨어질 때 - 스킬이나 에이전트를 많이 추가한 후 - 새 스킬을 추가할 여유가 있는지 확인할 때

어떻게 작동하나요? 설치된 모든 스킬, 에이전트, MCP 서버, 규칙 파일을 스캔해서 각각 얼마나 기억 공간을 차지하는지 계산합니다. 너무 큰 파일, 중복되는 내용, 불필요한 구성요소를 찾아내서 “이것을 줄이면 공간이 확보됩니다”라고 알려줍니다.

사용법: /context-budget

skill-stocktake

한 줄 요약: 설치된 스킬들의 품질을 검사하는 스킬입니다.

비유: 가게 주인이 재고를 점검하면서 “이건 유통기한 지났네, 이건 상태 좋네” 확인하는 것과 같습니다.

언제 쓰나요? - 스킬이 많아져서 정리가 필요할 때 - 스킬의 품질을 점검하고 싶을 때 - 새 스킬을 추가한 후 전체 상태를 확인할 때

어떻게 작동하나요? 모든 스킬 파일을 품질 체크리스트로 평가합니다. “빠른 스캔” 모드는 최근 변경된 스킬만, “전체 점검” 모드는 모든 스킬을 검사합니다. 각 스킬에 점수를 매기고, 개선이 필요한 부분을 알려줍니다.

사용법: /skill-stocktake 또는 /skill-stocktake full

skill-comply

한 줄 요약: 스킬이나 규칙이 실제로 잘 지켜지고 있는지 테스트하는 스킬입니다.

비유: 학교에서 “교칙을 잘 지키는지” 불시 점검하는 것과 같습니다. 규칙을 정해놨지만 실제로 따르고 있는지 확인합니다.

언제 쓰나요? - 새 규칙을 추가한 후 제대로 적용되는지 확인할 때 - “이 스킬이 실제로 작동하긴 하는 건가?” 의문이 들 때 - 품질 관리 차원에서 정기적으로 점검할 때

어떻게 작동하나요? 프로젝트의 설정 파일, 폴더 구조, 코드 패턴을 자동으로 스캔합니다. 어떤 기술을 쓰는지(React? Python?), 핵심 진입점이 어디인지, 어떤 컨벤션을 따르는지 분석합니다. 결과를 구조화된 온보딩 가이드로 만들어줍니다.

iterative-retrieval

비유: 도서관에서 처음엔 목록을 보고, 그다음 관련 서가로 가고, 거기서 또 관련 책을 찾는 것처럼, 단계별로 점점 정확한 정보를 찾아갑니다.

언제 쓰나요? - 여러 에이전트가 코드를 탐색해야 할 때 - 어떤 파일이 필요한지 미리 알 수 없을 때 - “전부 보내면 너무 많고, 안 보내면 부족한” 상황에서

어떻게 작동하나요? 4단계 루프로 작동합니다: 보내기 > 평가 > 부족하면 더 찾기 > 다시 평가. 처음에는 대략적인 정보를 보내고, 에이전트가 더 필요한 것을 요청하면 추가로 찾아줍니다. 이를 반복하면서 필요한 맥락을 점진적으로 완성합니다.

사용법: 멀티 에이전트 워크플로우에서 자동 적용

3. 코딩할 때 도움주는 것

코드를 작성하고, 리뷰하고, 구조를 잡는 데 도움을 주는 스킬들입니다.

coding-standards

한 줄 요약: 코드를 깔끔하고 일관되게 작성하는 규칙 모음입니다.

비유: 글쓰기에 맞춤법과 문법 규칙이 있듯이, 코드에도 “이렇게 쓰면 좋다”는 표준이 있습니다.

언제 쓰나요? - 새 프로젝트를 시작할 때 - 코드 품질을 확인할 때 - 일관된 코드 스타일을 유지하고 싶을 때

어떻게 작동하나요? 읽기 좋은 코드 작성법(변수명은 명확하게, 함수는 동사-명사 형태 등), DRY(반복하지 마라), KISS(단순하게 만들어라) 같은 원칙을 담고 있습니다. TypeScript, JavaScript, React 프로젝트에 특화된 가이드를 제공합니다.

사용법: 자동으로 참조됨 (코딩 관련 작업 시)

api-design

한 줄 요약: 서버와 통신하는 규격(API)을 잘 설계하는 방법을 담은 스킬입니다.

비유: 식당의 메뉴판을 잘 만드는 것과 같습니다. 손님(사용자)이 무엇을 주문(요청)할 수 있고, 어떤 형태로 받을 수 있는지 깔끔하게 정리합니다.

언제 쓰나요? - 새로운 API를 설계할 때 - 기존 API를 개선할 때 - 에러 처리, 페이지 나누기, 버전 관리가 필요할 때

어떻게 작동하나요? URL 구조, 응답 형태, 에러 코드, 페이지 처리, 필터링 등 API를 일관되고 사용하기 쉽게 만드는 규칙을 제공합니다. 개발자들이 많이 사용하는 REST 방식의 표준 패턴을 기반으로 합니다.

사용법: 자동으로 참조됨 (API 설계 시)

backend-patterns

한 줄 요약: 서버 쪽 프로그램을 잘 만드는 설계 패턴 모음입니다.

비유: 건물의 배관, 전기 설계도처럼, 눈에 보이지 않지만 중요한 서버 구조를 잘 잡는 방법입니다.

언제 쓰나요? - 서버 코드를 작성할 때 - 데이터 흐름 구조를 설계할 때 - 캐싱, 에러 처리, 인증 등을 구현할 때

어떻게 작동하나요? 코드를 층(Layer)별로 나누는 방법, 데이터 요청 최적화, 캐시 전략, 비동기 처리, 미들웨어 구현 등 서버 개발에 필요한 핵심 패턴을 제공합니다.

사용법: 자동으로 참조됨 (서버 개발 시)

frontend-patterns

한 줄 요약: 사용자가 보는 화면(프론트엔드)을 잘 만드는 패턴 모음입니다.

비유: 인테리어 디자인 원칙처럼, 사용자가 직접 보고 만지는 부분을 깔끔하고 빠르게 만드는 방법입니다.

언제 쓰나요? - React 컴포넌트를 만들 때 - 상태 관리나 데이터 가져오기를 구현할 때 - 성능 최적화가 필요할 때

어떻게 작동하나요? React 컴포넌트 조합, 상태 관리(useState, Zustand 등), 데이터 불러오기 패턴, 성능 최적화(메모이제이션, 코드 분할), 폼 처리, 접근성 등 프론트엔드의 핵심 패턴을 제공합니다.

사용법: 자동으로 참조됨 (프론트엔드 개발 시)

react-components

한 줄 요약: Stitch 디자인 도구의 결과물을 React 코드로 변환하는 스킬입니다.

비유: 건축가의 도면을 보고 실제 건물을 짓는 것처럼, 디자인 파일을 보고 실제 동작하는 코드를 만듭니다.

어떻게 작동하나요? Stitch MCP에서 디자인 데이터를 가져와 스크린샷과 HTML을 다운로드합니다. 디자인을 분석해서 독립적인 React 컴포넌트 파일들로 나누고, 이벤트 처리 로직은 별도 hook으로 분리합니다.

사용법: Stitch MCP 연동 후 사용

database-migrations

한 줄 요약: 데이터 저장소(데이터베이스)의 구조를 안전하게 변경하는 방법입니다.

비유: 건물을 리모델링할 때 벽을 하나 옮기려면 설계도를 바꾸고, 안전하게 작업하고, 원래대로 돌릴 수 있는 계획이 필요한 것과 같습니다.

언제 쓰나요? - 데이터 저장 구조를 변경할 때 - 새 테이블이나 컬럼을 추가할 때 - 서비스 중단 없이 구조를 바꿔야 할 때

어떻게 작동하나요? 모든 데이터 구조 변경은 “마이그레이션”이라는 기록으로 관리합니다. 직접 데이터를 수정하는 것이 아니라, “이런 변경을 하겠다”는 계획서를 작성하고 실행합니다. 문제가 생기면 되돌릴 수 있도록 합니다.

사용법: 자동으로 참조됨 (DB 변경 시)

deployment-patterns

한 줄 요약: 코드를 실제 서비스 환경에 올리는(배포하는) 방법 모음입니다.

비유: 제품을 공장에서 만들어 매장에 진열하는 과정과 같습니다. 새 버전을 안전하게 갈아 끼우는 다양한 전략이 있습니다.

언제 쓰나요? - 코드를 서버에 배포할 때 - CI/CD 파이프라인을 설정할 때 - 배포 전략(블루-그린, 카나리 등)을 결정할 때

어떻게 작동하나요? 단계적 배포(하나씩 교체), 블루-그린 배포(새 환경 준비 후 한번에 전환), 카나리 배포(소수에게 먼저 공개) 등의 전략을 제공합니다. 건강 체크, 롤백 계획, 환경별 설정 방법도 포함됩니다.

사용법: 자동으로 참조됨 (배포 관련 작업 시)

docker-patterns

한 줄 요약: 프로그램을 독립된 상자(컨테이너)에 넣어서 어디서든 돌릴 수 있게 하는 방법입니다.

비유: 이삿짐을 박스에 정리해서 포장하면 어디로든 옮길 수 있듯이, 프로그램도 컨테이너에 담으면 어떤 서버에서든 동일하게 실행됩니다.

언제 쓰나요? - 개발 환경을 컨테이너로 만들 때 - 여러 서비스를 동시에 구성할 때 - 컨테이너 보안이나 네트워크 설정이 필요할 때

어떻게 작동하나요? Docker Compose로 여러 서비스(앱, 데이터베이스, 캐시)를 한꺼번에 설정하는 방법, Dockerfile을 효율적으로 작성하는 방법, 네트워크와 저장소 설정 방법을 제공합니다.

사용법: 자동으로 참조됨 (Docker 관련 작업 시)

postgres-patterns

한 줄 요약: PostgreSQL 데이터베이스를 잘 사용하는 방법 모음입니다.

비유: 엑셀을 쓸 때 “이렇게 정리하면 검색이 빠르다”는 노하우와 같습니다.

언제 쓰나요? - 데이터베이스 구조를 설계할 때 - 느린 검색을 빠르게 만들 때 - 보안(Row Level Security) 설정이 필요할 때

어떻게 작동하나요? 데이터 조회 패턴별 최적의 인덱스 유형, 스키마 설계 원칙, 연결 풀링 설정 등을 제공합니다. 특히 검색 속도를 높이는 인덱스 설정 가이드가 핵심입니다.

사용법: 자동으로 참조됨 (PostgreSQL 작업 시)

content-hash-cache-pattern

한 줄 요약: 파일 처리 결과를 캐시(임시 저장)해서 같은 파일을 다시 처리하지 않는 패턴입니다.

비유: 한번 번역한 문서에 “번역 완료” 도장을 찍어놓고, 다음에 같은 문서가 오면 이미 번역된 결과를 바로 보여주는 것과 같습니다.

언제 쓰나요? - 같은 파일을 반복 처리하는 작업이 있을 때 - PDF 변환, 텍스트 추출 같은 시간이 오래 걸리는 작업에서 - 파일이 변경되면 자동으로 다시 처리해야 할 때

어떻게 작동하나요? 파일 내용의 고유 지문(SHA-256 해시)을 캐시 키로 사용합니다. 파일 이름이 바뀌어도 내용이 같으면 캐시를 재사용하고, 내용이 바뀌면 자동으로 무효화됩니다.

4. 테스트/품질 관리

언제 쓰나요? - 새 기능을 만들 때 - 버그를 고칠 때 - 코드를 정리(리팩토링)할 때

어떻게 작동하나요? 1. 먼저 “이 기능은 이래야 한다”는 테스트를 작성합니다 (아직 실패함). 2. 테스트를 통과시키는 최소한의 코드를 작성합니다. 3. 코드를 더 깔끔하게 정리합니다. 최소 80% 이상의 코드가 테스트로 검증되어야 합니다.

사용법: 자동으로 참조됨 (코드 작성 시)

verification-loop

한 줄 요약: 코드 변경 후 빌드, 타입 체크, 린트, 테스트를 순서대로 돌리는 검증 과정입니다.

비유: 비행기 이륙 전 체크리스트처럼, 코드를 배포하기 전에 모든 항목을 하나하나 확인합니다.

언제 쓰나요? - 기능 완성 후 최종 확인할 때 - Pull Request를 만들기 전에 - 리팩토링 후 아무것도 안 깨졌는지 확인할 때

어떻게 작동하나요? 빌드 > 타입 체크 > 린트(코드 스타일 검사) > 테스트 > 보안 검사 순서로 진행됩니다. 앞 단계가 실패하면 멈추고 고친 후 다시 시작합니다. 모든 단계를 통과해야 “완료”입니다.

사용법: 자동으로 참조됨 (작업 완료 후)

e2e-testing

한 줄 요약: 실제 사용자처럼 화면을 클릭하고 입력하면서 테스트하는 방법입니다.

비유: 미스터리 쇼퍼가 실제 고객처럼 매장을 이용하면서 서비스를 점검하는 것과 같습니다.

언제 쓰나요? - 로그인 > 기능 사용 > 로그아웃 같은 전체 흐름을 테스트할 때 - 화면이 실제로 잘 동작하는지 확인할 때 - CI/CD 파이프라인에 자동 테스트를 추가할 때

어떻게 작동하나요? Playwright라는 도구를 사용해서 브라우저를 자동으로 조작합니다. 버튼 클릭, 텍스트 입력, 페이지 이동 등을 사람 대신 자동으로 수행하고, 결과가 예상대로인지 확인합니다.

사용법: 자동으로 참조됨 (E2E 테스트 작성 시)

ai-regression-testing

한 줄 요약: AI가 작성한 코드에서 놓치기 쉬운 버그를 잡는 전용 테스트 방법입니다.

비유: 같은 사람이 글 쓰고 교정하면 오타를 못 찾듯이, AI도 자기가 쓴 코드를 자기가 리뷰하면 같은 실수를 놓칩니다. 이 스킬은 그런 “사각지대”를 잡습니다.

언제 쓰나요? - AI가 코드를 수정한 후 - 같은 버그가 계속 재발할 때 - AI의 “자기 검토”가 신뢰가 안 될 때

어떻게 작동하나요? AI가 코드를 작성하면 자동화된 테스트를 즉시 실행합니다. AI의 리뷰에 의존하지 않고, 실제로 코드를 돌려서 결과를 확인합니다. 특히 샌드박스 모드와 실제 모드 같은 다중 경로를 모두 검사합니다.

사용법: /bug-check 또는 자동으로 활성화

언제 쓰나요? - 성능 기준치를 설정할 때 - “느려진 것 같은데” 실제로 확인할 때 - 코드 변경 전후 성능 비교할 때

어떻게 작동하나요? 페이지 로딩 속도(LCP, CLS 등 Core Web Vitals), API 응답 시간, 번들 크기 등을 자동으로 측정합니다. 목표치(예: 로딩 2.5초 이내)와 비교해서 통과/미달을 판정합니다.

사용법: /benchmark 사이트주소 또는 자동 활성화

eval-harness

한 줄 요약: Claude의 작업 완료 기준을 정의하고 자동으로 평가하는 프레임워크입니다.

비유: 시험 문제와 채점 기준표를 만들어놓고, Claude가 작업을 끝내면 자동으로 채점하는 시스템입니다.

언제 쓰나요? - “Claude가 이 작업을 정말 잘 해냈는지” 객관적으로 평가할 때 - 모델 버전이 바뀔 때 성능을 비교할 때 - 에이전트의 신뢰도를 측정할 때

어떻게 작동하나요? 작업 전에 성공 기준을 정의합니다. Claude가 작업을 끝내면 그 기준에 맞는지 자동으로 평가합니다. pass@k(k번 중 몇 번 성공하나) 같은 지표로 신뢰도를 측정합니다.

사용법: 평가 기반 개발(EDD) 워크플로우에서 사용

plankton-code-quality

한 줄 요약: 코드를 수정할 때마다 자동으로 스타일 검사와 수정을 해주는 스킬입니다.

비유: 맞춤법 검사기가 글을 쓸 때마다 자동으로 틀린 부분을 고쳐주는 것과 같습니다.

언제 쓰나요? - 파일을 수정할 때마다 자동으로 실행됩니다 - 코드 스타일을 자동으로 통일하고 싶을 때

어떻게 작동하나요? 3단계로 작동합니다. 1단계: 포매터가 40~50%를 자동 수정합니다. 2단계: 고칠 수 없는 문제를 수집합니다. 3단계: 문제의 복잡도에 따라 적합한 AI 모델을 불러서 수정합니다.

사용법: 자동으로 실행됨 (PostToolUse hook)

browser-qa

한 줄 요약: 실제 브라우저에서 웹사이트를 자동으로 테스트하는 스킬입니다.

비유: 품질 관리팀이 실제로 사이트를 돌아다니며 “여기 깨졌네, 여기 느리네” 확인하는 것을 자동화한 것입니다.

언제 쓰나요? - 기능 배포 후 실제로 잘 되는지 확인할 때 - 레이아웃, 폼, 링크가 제대로 동작하는지 테스트할 때 - 접근성과 반응형 디자인을 검사할 때

어떻게 작동하나요? 브라우저 자동화 도구로 실제 페이지를 열어서 3단계로 테스트합니다. 1. 스모크 테스트: 페이지가 정상인지 기본 확인. 2. 인터랙션 테스트: 링크, 폼, 인증 흐름 테스트. 3. 비주얼 회귀: 여러 화면 크기에서 스크린샷 비교.

사용법: /browser-qa 사이트주소

click-path-audit

한 줄 요약: 버튼을 눌렀을 때 실제로 의도한 결과가 나오는지 추적하는 스킬입니다.

비유: “새 이메일” 버튼을 눌렀는데 실제로 아무 일도 안 일어나는 경우를 찾아냅니다. 개별 기능은 다 동작하는데, 조합하면 서로 취소해서 결과가 안 나오는 버그를 잡습니다.

언제 쓰나요? - 일반 디버깅으로 못 찾는 버그가 있을 때 - 대규모 코드 변경 후 버튼들이 제대로 작동하는지 확인할 때 - 사용자가 “이 버튼이 안 돼요”라고 할 때

어떻게 작동하나요? 모든 클릭 가능한 요소에 대해: 클릭 핸들러 확인 > 호출되는 함수를 순서대로 추적 > 각 함수가 어떤 상태를 읽고/쓰는지 확인 > 뒤의 함수가 앞의 결과를 되돌리는지 확인 > 최종 상태가 버튼 레이블과 일치하는지 확인합니다.

사용법: /click-path-audit

santa-method

한 줄 요약: 두 명의 독립 검토자가 모두 “합격”해야 결과물을 통과시키는 검증 방법입니다.

비유: 산타클로스가 “리스트를 만들고, 두 번 확인한다(Making a list, checking it twice)”는 것처럼, AI가 만든 결과물을 두 명의 독립 검토자가 각각 확인합니다.

언제 쓰나요? - 프로덕션에 배포되는 코드에 - 정확성이 중요한 기술 문서나 콘텐츠에 - 대량 생성 시 AI 환각(없는 사실 생성)을 방지할 때

어떻게 작동하나요? 1단계: 생성자(Agent A)가 결과물을 만듭니다. 2단계: 검토자 B와 C가 서로 모르는 상태에서 같은 기준으로 각각 검토합니다. 둘 다 합격하면 통과, 하나라도 실패하면 수정 후 재검토합니다.

사용법: 품질이 중요한 결과물 생성 시 적용

kdh-playwright-e2e-full-auto-24-7-tmux

비유: 24시간 돌아가는 자동 품질 검사 로봇이 사이트를 계속 테스트하고, 문제를 발견하면 자동으로 수정하고 배포합니다.

kdh-playwright-e2e-full-auto-24-7-vs

비유: 같은 자동 검사 로봇인데, VS Code 안에서 돌아가는 버전입니다.

언제 쓰나요? - VS Code Claude Code에서 15분 간격 자동 테스트를 돌릴 때 - 단발 E2E 테스트를 실행할 때

어떻게 작동하나요? 15분마다: 사이트 살아있나 확인 > 로그인 > API 스모크 테스트 > 페이지별 테스트 > 버그 발견 시 자동 수정 > 배포 > 정리 > 보고서. admin, ceo, api-only, quick 등 범위를 지정할 수 있습니다.

사용법: /loop 15m /kdh-playwright-e2e-full-auto-24-7
