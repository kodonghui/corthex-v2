## 전문가 리뷰

### Sally (UX)
관심종목 추가 플로우가 시나리오에 추가된 것은 좋은데, **검색 결과가 어디에 어떻게 표시되는지** 구체적 UI 정의가 없습니다. 드롭다운인지, 인라인 리스트인지에 따라 사용자 경험이 완전히 다릅니다. 특히 모바일에서 검색 결과가 종목 사이드바의 max-h-[180px] 안에서 표시되면 공간이 부족합니다. 3클릭 체크는 통과합니다: 종목 선택(1클릭), 비교 토글(1클릭), 채팅(0클릭/데스크톱, 1클릭/태블릿), 검색→추가(2클릭). 모든 핵심 동작이 3클릭 이내입니다.

### Winston (Architect)
**Banana2 프롬프트에 태블릿 버전이 없었습니다.** 데스크톱(1440px)과 모바일(375px)만 있고, 768px 태블릿 프롬프트가 빠져 있어 Banana2가 슬라이드인 채팅 디자인을 생성할 수 없었습니다. 이 문제를 발견하여 태블릿 프롬프트를 추가했습니다. 프롬프트에 backdrop, close 버튼, floating action button 트리거를 명시했으므로 디자인 생성이 가능해졌습니다. 컴포넌트 구조는 5개로 적절하며, 기존 파일 수정 범위에서 벗어나지 않습니다.

### Amelia (Dev)
기존 20개 testid에서 **3개를 추가 발견하여 반영했습니다**: `stock-search-results`(검색 결과 드롭다운), `stock-drag-handle`(드래그 핸들), `compare-stock-checkbox`(비교 종목 체크박스). 특히 관심종목 추가 테스트를 위해 검색 결과 목록의 testid가 필수적입니다. 검색 입력(`stock-search`)은 있었지만 결과 표시 영역의 testid가 없으면 "결과가 표시되었는지" 자체를 검증할 수 없었습니다. Banana2 프롬프트에 에러 상태 디자인 요청(7번 항목)이 포함되어 있어 디자인 커버리지도 충분합니다.

### Quinn (QA)
에러/빈 상태 테스트(9~10번)가 추가되어 좋습니다. 하지만 **관심종목 추가와 드래그 정렬 테스트가 테스트 항목에 없었습니다.** 섹션 1 시나리오에 "종목 검색 → 관심종목에 추가"가 명시되어 있는데 이를 검증하는 테스트가 없으면 시나리오-테스트 간 불일치입니다. 이번 라운드에서 15번(관심종목 추가)과 16번(드래그 정렬) 테스트를 추가하여 해결했습니다. 비교 모드에서 종목 1개만 선택 시의 엣지케이스는 구현 단계에서 결정해도 무방합니다.

### John (PM)
v1 기능 6가지가 모두 체크되어 기능 커버리지는 완벽합니다. **자동 갱신(60초) 중 API 실패 시 기존 데이터 유지 vs 전체 에러 전환이 미정의입니다.** UX 관점에서는 기존 가격을 유지하고 상단에 작은 에러 배너만 표시하는 것이 합리적이지만, 이건 구현 시 결정해도 되는 경미한 수준입니다. 스펙에 "갱신 실패 시 기존 데이터 유지 + 배너 안내"를 권장사항으로 추가하면 좋겠습니다.

### Bob (SM)
UI-only 범위 확인 완료. 5개 컴포넌트 수정 + 태블릿 슬라이드인(CSS position: fixed + isChatOpen 토글)으로 구현 가능합니다. 새로 추가된 testid 3개도 기존 요소에 data-testid 속성을 추가하는 것이므로 범위 안입니다. **Banana2 태블릿 프롬프트 추가로 디자인 작업이 1개 늘었지만,** 별도 이미지 생성이므로 개발 범위에는 영향이 없습니다. "절대 건드리면 안 되는 것" 리스트가 명확하여 범위 침범 위험이 낮습니다.

### Mary (BA)
트레이딩 워크스테이션의 비즈니스 가치는 명확합니다. "AI가 투자 분석을 해준다"는 CORTHEX의 핵심 차별점이며, 3패널 구조가 이를 효과적으로 전달합니다. **태블릿 사용자에게도 채팅 접근성이 보장된다는 점이 비즈니스적으로 중요합니다** — 태블릿으로 차트를 보면서 AI 분석을 요청하는 유스케이스가 충분히 예상되기 때문입니다. 에러/빈 상태 추가로 첫 사용자 온보딩 경험도 개선됩니다.

## 크로스톡

**Amelia → Quinn:** 드래그 정렬 테스트를 위해 `stock-drag-handle` testid를 추가했습니다. Playwright의 `dragTo` API로 테스트 가능하지만, 드래그 라이브러리(dnd-kit 등) 의존적이라 E2E에서 불안정할 수 있어요. 최소한 "드래그 핸들이 렌더링되었는지" 확인하는 테스트로 시작하고, 실제 정렬 변경은 선택적으로 테스트하는 게 현실적입니다.

**Winston → Sally:** 검색 결과 드롭다운과 슬라이드인 채팅의 z-index 계층 정리가 필요합니다. 검색 드롭다운(z-10) < 슬라이드인 채팅(z-20) < 모달(z-30) 정도의 계층을 권장합니다. Sally가 지적한 모바일 검색 결과 공간 문제는 오버레이 드롭다운으로 해결 가능한데, 이건 구현 시 CSS로 처리할 수 있는 범위입니다.

**John → Bob:** 자동 갱신 중 에러 처리 — 기존 가격 유지 + 상단 작은 에러 배너가 가장 합리적입니다. 전체를 에러 상태로 전환하면 사용자가 보던 데이터가 사라지니까요. Bob, 이건 에러 배너 컴포넌트 추가가 아니라 기존 에러 상태에 "부분 에러" 변형을 추가하는 것이므로 UI-only 범위 안이겠죠?
