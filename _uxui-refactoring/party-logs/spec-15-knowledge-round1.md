## 리뷰 결과

### John (PM)
v1-feature-spec.md 16번(정보국/Knowledge)과 20번(에이전트 메모리)를 대조했습니다. 스펙의 9번 기존 기능 체크리스트에 문서 관리 12개, 에이전트 기억 9개 항목이 매우 상세하게 나열되어 있어 커버리지가 우수합니다. 다만 v1 16번에서 "부서별 지식 자동 주입 (에이전트 시스템 프롬프트에)"이 핵심 기능으로 언급되어 있는데, 이것은 백엔드 로직이지만 사용자가 "이 문서가 AI에게 전달된다"는 사실을 인지할 수 있어야 합니다. 안내 텍스트 정도는 UI 범위 내에서 추가할 수 있습니다.

### Winston (Architect)
10개 컴포넌트 중 9개가 pages/knowledge.tsx 인라인으로 정의되어 있는데, 문서 탭(FolderTree + DocList + DocDetail) + 에이전트 기억 탭 + 여러 모달까지 포함하면 파일 크기가 상당히 커질 수 있습니다. 하지만 UXUI 리팩토링에서 구조 변경은 범위 밖이므로 기존 구조를 유지하는 것이 맞습니다. FolderNode의 재귀 구조는 건드리면 안 되는 목록에 올바르게 포함되어 있습니다. MarkdownRenderer가 공유 컴포넌트로 분리되어 있어 재사용 패턴이 좋습니다.

### Sally (UX)
핵심 동작 3클릭 분석: (1) 문서 찾기: 폴더 클릭 + 문서 클릭 = 2클릭 OK. (2) 문서 생성: "+ 새 문서" + 입력 + 저장 = 2클릭(모달 내부) OK. (3) 기억 조회: 에이전트 기억 탭 + 에이전트 필터 = 2클릭 OK. 모든 핵심 동작이 3클릭 이내입니다. 다만 문제점 2번에서 "폴더 컨텍스트 메뉴가 hover 시에만 보여서 터치 디바이스에서 접근 어려움"이 지적되었는데, 개선 방향에서 구체적인 해결책(롱프레스, 또는 항상 보이는 ... 버튼)이 명시되지 않았습니다.

### Amelia (Dev)
data-testid 목록이 40개로 포괄적이며, 폴더 트리, 문서 CRUD, 태그, 드래그앤드롭, 버전 이력, 에이전트 기억까지 모두 커버합니다. 그러나 에러 상태 testid(`knowledge-error-state`)와 로딩 상태 testid(`knowledge-loading`)가 누락되어 있었습니다. 다른 스펙들(ops-log, messenger, sns)에서는 모두 포함되어 있으므로 일관성을 위해 추가가 필요합니다. 드래그앤드롭 관련 testid(`knowledge-dropzone`)는 있어서 파일 업로드 테스트가 가능합니다.

### Quinn (QA)
Playwright 테스트가 34개 항목으로 가장 많은 테스트 시나리오를 가지고 있어 커버리지가 우수합니다. 빈 상태가 문서(`knowledge-doc-empty`)와 기억(`knowledge-memory-empty`) 두 곳 모두 정의되어 있습니다. 그러나 API 에러 상태 테스트와 로딩 스켈레톤 테스트가 빠져 있었습니다. 또한 파일 업로드 실패 시(파일 크기 초과, 지원하지 않는 형식 등) 에러 메시지 표시 시나리오도 필요합니다. 드래그앤드롭 테스트(33번)가 있는 것은 좋지만, 드롭 실패 시나리오가 없습니다.

### Bob (SM)
작업 범위가 잘 정의되어 있습니다. "절대 건드리면 안 되는 것"에 폴더 트리 재귀 구조, buildParams(), 파일 업로드 로직, 드래그앤드롭 이벤트 핸들러, 버전 복원 mutation, 기억 필터/검색/페이징 로직이 포함되어 있어 경계가 명확합니다. 개선 방향의 "폴더 리사이즈 드래그"나 "폴더 간 문서 드래그 이동"은 UI만 변경 범위를 넘을 수 있으므로 "검토" 수준으로 유지하는 것이 적절합니다.

### Mary (BA)
지식베이스는 AI 에이전트의 "기억"과 "참고 자료"를 관리하는 핵심 인프라입니다. 폴더 계층 구조와 태그 기반 검색은 문서 관리의 기본이며, 에이전트 기억 탭은 "AI가 무엇을 학습했는가"를 투명하게 보여주어 신뢰를 구축합니다. 신뢰도 시각화와 사용 횟수 표시는 데이터 기반으로 기억의 가치를 판단하게 해줍니다. 비즈니스 가치가 분명합니다.

## 크로스톡
- **John** -> **Sally**: "부서별 지식 자동 주입"이 UI에서 인지되려면, 문서 상세 뷰나 폴더 설명에 "이 폴더의 문서는 해당 부서 에이전트에게 자동 참조됩니다" 같은 안내가 필요합니다. 이는 텍스트 추가만으로 가능하므로 UI 범위 내입니다.

- **Sally** -> **Bob**: 폴더 컨텍스트 메뉴의 모바일 접근성은 ... 버튼을 항상 표시하는 방식이면 CSS 변경만으로 가능하므로 UI 범위 안입니다.
