## 리뷰 결과

### John (PM)
v1-feature-spec.md 12번(기밀문서/Archive)을 대조했을 때, 스펙의 9번 기존 기능 체크리스트가 보고서 관련 기능을 잘 커버하고 있습니다. 다만 v1에서 "부서별, 등급별 필터"가 있었는데, 현재 스펙에서는 탭 필터(전체/내 보고서/받은 보고서)만 존재합니다. 보고서 검색 기능이 4.3에서 언급되었으나 실제 data-testid에 검색 관련 항목이 누락되어 있어 구현 시 빠질 위험이 있습니다.

### Winston (Architect)
컴포넌트 구조가 ReportsPage 안에 인라인으로 5개 컴포넌트를 두는 방식인데, 보고서 페이지의 복잡도를 감안하면 합리적입니다. MarkdownRenderer와 ShareToConversationModal은 기존 공유 컴포넌트를 재사용하는 것이 좋습니다. 다만 ReportEditor에서 마크다운 프리뷰를 좌우 분할로 구현한다면, 에디터 컴포넌트의 크기가 상당히 커질 수 있으므로 별도 파일로 분리하는 것도 고려할 만합니다.

### Sally (UX)
핵심 동작 3클릭 분석: (1) 보고서 작성: "+ 새 보고서" 클릭 → 내용 입력 → 저장 = 3클릭 OK. (2) CEO에게 보고: 카드 클릭 → 보고 버튼 → 확인 = 3클릭 OK. (3) 코멘트 작성: 카드 클릭 → 입력 → 전송 = 3클릭 OK. 사용자 흐름은 잘 설계되었습니다. 단, 마크다운 에디터에서 프리뷰 전환이 모바일에서 탭 방식인데, 탭 전환 시 편집 위치를 잃지 않도록 주의가 필요합니다.

### Amelia (Dev)
data-testid 목록이 32개로 상당히 포괄적입니다. 그러나 4.2에서 언급된 검색 기능(`reports-search-input`)과 상태 필터가 testid에 없습니다. 또한 마크다운 프리뷰 탭 전환 버튼(`reports-preview-tab`, `reports-edit-tab`)도 누락되어 있습니다. 구현 관점에서 testid가 없으면 해당 기능의 Playwright 테스트를 작성할 수 없게 됩니다.

### Quinn (QA)
빈 상태(`reports-empty-state`)와 로딩 스켈레톤(`reports-loading-skeleton`)은 정의되어 있습니다. 그러나 에러 상태(API 실패 시)에 대한 정의가 없습니다. 보고서 상세 뷰에서 네트워크 오류가 발생했을 때 어떤 UI를 보여줄지 명시가 필요합니다. 또한 코멘트 전송 실패 시의 재시도 UI도 정의되어야 합니다. 빈 코멘트 전송 방지는 테스트 17번에 제목 없이 저장 방지만 있고, 빈 코멘트 전송 방지는 없습니다.

### Bob (SM)
작업 범위가 UI만 변경으로 잘 제한되어 있고, API 엔드포인트 변경 없음이 명확합니다. 컴포넌트 7개 중 MarkdownRenderer와 ShareToConversationModal은 기존 유지이므로 실제 변경은 5개 컴포넌트입니다. 마크다운 프리뷰 좌우 분할은 새로운 UI 패턴이므로 예상보다 시간이 걸릴 수 있지만, 전체적으로 현실적인 범위입니다.

### Mary (BA)
보고서 라이프사이클(draft → submitted → reviewed)이 비즈니스 흐름으로 명확합니다. CEO에게 보고하는 핵심 가치가 잘 반영되어 있고, 코멘트를 통한 피드백 루프도 비즈니스적으로 중요한 기능입니다. 다만 보고서 검색/정렬 기능이 개선 방향에만 있고 실제 구현 스펙(testid, 테스트 항목)에 반영되지 않아 비즈니스 요구사항이 누락될 위험이 있습니다.
