## 전문가 리뷰

### Sally (UX)
핵심 동작을 적대적으로 재검증합니다. 에이전트 생성: "+ 새 AI 직원 클릭 → 폼 입력 → 생성 버튼" 3단계. 에이전트 편집: "행 클릭 → 필드 수정 → 저장" 3단계. 둘 다 3클릭 이내입니다. 하지만 **슬라이드 패널의 "실수로 닫기 쉬움" 문제(문제점 #2)**에 대한 개선 방향이 섹션 4에 없습니다. 배경 클릭으로 패널이 닫히면 편집 중인 내용이 날아갈 수 있어요. "저장하지 않은 변경이 있습니다" 확인 다이얼로그를 추가하는 것이 이상적이지만, 이것은 동작 로직 변경에 해당하므로 현재 UI-only 범위를 넘을 수 있습니다. 최소한 문제점으로만이라도 명시해두면 향후 개선 대상이 됩니다 — 현재 문제점 #2에 이미 기술되어 있으므로 인지는 되어 있습니다.

### Winston (Architect)
컴포넌트 목록이 1행("AgentsPage" 단일 파일)인 것은 "UI만 변경, 구조 유지" 원칙에 맞습니다. 다만 이 파일 안에 테이블, 디테일 패널, 생성 폼, Soul 에디터, 비활성화 모달 등 최소 5~6개의 논리적 섹션이 있을 텐데, **각 섹션의 대략적 함수명이나 위치 힌트가 없어서 구현자가 어디를 수정해야 하는지 파악하기 어렵습니다**. 예를 들어 "Soul 에디터는 `renderSoulEditor()` 또는 `<SoulEditor />` 부근"처럼 힌트를 주면 좋겠지만, 이것은 실제 코드를 확인해야 하므로 구현 단계에서 보완할 사항입니다. 현재 스펙 수준에서는 문제없습니다.

### Amelia (Dev)
testid 27개로 주요 요소가 커버됩니다. 한 가지 적대적 지적: **`agents-row`는 복수의 행이 있을 때 특정 행을 구분할 수 없습니다**. Playwright에서 "3번째 에이전트의 편집 버튼 클릭" 같은 테스트를 작성하려면 `agents-row-{id}` 패턴이 필요합니다. 다만 이것은 구현 시 동적으로 `data-testid={`agents-row-${agent.id}`}` 식으로 처리하면 되고, 스펙에서는 `agents-row` 기본 패턴만 명시해도 충분합니다. `nth()` 셀렉터로도 대응 가능하므로 Critical은 아닙니다.

### Quinn (QA)
테스트 16개로 확장되어 happy path + edge case 커버리지가 많이 좋아졌습니다. 적대적으로 하나 더: **비활성화 후 에이전트가 목록에서 어떻게 표시되는지 테스트가 불명확합니다**. 테스트 #8 "비활성화 클릭 → 확인 → 에이전트 비활성화"의 기대 결과가 모호해요 — 목록에서 사라지는 건지, 회색으로 표시되는 건지, 상태 필터에서만 보이는 건지. 비활성화 에이전트의 시각적 표현(색상 테이블에 "비활성 에이전트: 회색 뱃지"가 있음)과 테스트 기대 결과를 맞춰야 합니다. 다만 이것은 테스트 시나리오 구체화 수준이지, 스펙 구조의 문제는 아닙니다.

### John (PM)
기능 커버리지가 8개 항목 모두 체크되어 있고, "절대 건드리면 안 되는 것" 5개도 명확합니다. 적대적 관점에서 찾은 것: **계급 변경 시 모델 자동 매칭(handleTierChange)**이 "건드리면 안 되는 것"에 있지만, 이 동작의 사용자 시나리오가 섹션 1에 명시되지 않았습니다. 관리자가 계급을 바꾸면 모델이 자동으로 바뀌는 것은 중요한 UX 동작인데, 시나리오에 "계급 변경 시 추천 모델 자동 선택"이 빠져 있어요. 다만 테스트 #16에 "계급→모델 자동 매칭"이 추가되어 있으므로 검증은 가능합니다.

### Bob (SM)
범위가 깔끔합니다 — 단일 파일 수정, API 변경 없음, 카드 뷰는 Enhancement로 분리 완료. Sally가 지적한 슬라이드 패널 닫기 문제는 **동작 로직 변경**에 해당하므로 현재 "UI만 변경" 범위 밖입니다. 이 부분은 문제점 #2에 기록되어 있으니 향후 Enhancement로 다루면 됩니다. 전체적으로 스코프 크립 없이 잘 관리되고 있습니다.

### Mary (BA)
에이전트 관리는 CORTHEX v2의 "동적 조직 관리" 핵심 가치와 직결되는 페이지입니다. v1이 29명 고정이었다면 v2는 관리자가 자유롭게 에이전트를 추가/제거하는 것이 차별점인데, 이 경험이 시각적으로도 매력적이어야 합니다. Banana2 프롬프트에서 "scannable agent list"와 "spacious Soul editor"를 강조한 것이 관리 효율과 직결되어 비즈니스 가치가 높습니다. 비서 에이전트 뱃지 추가도 CEO의 주 인터페이스를 시각적으로 구분하는 데 의미 있습니다.

## 크로스톡

**Sally <-> Bob:** 슬라이드 패널 "실수로 닫기" 문제에 대해 토론했습니다. Sally는 "저장 안 한 변경 있으면 경고"를 제안했고, Bob은 "동작 로직 변경이라 현재 범위 밖"이라고 답했습니다. 합의: 문제점 #2에 이미 기록되어 있으므로 현재 범위에서는 스타일만 개선하고, 닫기 방지 로직은 향후 Enhancement로 분리합니다.

**Quinn <-> John:** 비활성화 테스트 기대 결과의 모호함과 계급→모델 자동 매칭 시나리오 누락을 논의했습니다. Quinn의 "비활성화 후 시각적 표현"은 색상 테이블에 정의되어 있고(회색 뱃지), John의 handleTierChange는 테스트 #16으로 커버됩니다. 둘 다 스펙 품질에 큰 영향을 주지 않는 사소한 개선 사항입니다.

**Amelia <-> Winston:** `agents-row` 동적 testid 패턴과 인라인 컴포넌트 위치 힌트에 대해 논의했습니다. 둘 다 구현 단계에서 자연스럽게 해결될 사항이며, 스펙 수준에서는 기본 패턴만 명시하면 충분하다는 데 합의했습니다.
