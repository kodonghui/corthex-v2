## 리뷰 결과

### John (PM)
v1-feature-spec.md 14번(자동화/Workflow)과 15번(크론기지/Schedule)을 대조했습니다. 스펙의 9번 기존 기능 체크리스트가 16개 항목으로 매우 상세하게 커버되어 있습니다. v1에서 "워크플로우 CRUD"와 "다단계 스텝 정의"가 체인 작업으로 잘 매핑되어 있고, "활성/비활성 토글"도 스케줄과 트리거 모두에 포함되어 있습니다. 다만 v1 15번에서 "last_run, next_run 추적"이 있었는데, 스케줄 카드에 "다음 실행" 시간만 있고 "마지막 실행" 시간은 레이아웃에 없습니다.

### Winston (Architect)
8개 컴포넌트가 모두 jobs.tsx 안에 인라인으로 정의되어 있습니다. 이 페이지는 3가지 탭(일회성/스케줄/트리거) + 등록 모달 + 체인 그룹까지 있어 파일이 매우 커질 수 있습니다. 그러나 UXUI 리팩토링 범위에서는 구조 변경보다 스타일 변경이 목적이므로, 기존 인라인 구조를 유지하는 것이 맞습니다. ProgressBar가 @corthex/ui 공유 컴포넌트를 사용하는 것은 좋은 재사용 패턴입니다.

### Sally (UX)
핵심 동작 3클릭 분석: (1) 작업 등록: "+ 작업 등록" → 에이전트 선택 + 지시 입력 → 등록 = 3클릭 OK (모달 내부 동작은 1클릭). (2) 작업 결과 확인: 카드 클릭(확장) = 1클릭 OK. (3) 스케줄 중지: 스케줄 탭 → 중지 버튼 = 2클릭 OK. 사용자 흐름이 우수합니다. 다만 모달에서 유형 선택(라디오) 후 해당 필드가 나타나는 패턴이 초보 사용자에게 혼란을 줄 수 있습니다. 유형별로 어떤 필드가 나타나는지 시각적 가이드가 필요합니다.

### Amelia (Dev)
data-testid가 45개로 매우 포괄적입니다. WebSocket 진행률 관련 testid(`jobs-card-progress`)가 있어 실시간 UI 테스트가 가능합니다. 그러나 스케줄 편집 모달과 트리거 편집 모달의 testid가 등록 모달(`jobs-modal`)을 재사용하는 것인지, 별도인지 불명확합니다. 수정 모드에서의 모달 제목("작업 수정")을 구분할 testid가 없습니다. 또한 에러 상태 testid(`jobs-error-state`)가 누락되어 있습니다.

### Quinn (QA)
빈 상태(`jobs-empty-state`)는 정의되어 있으나, 탭별 빈 상태가 구분되는지 불명확합니다. 일회성 탭이 비어있고 스케줄 탭에 항목이 있을 때, 일회성 탭의 빈 상태만 표시되어야 합니다. 또한 체인 작업에서 중간 단계가 실패했을 때의 UI 상태가 명확하지 않습니다. 후속 작업이 "대기(체인)" 상태로 남는지, "취소됨"으로 바뀌는지에 따라 UI가 달라야 합니다. WebSocket 연결 끊김 시의 fallback UI도 정의가 필요합니다.

### Bob (SM)
작업 범위가 잘 제한되어 있습니다. WebSocket 이벤트 핸들링은 건드리지 않고, 진행률 바 UI만 개선하는 것이 명확합니다. 모달 단계 분리(4.2)가 "고려"로 되어 있어 필수가 아닌 점도 범위 관리에 좋습니다. 다만 "진행 중인 작업 상단 고정"(4.3)은 리스트 정렬 로직 변경이 필요할 수 있어 UI만 변경 범위를 약간 넘을 수 있습니다.

### Mary (BA)
"시켜놓고 퇴근" 컨셉이 비즈니스적으로 명확합니다. 3가지 작업 유형이 각각의 사용 시나리오에 맞게 구분되어 있습니다. 체인 작업은 복잡한 워크플로우를 지원하는 차별화 기능입니다. 스케줄의 마지막 실행 시간이 없으면 사용자가 "이 스케줄이 제대로 돌아가고 있나?"를 확인하기 어렵다는 John의 지적에 동의합니다.

## 크로스톡
- **John** → **Mary**: "마지막 실행 시간이 없으면 비즈니스적으로 스케줄 신뢰도를 판단할 수 없습니다." Mary 동의: "사용자가 출근해서 '어젯밤에 잘 돌았나?' 확인하는 게 핵심 시나리오이므로 last_run 표시가 필요합니다."
- **Quinn** → **Amelia**: "체인 실패 시 후속 작업 상태가 불명확한데, testid에서도 실패+후속 상태를 구분할 수 있어야 합니다."
- **Sally** → **Bob**: "모달 단계 분리가 선택사항이라면, 현재 모달에서 유형 전환 시 적어도 필드 영역에 간단한 설명 텍스트를 추가하는 건 UI만 변경 범위 안입니다."
