## 전문가 리뷰

### Sally (UX)
토론 선택 → 타임라인 확인이 1클릭, 새 토론이 2클릭(버튼+생성), Diff 뷰가 2클릭(선택+토글)으로 모든 핵심 동작이 3클릭 이내입니다. **태블릿(768px~1023px)에서 InfoPanel이 숨겨질 때, 참여자 수/토론 유형/라운드 진행도 같은 핵심 정보를 타임라인 헤더에 요약해야 합니다.** Round 1에서 이 문제를 지적했는데, 섹션 8의 태블릿 breakpoint에 이 요약 표시가 명시되지 않아 구현 시 빠질 수 있습니다. 이번 라운드에서 반응형 섹션에 태블릿 참여자 요약을 추가했습니다.

### Winston (Architect)
컴포넌트 6개 구조가 합리적입니다. DebateResultCard를 "관련 컴포넌트"로 포함한 것은 디자인 일관성 측면에서 좋은 판단입니다. **Banana2 데스크톱 프롬프트에 Diff 뷰 요소가 누락되어 있었습니다.** Required functional elements에 Loading/Error state만 있고 Diff 뷰가 빠져 있어 Banana2가 Diff 뷰 디자인을 생성하지 않을 수 있었습니다. 이번 라운드에서 7번 항목으로 Diff 뷰 토글과 비교 뷰를 프롬프트에 추가했습니다. 기존 timeline API의 데이터를 클라이언트에서 재가공하는 방식이므로 API 변경 없이 구현 가능합니다.

### Amelia (Dev)
testid가 23개에서 **25개로 확충되었습니다**: `debate-retry-btn`(에러 재시도 버튼)과 `debate-participant-summary`(태블릿/모바일 참여자 요약) 추가. 에러 상태에서 재시도 버튼의 별도 testid가 없으면 "재시도 클릭 후 데이터 갱신" 테스트를 작성하기 어렵습니다. `debate-item`이 목록에서 여러 개 나오는데, Playwright에서 `nth` selector나 `filter({ hasText })` 패턴으로 특정 항목을 선택할 수 있으므로 suffix 규칙 없이도 테스트 가능합니다.

### Quinn (QA)
테스트 13개로 주요 시나리오가 커버됩니다. **진행 중인 토론의 실시간 갱신(5초)을 확인하는 테스트가 빠져 있습니다.** 이건 v1의 핵심 기능 중 하나인데 테스트 항목에 없습니다. "진행 중 토론 선택 → 5초 후 새 발언 자동 추가" 같은 테스트가 필요합니다. 다만 E2E에서 5초 대기 테스트는 불안정할 수 있으므로, 최소한 `debate-live-indicator`가 표시되는지 확인하는 정도로도 충분합니다. 에러 상태 테스트(12번)와 로딩 상태 테스트(11번)가 추가된 것은 좋습니다.

### John (PM)
v1 기능 8항목 체크리스트가 완비되어 기능 커버리지가 충분합니다. Diff 뷰와 합의/비합의 뱃지가 모두 포함되었습니다. **한 가지 우려: "토론 삭제" 기능이 v1에 있었는지 확인이 필요합니다.** 현재 스펙에는 생성만 있고 삭제가 없는데, v1에 삭제가 없었다면 문제없지만 있었다면 누락입니다. 기존 기능 참고사항에 삭제 기능 유무를 명확히 해야 합니다.

### Bob (SM)
범위가 적절합니다. 6개 컴포넌트(DebateResultCard 포함)의 UI 스타일 변경이고, Diff 뷰도 기존 timeline API 데이터를 클라이언트에서 재가공하므로 UI-only 범위입니다. **"절대 건드리면 안 되는 것" 4개가 명확하여 scope creep 위험이 낮습니다.** Banana2 프롬프트에 Diff 뷰가 추가되어 디자인 작업 범위가 약간 늘었지만, 기존 프롬프트 내 항목 추가이므로 별도 이미지 생성이 필요하지 않습니다.

### Mary (BA)
비즈니스 가치가 명확합니다: AI 에이전트 간 토론을 더 읽기 쉽게 만들어 CEO의 의사결정 지원을 강화합니다. **Diff 뷰 복원은 핵심 가치입니다** — CEO가 "이 에이전트가 다른 에이전트의 주장에 어떻게 반응했는지"를 한눈에 파악할 수 있으면 토론의 실질적 가치가 크게 올라갑니다. 라운드 진행도와 발언자별 색상 구분도 사용성을 높이는 데 기여합니다.

## 크로스톡

**Sally → Winston:** Banana2 프롬프트에 Diff 뷰를 추가한 건 좋은데, 모바일 프롬프트에도 Diff 뷰 언급이 필요한지 고민됩니다. 반응형 섹션에서 모바일 Diff는 "세로 스택"으로 정의했지만, Banana2 모바일 프롬프트에는 이 정보가 없어요. 다만 모바일 프롬프트는 "Debate Detail shows timeline only"로 간결하게 유지하는 게 맞고, Diff 뷰는 데스크톱 프롬프트의 디자인을 참고해서 모바일 구현 시 CSS로 세로 전환하면 됩니다.

**Quinn → John:** 토론 삭제 기능 — 현재 API 엔드포인트 목록에 DELETE가 없으므로 v1에서도 삭제 기능이 없었을 가능성이 높습니다. 만약 있었다면 이건 UI 스펙이 아니라 기능 스펙에서 다뤄야 할 문제이고, 현재 UXUI 리팩토링 범위에서는 기존 UI에 없던 기능을 추가하지 않는 것이 원칙이므로 문제없습니다.

**Amelia → Quinn:** 실시간 갱신 테스트 — `debate-live-indicator` 존재 확인 테스트로 충분하다고 봅니다. 5초 대기 후 DOM 변화를 확인하는 건 E2E 테스트의 안정성을 해칠 수 있어요. 현재 테스트 항목에 라이브 인디케이터 확인이 간접적으로 포함되어 있긴 하지만, 별도 항목으로 명시하면 더 명확해질 것입니다.
