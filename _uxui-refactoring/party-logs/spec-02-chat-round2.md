## 전문가 리뷰

### Sally (UX)
사용자 흐름은 3클릭 이내로 잘 설계되어 있습니다 — 세션 클릭(1) → 메시지 입력(2) → 전송(3). 그러나 **모바일(375px)에서 파일 첨부 미리보기의 공간 처리**가 원래 명시되어 있지 않았습니다. 최대 5개 첨부 파일의 미리보기 칩이 375px 폭에서 어떻게 배치되는지(flex-wrap vs 가로 스크롤) 정의가 필요했고, 수정 후 flex-wrap으로 줄 바꿈 처리하는 것으로 명시되었습니다. 또한 위임 패널의 모바일 전환 UX — 헤더의 "채팅 보기 | 위임 내역" 탭이 375px에서 에이전트명과 함께 들어가면 공간 부족이 우려되지만, 현재 코드 동작을 유지하는 UI-only 범위이므로 수용 가능합니다.

### Winston (Architect)
컴포넌트 5개(ChatPage, SessionPanel, ChatArea, AgentListModal, ToolCallCard) 구조가 합리적이고, "건드리면 안 되는 것" 8개 항목이 명확합니다. **한 가지 누락: DebateResultCard가 `agora/debate-result-card.tsx`에서 import되어 ChatArea 안에 인라인으로 표시되는데, 컴포넌트 목록이나 Banana2 프롬프트에 포함되어 있지 않았습니다.** 이것은 agora 페이지 스펙에서 다루는 컴포넌트이므로, 섹션 9에 참조 메모("agora 페이지 컴포넌트 재사용, 해당 스펙 참조")를 추가하는 것으로 해결되었습니다. 구조적으로 크로스 페이지 컴포넌트 참조가 명시된 것은 좋은 패턴입니다.

### Amelia (Dev)
data-testid가 34개로 커버리지가 우수합니다. `chat-input`의 설명이 원래 "입력 textarea"로 되어 있었는데, 실제 코드에서는 `<Input>` 컴포넌트(corthex/ui)를 사용합니다. 수정 후 "입력 필드 (Input 컴포넌트)"로 정정되어 Playwright 테스트 작성 시 혼동이 줄어들었습니다. Playwright 테스트도 18개에서 23개로 확장되어, 스트리밍 중지, 도구 호출 카드, 위임 패널 전환, 타이핑 인디케이터, 연결 상태 배너까지 커버합니다.

### Quinn (QA)
edge case가 Round 1 이후 크게 보강되었습니다. 연결 끊김/재연결 배너, 타이핑 인디케이터, 에러 재시도가 모두 정의되어 있어요. **파일 첨부 실패 케이스**(5개 초과 시 버튼 비활성화, 파일 크기 초과, 업로드 중 네트워크 에러)가 별도로 명시되어 있지 않지만, 이것은 기존 코드의 동작을 그대로 유지하는 것이므로 UI-only 리팩토링 범위에서는 Low입니다. 또한 **스트리밍 중 입력 비활성화** 상태가 Playwright 테스트에 명시적으로 없지만, 이것도 기존 동작 유지이므로 구현 시 자연스럽게 커버됩니다.

### John (PM)
기존 기능 체크리스트가 5개에서 21개로 확장되어 커버리지가 매우 좋습니다. DebateResultCard 참조가 섹션 9에 추가되면서, 채팅 페이지에서 사용하는 모든 기능이 기록되었습니다. **모바일 Banana2 프롬프트가 데스크톱 대비 간소하다는 점**이 약간 아쉽지만, "Same page described above" + 모바일 전용 동작(2뷰 토글, 뒤로가기, 풀스크린 모달)이 기술되어 있으므로 Banana2가 충분히 이해할 수 있습니다.

### Bob (SM)
스코프가 UI-only로 잘 유지되어 있습니다. 컴포넌트 5개 + testid 34개 + Playwright 23개는 채팅 페이지 하나치고 상당한 양이지만, Priority 1 페이지에 걸맞는 정밀도입니다. "건드리면 안 되는 것" 8항목이 명확하여 구현자가 범위를 벗어날 위험이 낮습니다. 작업량 리스크는 수용 가능 범위입니다.

### Mary (BA)
비즈니스 가치가 매우 명확합니다. 1:1 채팅은 사용자가 AI 에이전트와 가장 직접적으로 소통하는 핵심 접점이며, 위임 패널의 추가로 "지시 → 위임 → 결과 확인"이라는 전체 비즈니스 워크플로우가 가시화됩니다. 파일 첨부와 도구 호출 카드는 에이전트가 실제로 일하고 있다는 것을 사용자에게 보여주는 핵심 신뢰 요소입니다. 토론 명령어의 발견성(discoverability)이 낮다는 점은 있지만, 이는 기능 추가(슬래시 명령어 자동완성)에 해당하므로 현재 UI-only 범위 밖입니다.

**Sally → John:** "모바일 Banana2 프롬프트가 'Same page described above'로 참조하는 건 괜찮은데, 위임 패널의 모바일 적응은 더 구체적으로 기술하는 게 좋지 않을까요? 데스크톱에서는 탭 전환이지만 모바일에서는 별도 풀스크린일 수도 있잖아요."

**Quinn → Amelia:** "파일 첨부 에러 상태에 대한 testid(`chat-file-error` 같은)도 추가할 필요가 있지 않을까요?"

**Amelia → Quinn:** "기존 코드에서 파일 에러는 toast 알림으로 처리되고 별도 UI 요소가 아니에요. toast는 전역 컴포넌트라 이 페이지 스펙의 testid에 넣기보다는, 공통 컴포넌트 테스트에서 다루는 게 맞습니다."
