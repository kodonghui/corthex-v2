## 리뷰 결과

### John (PM)
v1-feature-spec.md 23번 항목에 따르면 사내 메신저는 v2에서 새로 추가된 기능이며, 스펙에 나열된 15개 기능(채널 CRUD, 실시간 메시징, 대화, @멘션, 스레드, 리액션, 파일 첨부, 검색, 미읽음, 온라인 상태, 멘션 알림, 멤버 관리, 채널 나가기/삭제, 드래그앤드롭)이 모두 커버되어 있습니다. v1에는 없었던 기능이므로 호환성 이슈는 없으나, 모든 기능이 실제 동작해야 합니다.

### Winston (Architect)
10개 컴포넌트로 구성되어 있으며, ChannelsView와 ConversationsView가 명확히 분리되어 있어 좋습니다. 다만 MessengerPage 내부에 ChannelSettingsModal, ThreadPanel, AttachmentRenderer가 인라인으로 정의되어 있는데, 파일 크기가 커질 수 있습니다. 컴포넌트 분리는 UI 리팩토링 범위에 포함되므로 필요시 별도 파일로 분리할 수 있습니다. WebSocket 구독 로직은 건드리면 안 되는 목록에 올바르게 포함되어 있습니다.

### Sally (UX)
메시지 전송이 1클릭(입력 후 전송/Enter)으로 가능하고, 채널 전환도 1클릭입니다. 모바일에서 채널 목록 -> 채팅 전환이 깔끔하게 설계되어 있습니다. 이모지 리액션의 모바일 접근성 개선(롱프레스)은 중요한 개선 포인트입니다. 다만 스레드 패널이 모바일에서 풀스크린 오버레이로 열릴 때 "뒤로 가기"가 직관적인지 확인이 필요합니다 -- 스레드 닫기 버튼의 위치와 크기가 중요합니다.

### Amelia (Dev)
data-testid 목록이 43개로 매우 상세합니다. 그러나 대화(Conversations) 모드의 빈 상태 testid(`messenger-conv-empty`)가 없습니다. 채널 모드에는 `messenger-empty-state`가 있지만 대화 모드 전용 빈 상태가 별도로 필요합니다. 또한 로딩 상태(`messenger-loading`)와 에러 상태(`messenger-error-state`)도 다른 스펙들과 일관성을 위해 추가해야 합니다.

### Quinn (QA)
Playwright 테스트가 22개 항목으로 주요 시나리오를 커버하지만, 로딩/에러 상태 테스트가 빠져 있습니다. 특히 WebSocket 연결 실패 시 사용자에게 보여줄 폴백 UI가 정의되지 않았습니다. 파일 첨부 시 50MB 초과 파일을 선택했을 때의 에러 메시지도 테스트 시나리오에 포함되어야 합니다. 또한 대화 모드에서 방이 없을 때의 빈 상태 테스트도 필요합니다.

### Bob (SM)
UI 변경 범위가 적절합니다. WebSocket, 스토어, API 호출을 건드리지 않으면서 스타일/레이아웃만 개선하는 것이 명확합니다. 다만 "이모지 리액션 모바일 롱프레스" 구현은 단순 CSS 변경이 아니라 이벤트 핸들러 추가가 필요할 수 있어, UI만 변경이라는 범위를 약간 넘을 수 있습니다. 사전에 범위 확인이 필요합니다.

### Mary (BA)
사내 메신저는 v2의 핵심 차별화 기능입니다. 같은 회사의 인간 직원끼리 실시간 소통할 수 있어, 단순 AI 관리 도구를 넘어 협업 플랫폼으로 확장합니다. 채널 + 대화 이중 구조는 Slack 스타일의 검증된 패턴이며, 스레드와 리액션이 토론을 구조화합니다. 멘션 팝업에 역할/부서 태그 추가는 조직 맥락을 제공하여 실질적으로 유용한 개선입니다.
