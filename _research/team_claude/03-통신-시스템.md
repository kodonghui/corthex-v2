# 03. 팀 통신 시스템

## 통신 방식 개요

팀 에이전트들은 **3가지 채널**로 소통합니다:

```
┌─────────────────────────────────────────────────┐
│                통신 시스템                         │
├─────────────────────────────────────────────────┤
│                                                 │
│  1. SendMessage (DM)      A ──────→ B           │
│     특정 팀원에게 직접 메시지                       │
│                                                 │
│  2. Broadcast             A ──┬──→ B            │
│     전체 팀원에게 동시 전송     ├──→ C            │
│     (비용 주의!)               └──→ D            │
│                                                 │
│  3. Task List (공유)      [Task Board]           │
│     태스크 상태로 간접 소통     ├── #1: 완료 (A)   │
│                               ├── #2: 진행중 (B) │
│                               └── #3: 대기       │
└─────────────────────────────────────────────────┘
```

## 1. 다이렉트 메시지 (DM)

팀원 A가 팀원 B에게 직접 메시지를 보내는 것.
**팀장을 거치지 않아도 됩니다!**

```
SendMessage({
  type: "message",
  recipient: "backend",           // 팀원 이름 (반드시 이름으로!)
  content: "API 엔드포인트 /users 구조 알려줘. 프론트에서 타입 맞춰야 해",
  summary: "API 스키마 요청"       // UI에 미리보기로 표시됨
})
```

### 특징

- **자동 배달**: 상대방이 작업 중이면 큐에 쌓였다가 턴 끝나면 자동 전달
- **비동기**: 메시지 보내고 바로 자기 일 계속 가능
- **Idle 알림**: 팀원이 할 일 없으면 자동으로 idle 알림이 팀장에게 감
- **P2P DM 알림**: 팀원끼리 DM하면, 팀장에게 요약이 자동으로 전달됨

## 2. 브로드캐스트

**전체 팀원에게 동시에** 같은 메시지를 보냄.

```
SendMessage({
  type: "broadcast",
  content: "긴급! DB 스키마 변경됨. users 테이블에 role 컬럼 추가됐으니 참고",
  summary: "DB 스키마 변경 공지"
})
```

### ⚠️ 비용 주의!

- 팀원 N명 = N번 메시지 전송 = **N배 토큰 소비**
- **정말 전원이 알아야 할 때만** 사용
- 개인한테 보낼 수 있으면 DM 사용

### 브로드캐스트가 적절한 경우

- "작업 전부 중단! 치명적 버그 발견"
- "DB 스키마 바뀜, 전원 참고해야 함"
- "새 공유 유틸 함수 만들었음, 이거 쓰세요"

## 3. 공유 태스크 리스트

**모든 팀원이 같은 태스크 보드를 봅니다.**

### 태스크 JSON 구조

```json
{
  "id": "task-001",
  "subject": "로그인 API 구현",
  "description": "POST /auth/login 엔드포인트. JWT 발급, refresh token 포함",
  "status": "pending",           // pending | in_progress | completed
  "owner": null,                 // 담당자 이름 (null = 미배정)
  "blockedBy": [],               // 의존하는 태스크 ID 목록
  "metadata": {}                 // 추가 정보
}
```

### 자기조직화 패턴

팀원들이 **스스로 일감을 가져가는** 패턴:

```
1. 팀장: TaskCreate로 15개 태스크 생성 (의존성 포함)
2. 팀원 A: TaskList 확인 → 미배정 + 미차단 태스크 발견 → TaskUpdate로 자기가 가져감
3. 팀원 A: 완료 → TaskUpdate로 completed 표시
4. 의존성 있던 태스크가 자동 언블로킹
5. 팀원 B: 언블로킹된 태스크 발견 → 가져감
6. 반복...
```

- **파일 잠금(file locking)** 으로 동시 클레임 방지
- 팀장 개입 없이 **자동으로 분배됨**

## 4. 종료 프로토콜

### 정상 종료 요청

```
// 팀장이 팀원에게
SendMessage({
  type: "shutdown_request",
  recipient: "frontend",
  content: "모든 작업 완료. 종료해주세요"
})

// 팀원이 응답 (수락)
SendMessage({
  type: "shutdown_response",
  request_id: "req-abc123",      // 요청에 포함된 ID
  approve: true
})

// 팀원이 응답 (거부 - 아직 할 일 있음)
SendMessage({
  type: "shutdown_response",
  request_id: "req-abc123",
  approve: false,
  content: "아직 task #5 작업 중입니다. 5분 더 필요합니다"
})
```

## 5. 플랜 승인 (품질 게이트)

팀원을 `plan` 모드로 스폰하면, 구현 전에 팀장 승인이 필요합니다:

```
// 팀원이 계획을 세우고 ExitPlanMode 호출
// → 팀장에게 자동으로 plan_approval_request 전송

// 팀장이 승인
SendMessage({
  type: "plan_approval_response",
  request_id: "plan-xyz",
  recipient: "backend",
  approve: true
})

// 팀장이 거부 + 피드백
SendMessage({
  type: "plan_approval_response",
  request_id: "plan-xyz",
  recipient: "backend",
  approve: false,
  content: "에러 핸들링 추가해주세요. 403 케이스가 빠져있어요"
})
```

## 6. 통신 흐름도 (실제 예시)

```
시간 ──────────────────────────────────────────────→

팀장:  [태스크 생성] ─── [DM: frontend] ─── [DM: backend] ─── [결과 수신] ─── [종합]
         │                    │                  │                  ↑
         │                    ↓                  ↓                  │
프론트:  │              [작업 시작] ──→ [DM: backend에 질문] ──→ [완료 보고]
         │                                       │
         │                                       ↓
백엔드:  │                             [작업 시작] ──→ [프론트에 답변] ──→ [완료 보고]
```

**핵심**: 팀원끼리 직접 질문/답변이 가능하므로, 팀장이 중계할 필요 없음!
