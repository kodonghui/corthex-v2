# 05. 협업 패턴 — 토론, 디베이트, 병렬 리뷰

## 팀 에이전트로 가능한 협업 패턴들

### 패턴 1: 가설 토론 (Hypothesis Debate) ⭐

**2명의 팀원이 서로 반대 입장을 갖고 토론하는 패턴**

```
┌──────────┐                        ┌──────────┐
│ 팀원 A    │ ─── "A 방식이 좋아" ──→ │ 팀원 B    │
│ (찬성)    │ ←── "아니 B가 낫지" ─── │ (반대)    │
│           │ ─── "근데 B는..." ───→ │           │
│           │ ←── "그건 이렇게..." ── │           │
└──────────┘                        └──────────┘
        │                                │
        └────── 양쪽 결론 ────→ 팀장 종합 ─┘
```

**사용 예시**: "React Server Components vs Client Components 중 어떤 걸 쓸까?"

```
팀장 프롬프트:

TeamCreate({ team_name: "arch-debate" })

팀원 A 스폰 시 프롬프트:
"너는 RSC 찬성 입장이야. 우리 프로젝트 구조를 분석하고
 RSC를 쓰면 좋은 이유를 구체적으로 정리해.
 팀원 B('opponent')가 반론하면 재반론해.
 3번 주고받으면 최종 의견을 팀장에게 보내."

팀원 B 스폰 시 프롬프트:
"너는 Client Components 찬성 입장이야. 우리 프로젝트 구조를 분석하고
 CC를 쓰면 좋은 이유를 구체적으로 정리해.
 팀원 A('proponent')가 주장하면 반론해.
 3번 주고받으면 최종 의견을 팀장에게 보내."
```

### 패턴 2: 병렬 코드 리뷰 (Multi-Lens Review)

**같은 코드를 서로 다른 관점에서 동시에 리뷰**

```
                    ┌── 보안 전문가 ──→ "SQL 인젝션 위험, XSS..."
                    │
 [같은 PR] ────────├── 성능 전문가 ──→ "N+1 쿼리, 메모리 누수..."
                    │
                    └── 테스트 전문가 ─→ "엣지 케이스, 커버리지..."

                            ↓ 3개 리뷰 종합 ↓

                    팀장: 우선순위 매기고 통합 리뷰 작성
```

**사용 코드**:

```
팀장:
1. TeamCreate({ team_name: "code-review" })
2. 보안 팀원 스폰 - "이 PR을 보안 관점으로만 리뷰해"
3. 성능 팀원 스폰 - "이 PR을 성능 관점으로만 리뷰해"
4. 테스트 팀원 스폰 - "이 PR을 테스트 커버리지 관점으로만 리뷰해"
5. 3개 결과 수신 후 종합
```

### 패턴 3: 탐색-구현 분업 (Scout-Builder)

**한 명은 조사, 한 명은 구현**

```
탐색자:  [라이브러리 조사] → [DM: 빌더에게 "이 라이브러리 써"] → [API 문서 정리] → [DM: 빌더]
                                    ↓
빌더:   [환경 세팅] ←──────── [조사 결과 받음] → [구현 시작] → [구현 완료]
```

### 패턴 4: 크로스-레이어 동시 개발

**프론트/백엔드/DB를 동시에 개발**

```
┌─────────────┐  "API 스키마 정했음"  ┌─────────────┐
│  프론트 팀원  │ ←───────────────── │  백엔드 팀원  │
│  React 컴포넌트│                    │  Hono 라우트  │
│  TanStack Query│ "타입 맞춰줘"     │  Drizzle 스키마│
│              │ ──────────────────→ │              │
└─────────────┘                      └─────────────┘
                       ↕ 공유 types
```

### 패턴 5: BMAD 파티 모드 (진짜 토론!) ⭐⭐

**현재 프로젝트에서 가장 유용할 패턴**

지금 BMAD 파이프라인은 **1명의 Worker가 혼자 3라운드 자기 검증**을 하는데,
팀 에이전트를 쓰면 **진짜 여러 에이전트가 토론**할 수 있습니다:

```
현재 방식 (가짜 파티):
┌────────────────────────────────────────┐
│  Worker 1명이 혼자서:                    │
│  Round 1: "PM 역할" → 피드백             │
│  Round 2: "Architect 역할" → 피드백      │
│  Round 3: "QA 역할" → 최종 판정          │
│  → 결국 같은 뇌가 역할극하는 것            │
└────────────────────────────────────────┘

팀 에이전트 방식 (진짜 파티):
┌──────────┐  ┌──────────┐  ┌──────────┐
│  PM 팀원   │  │ Arch 팀원 │  │ QA 팀원   │
│ (독립 뇌)  │  │ (독립 뇌)  │  │ (독립 뇌)  │
│            │  │           │  │           │
│ "UX 관점에서│  │ "기술적으로│  │ "테스트   │
│  이건 별로" │  │  이건 위험"│  │  커버 안됨"│
└──────┬─────┘  └─────┬─────┘  └─────┬─────┘
       │              │              │
       └──────────────┼──────────────┘
                      ↓
              팀장이 종합 판정
```

**왜 더 좋은가?**
- 각 에이전트가 **독립된 컨텍스트**를 가지므로 이전 역할의 편향이 없음
- 실제로 다른 "뇌"가 검토하므로 **맹점 발견** 확률 높음
- 서로 메시지를 주고받으며 **실제 논쟁** 가능

### 패턴 6: 경쟁 구현 (Race Pattern)

**같은 기능을 2가지 방식으로 동시 구현, 더 나은 것 채택**

```
팀원 A: zustand로 상태관리 구현
팀원 B: jotai로 상태관리 구현

→ 둘 다 완성 후 팀장이 비교
→ 번들 크기, 타입 안전성, DX 비교
→ 더 나은 것 채택, 나머지 삭제
```

### 패턴 7: 자기조직화 작업 처리 (Swarm)

**팀장이 태스크만 만들고, 팀원들이 알아서 가져가는 패턴**

```
팀장:
1. 15개 태스크 생성 (의존성 포함)
2. 5명 팀원 스폰 (프롬프트: "TaskList 확인하고 미배정 태스크 가져가서 완료해")
3. 앉아서 기다림

팀원들:
- 각자 TaskList 확인
- 미배정 + 미차단 태스크 클레임 (가장 낮은 ID 우선)
- 완료 → completed 표시 → 다음 태스크 자동 가져감
- 의존 태스크가 자동 언블로킹됨
- 질문 있으면 다른 팀원에게 DM

→ 팀장 개입 최소화!
```

## Hook 기반 품질 게이트

팀 전용 Hook 이벤트 3개:

### TeammateIdle Hook
```json
{
  "hooks": {
    "TeammateIdle": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "echo '팀원 idle 됐음, 다음 일 배정 가능'"
      }]
    }]
  }
}
```
- 팀원이 idle 상태가 되면 발동
- exit code 2 → 피드백 보내고 계속 작업하게 함
- 활용: 자동으로 다음 태스크 배정

### TaskCompleted Hook
```json
{
  "hooks": {
    "TaskCompleted": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "node scripts/verify-task.js"
      }]
    }]
  }
}
```
- 태스크 완료 표시 시 발동
- exit code 2 → 완료 거부 + 피드백 전송
- 활용: 테스트 통과, 린트 체크 등 자동 검증
