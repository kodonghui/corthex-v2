# 06. 실전 코드 예제

## 예제 1: 기본 팀 생성 및 작업 분배

사용자가 Claude Code에게 이렇게 말합니다:

```
"이 프로젝트에서 프론트엔드와 백엔드를 동시에 리팩토링해줘.
프론트는 React 19 패턴으로, 백엔드는 Hono RPC로 바꿔줘."
```

Claude Code(팀장)가 실행하는 흐름:

```
1. TeamCreate
   team_name: "refactor-team"
   description: "Frontend React 19 + Backend Hono RPC refactoring"

2. TaskCreate (여러 개)
   - Task 1: "Analyze current frontend patterns" (owner: frontend)
   - Task 2: "Analyze current backend routes" (owner: backend)
   - Task 3: "Refactor to React 19 use() pattern" (blockedBy: [1], owner: frontend)
   - Task 4: "Migrate to Hono RPC" (blockedBy: [2], owner: backend)
   - Task 5: "Update shared types" (blockedBy: [3, 4], owner: frontend)
   - Task 6: "Integration test" (blockedBy: [5])

3. Agent 스폰 (frontend 팀원)
   name: "frontend"
   team_name: "refactor-team"
   subagent_type: "general-purpose"
   prompt: "너는 프론트엔드 전문가야. refactor-team의 팀원이야.
            TaskList를 확인하고 너에게 배정된 태스크를 순서대로 처리해.
            backend 팀원과 타입 관련 협의가 필요하면 직접 메시지해.
            완료되면 TaskUpdate로 표시하고 다음 태스크를 가져가."

4. Agent 스폰 (backend 팀원)
   name: "backend"
   team_name: "refactor-team"
   subagent_type: "general-purpose"
   prompt: "너는 백엔드 전문가야. refactor-team의 팀원이야.
            TaskList를 확인하고 너에게 배정된 태스크를 순서대로 처리해.
            frontend 팀원과 API 스키마 관련 협의가 필요하면 직접 메시지해.
            완료되면 TaskUpdate로 표시하고 다음 태스크를 가져가."
```

## 예제 2: 토론/디베이트 팀

```
"Zustand vs Jotai 중 어떤 상태관리가 우리 프로젝트에 맞을지 분석해줘"
```

```
1. TeamCreate
   team_name: "state-debate"
   description: "Zustand vs Jotai analysis debate"

2. Agent 스폰 ("zustand-advocate")
   prompt: |
     너는 Zustand 전문가야. 우리 프로젝트(CORTHEX v2)의 코드베이스를 분석하고:
     1. 현재 상태관리 패턴 파악
     2. Zustand이 적합한 이유를 구체적으로 (코드 예시 포함)
     3. 'jotai-advocate' 팀원이 반론하면 재반박해
     4. 3라운드 토론 후 최종 의견을 팀장에게 보고

     토론 시작하려면 jotai-advocate에게 먼저 메시지 보내:
     "Zustand이 낫다고 생각하는 이유: [구체적 근거]"

3. Agent 스폰 ("jotai-advocate")
   prompt: |
     너는 Jotai 전문가야. 우리 프로젝트(CORTHEX v2)의 코드베이스를 분석하고:
     1. 현재 상태관리 패턴 파악
     2. Jotai가 적합한 이유를 구체적으로 (코드 예시 포함)
     3. 'zustand-advocate'가 주장하면 반박해
     4. 3라운드 토론 후 최종 의견을 팀장에게 보고

     zustand-advocate의 첫 메시지를 기다려.
     메시지 받으면 반론하고, 서로 3번 주고받아.
```

## 예제 3: BMAD 진짜 파티 모드 (기획 토론)

```
"PRD를 3명의 전문가가 검토하게 해줘"
```

```
1. TeamCreate
   team_name: "prd-review-party"

2. Agent 스폰 ("pm-reviewer")
   prompt: |
     너는 PM(Product Manager)이야.
     _bmad-output/planning-artifacts/prd.md를 읽고 제품 관점에서 리뷰해:
     - 사용자 여정에 빠진 게 없나?
     - 우선순위가 올바른가?
     - 시장 적합성은?

     발견한 이슈를 'architect-reviewer'와 'qa-reviewer'에게 공유하고,
     그들의 의견도 들어서 최종 리뷰를 팀장에게 보내.

3. Agent 스폰 ("architect-reviewer")
   prompt: |
     너는 소프트웨어 아키텍트야.
     _bmad-output/planning-artifacts/prd.md를 읽고 기술 관점에서 리뷰해:
     - 기술적으로 구현 가능한가?
     - 아키텍처 복잡도는 적절한가?
     - 빠진 기술 요구사항은?

     pm-reviewer, qa-reviewer와 의견 교환하고
     최종 리뷰를 팀장에게 보내.

4. Agent 스폰 ("qa-reviewer")
   prompt: |
     너는 QA 엔지니어야.
     _bmad-output/planning-artifacts/prd.md를 읽고 품질 관점에서 리뷰해:
     - 테스트 가능한 요구사항인가?
     - 엣지 케이스는 정의되었나?
     - 비기능 요구사항 (성능, 보안) 충분한가?

     pm-reviewer, architect-reviewer와 의견 교환하고
     최종 리뷰를 팀장에게 보내.
```

## 예제 4: 자기조직화 Swarm (대량 작업)

```
"20개 API 엔드포인트에 validation 추가해줘"
```

```
1. TeamCreate({ team_name: "validation-swarm" })

2. TaskCreate × 20개
   각각: "Add zod validation to POST /api/users", etc.
   의존성 없음 (독립적 태스크들)

3. Agent 스폰 × 4명 (worker-1 ~ worker-4)
   공통 프롬프트:
   "너는 validation-swarm 팀원이야.
    1. TaskList를 확인해
    2. 미배정(owner: null) 태스크 중 가장 낮은 ID를 가져가 (TaskUpdate로 owner 설정)
    3. 작업 완료하면 TaskUpdate로 completed 표시
    4. 다시 1번으로 돌아가서 다음 태스크 확인
    5. 모든 태스크가 완료되면 팀장에게 보고

    주의: 다른 팀원이 같은 파일을 수정할 수 있으니,
    파일 수정 전에 최신 상태를 다시 확인해."
```

## 예제 5: Git Worktree와 팀 결합

**각 팀원이 독립된 Git 브랜치에서 작업**

```
1. TeamCreate({ team_name: "feature-parallel" })

2. Agent 스폰 ("auth-dev")
   isolation: "worktree"    ← 독립된 Git Worktree에서 작업!
   prompt: "feature/auth 브랜치에서 인증 모듈 구현"

3. Agent 스폰 ("dashboard-dev")
   isolation: "worktree"    ← 다른 독립 Worktree!
   prompt: "feature/dashboard 브랜치에서 대시보드 구현"

→ 파일 충돌 없음! 각자 자기 브랜치에서 작업
→ 완료 후 팀장이 머지
```

## 예제 6: 플랜 승인이 있는 안전한 팀

**중요 작업은 팀장 승인 후 구현**

```
Agent 스폰 ("db-migrator")
   mode: "plan"             ← 계획만 먼저 세우게 함!
   prompt: "DB 마이그레이션 계획을 세워줘. 구현은 승인 후에."

→ 팀원이 계획 작성 → ExitPlanMode 호출
→ 팀장에게 plan_approval_request 자동 전송
→ 팀장이 계획 검토
→ 승인: plan_approval_response(approve: true) → 팀원이 구현 시작
→ 거부: plan_approval_response(approve: false, content: "이유") → 수정
```

## 프롬프트 작성 팁

### 좋은 팀원 프롬프트

```
"너는 {team-name}의 {role} 팀원이야.

할 일:
1. {구체적인 첫 번째 작업}
2. {구체적인 두 번째 작업}
3. 완료 후 TaskUpdate로 상태 변경

협업:
- {특정 팀원}에게 {어떤 상황}에 메시지 보내
- {특정 팀원}의 메시지를 받으면 {어떻게 대응}

완료 기준:
- {테스트 통과 여부}
- {구체적 결과물}

주의사항:
- {파일 충돌 방지}
- {권한 범위}"
```

### 나쁜 팀원 프롬프트

```
"알아서 해."     ← 뭘 해야 하는지 모름
"기다려."        ← 작업 없이 스폰하면 리소스 낭비
"전부 해."       ← 범위가 너무 넓어서 파일 충돌 위험
```
