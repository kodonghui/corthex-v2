# BMAD 프로젝트 생성 워크플로우
> 버그 최소화 + Party Mode 최대 활용 버전

---

## Party Mode 반복 원칙 (핵심)

```
단계 완료
    ↓
🎉 Party Mode 실행  (Claude가 자동 실행)
    ↓
에이전트 피드백 확인
    ↓
주요 반대 의견 있음? ──YES──→ 문서 수정 → 🎉 Party Mode 재실행
    ↓ NO
모든 에이전트 합의
    ↓
다음 단계로 진행
```

> **합의 기준:** 주요 반대 의견 0개 + 남은 의견이 모두 "사소한 것"으로만 남을 때

---

## 전체 흐름 요약

```
[0단계] 리서치
    ↓
[1단계] 아이디어 구체화  →  🎉 Party Mode #1 (합의까지 반복)
    ↓
[2단계] 요구사항 문서    →  🎉 Party Mode #2 (합의까지 반복)
    ↓
[3단계] 설계             →  🎉 Party Mode #3 (합의까지 반복)
    ↓
[4단계] 개발 준비 점검   →  🎉 Party Mode #4 (합의까지 반복)
    ↓
[5단계] 스토리 생성
    ↓
[6단계] 개발 루프         →  🎉 Party Mode #5 (Epic마다, 합의까지 반복)
    ↓
[7단계] 마무리 & 회고
```

---

## 0단계: 리서치 (선택, 새 도메인일 때)

> 모르는 분야라면 먼저 조사부터. 알고 있다면 건너뛰어도 됨.

| 순서 | 명령어 | 목적 |
|------|--------|------|
| 0-1 | `/domain-research` | "이 산업이 어떻게 돌아가는지" 조사 |
| 0-2 | `/market-research` | 경쟁사 분석, 고객 파악 |
| 0-3 | `/technical-research` | 쓸 기술이 적합한지 검증 |

---

## 1단계: 아이디어 구체화

| 순서 | 명령어 | 목적 |
|------|--------|------|
| 1-1 | `/brainstorming` | 아이디어 발산, 방향 잡기 |
| 1-2 | `/create-product-brief` | 아이디어를 제품 개요서로 정리 |

### 🎉 Party Mode #1 — Product Brief 검토

```
/party-mode
```

**목적:** PM + Analyst + UX Designer + Architect 가 Product Brief를 보고 한꺼번에 의견 제시

**체크포인트:**
- [ ] 타겟 사용자가 명확한가?
- [ ] 핵심 가치가 차별화되는가?
- [ ] 범위가 너무 크거나 작지 않은가?
- [ ] 기술적으로 실현 가능한가?

**피드백 받은 후 Product Brief 수정하고 다음으로**

---

## 2단계: 요구사항 문서 (PRD)

| 순서 | 명령어 | 목적 |
|------|--------|------|
| 2-1 | `/create-prd` | 상세 요구사항 문서 작성 |
| 2-2 | `/validate-prd` | PRD 빠진 내용 자동 검증 |
| 2-3 | `/edit-prd` | 검증에서 나온 문제점 수정 |
| 2-4 | `/review-adversarial-general` | 냉정한 비판적 시각으로 재검토 |

### 🎉 Party Mode #2 — PRD 심층 검토

```
/party-mode
```

**목적:** 모든 에이전트가 PRD를 읽고 구현 관점에서 문제 제기

**체크포인트:**
- [ ] 모호한 요구사항이 없는가?
- [ ] 서로 충돌하는 요구사항이 없는가?
- [ ] 엣지 케이스(예외 상황)가 빠지지 않았는가?
- [ ] 성능/보안 요구사항이 명시됐는가?

**피드백 반영 후 PRD 최종 확정**

---

## 3단계: 설계

| 순서 | 명령어 | 목적 |
|------|--------|------|
| 3-1 | `/create-architecture` | 기술 구조 설계 (DB, API, 서버 구성 등) |
| 3-2 | `/create-ux-design` | 화면별 UX/UI 상세 스펙 작성 |

### 🎉 Party Mode #3 — 설계 교차 검토

```
/party-mode
```

**목적:** Architecture + UX + Dev + QA 에이전트가 설계의 구현 가능성 및 충돌 여부 검토

**체크포인트:**
- [ ] 아키텍처와 UX 스펙이 서로 맞는가?
- [ ] 구현하기 어렵거나 불가능한 UX가 없는가?
- [ ] 데이터 모델이 모든 화면을 커버하는가?
- [ ] 성능 병목이 예상되는 곳이 있는가?
- [ ] 보안 취약점이 설계에 포함됐는가?

**피드백 반영 후 설계 확정**

## 4단계: 개발 준비 점검 (가장 중요한 게이트)

| 순서 | 명령어 | 목적 |
|------|--------|------|
| 4-1 | `/check-implementation-readiness` | 기획/설계가 코딩 시작 가능한 수준인지 자동 점검 |
| 4-2 | `/generate-project-context` | Claude가 프로젝트 이해하도록 컨텍스트 파일 생성 |

### 🎉 Party Mode #4 — 개발 시작 최종 승인

```
/party-mode
```

**목적:** 전체 에이전트가 "지금 코딩 시작해도 되냐?" 최종 확인

**이 게이트를 통과해야만 코딩 시작. 문제가 발견되면 해당 단계로 돌아가서 수정.**

**체크포인트:**
- [ ] PRD ↔ Architecture ↔ UX 스펙이 일관성 있는가?
- [ ] 모든 의존성(외부 API, 라이브러리)이 명확한가?
- [ ] 개발 환경 세팅이 완료됐는가?
- [ ] 1순위(P1) 기능의 스펙이 완전한가?

## 5단계: 스토리 생성

| 순서 | 명령어 | 목적 |
|------|--------|------|
| 5-1 | `/create-epics-and-stories` | 전체 기능을 Epic/Story로 분해 |
| 5-2 | `/sprint-planning` | 이번 스프린트 할 Story 선정 |
| 5-3 | `/create-story [story ID]` | 각 Story를 상세 작업지시서로 만들기 |

> **팁:** Story를 만들 때 너무 크게 잡지 말 것. 하나의 Story = Claude가 하루 안에 끝낼 수 있는 크기

---

## 6단계: 개발 루프 (반복)

```
create-story → dev-story → code-review → qa-generate-e2e-tests
      ↑                                              |
      └──────────── 다음 Story ─────────────────────┘
```

| 순서 | 명령어 | 목적 |
|------|--------|------|
| 6-1 | `/create-story` | 다음 Story 상세 지시서 작성 |
| 6-2 | `/dev-story` | 실제 코딩 실행 |
| 6-3 | `/code-review` | 작성된 코드 날카롭게 검토 |
| 6-4 | `/qa-generate-e2e-tests` | 자동화 테스트 생성 |

### 🎉 Party Mode #5 — Epic 완료마다 실행

```
/party-mode
```

**주기:** Epic(대기능 묶음) 하나가 끝날 때마다 실행

**목적:** 지금까지 만든 것들이 전체 방향과 맞는지, 다음 Epic에 영향 없는지 확인

**체크포인트:**
- [ ] 구현된 기능이 PRD 요구사항을 만족하는가?
- [ ] 예상치 못한 기술 부채가 쌓이지 않았는가?
- [ ] 다음 Epic 시작 전 수정할 것이 있는가?
- [ ] 성능/보안 이슈가 발생하지 않았는가?

## 버그 최소화를 위한 핵심 원칙

### ❌ 하면 안 되는 것
- Party Mode 통과 전에 다음 단계로 넘어가기
- PRD에 "나중에 결정" 남기고 개발 시작하기
- Story 크기를 너무 크게 잡기
- code-review 없이 다음 Story 진행하기

### ✅ 반드시 해야 하는 것
- 각 단계 피드백을 문서에 반영하고 나서 넘어가기
- 설계 변경이 생기면 PRD/Architecture/UX 동시 업데이트
- 테스트 코드는 Story 개발 직후 바로 생성
- 문서와 코드가 일치하는지 주기적 확인
