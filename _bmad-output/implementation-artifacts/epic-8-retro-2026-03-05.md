## 잘 된 점

1. **모든 빌드 첫 시도 성공** — 5개 스토리 모두 코드 작성 후 첫 `turbo build`에서 3/3 성공. 타입 오류나 import 누락 없이 깔끔한 구현
2. **tool-executor 대폭 간소화** — 352줄의 단일 파일을 HandlerRegistry 패턴으로 분리. 핸들러 추가 시 switch/case 수정 없이 `registry.register()` 한 줄로 가능
3. **ToolExecContext 확장 설계** — `Omit<ToolExecContext, 'getCredentials' | 'config'>` 패턴으로 호출자(ai.ts)와 실행자(tool-executor) 사이의 타입 호환성을 깔끔하게 해결
4. **보안 갭 발견 및 즉시 수정** — 8-5에서 tools.ts GET/:id, PUT/:id에 companyId 필터 누락 발견. `or(isNull, eq)` 패턴으로 platform 도구 접근은 허용하면서 타사 도구 접근은 차단
5. **P2 스키마 선제 준비** — Epic 9~13에 필요한 테이블을 미리 설계. nightJobSchedules, nightJobTriggers, files 테이블 + delegations 계층 확장
6. **Epic 7 기술 부채 해소** — "create_report 도구 DB 미등록" 문제가 8-1 시드에서 자연스럽게 해결됨

## 어려웠던 점 / 개선 필요

1. **테넌트 격리 사후 검증** — admin tools/credentials API의 companyId 누락이 Epic 2에서부터 존재했으나 8-5에서야 발견. 새 API 추가 시 격리 체크리스트 필요
2. **통합 테스트 실행 불가** — tenant-isolation.test.ts는 서버 실행이 필요한 통합 테스트라 CI에서 실행 불가. 격리 검증이 수동에 의존
3. **nightJobs FK 순서 제약** — Drizzle ORM에서 nightJobs가 nightJobSchedules보다 먼저 정의되어 `.references()` 사용 불가. relations로 대체했으나 DB 레벨 FK 제약 없음
4. **Admin UI 타입 안전성** — tools.tsx에서 `AppEnv` 타입 없이 `new Hono()` 사용하던 것을 8-5에서 수정. Hono 타입 제네릭 누락이 반복됨
