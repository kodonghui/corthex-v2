## 잘 된 점

### 1. 완벽한 의존성 체인 실행
- 17-1(DB/WS 인프라) > 17-2(읽기전용 캔버스) > 17-3(편집+실행) > 17-4(공유/복제) > 17-5(모바일) 순서가 자연스러운 기능 빌드업
- 각 스토리가 이전 인프라를 확장하며 재구현 없이 코드 재사용 극대화
- 특히 17-1의 Drizzle 스키마 + WS 채널이 17-2~17-3의 기반이 되어 개발 속도 향상

### 2. 기존 인프라 재활용률 극대화
- **React Flow**: nexus.tsx에서 이미 사용 중이던 라이브러리를 읽기전용(17-2) + 편집모드(17-3)로 확장
- **WebSocket 8채널 멀티플렉싱**: nexus 채널을 기존 agent-status 패턴과 동일하게 추가
- **PWA 인프라(Epic 16-5)**: Service Worker 캐시가 NEXUS 모바일에도 자동 적용
- **TanStack Query**: useQuery/useMutation 패턴 일관 사용
- **Dagre 자동 레이아웃**: 캔버스 좌표 fallback으로 재사용
- **파일 관리(Epic 13)**: 워크플로우 파일 입출력에 기존 API 활용 가능 기반 마련

### 3. PascalCase 컨벤션의 일관된 적용
- CLAUDE.md 예외 규칙(Nexus 컴포넌트만 PascalCase) 5개 스토리에서 일관 유지
- NexusInfoPanel, WorkflowListPanel, WorkflowEditor, ExecutionHistoryPanel 모두 PascalCase
- AgentNode, DepartmentNode, CompanyNode 기존 컴포넌트와 네이밍 일관성 유지

### 4. 서버-클라이언트 타입 안전성
- `packages/shared/src/types.ts`에 모든 새 타입(NexusGraphNode, NexusGraphEdge, NexusWorkflow, NexusExecution) 중앙 정의
- WsChannel union에 'nexus' 추가로 컴파일 타임 채널 검증
- Zod 스키마로 API 입력 유효성 검증 일관 적용

### 5. 모바일 반응형 패턴 확립
- 17-5에서 하단 슬라이드업 시트 패턴 최초 도입 (프로젝트 내 새 UI 패턴)
- 더보기(···) 드롭다운으로 모바일 툴바 복잡도 해결
- Epic 16-5의 모바일 패턴(hidden md:block, showChat 토글)을 NEXUS에 성공적 적용

## 교훈

### 1. 워크플로우 실행 엔진은 아직 Stub
- 17-3에서 실행 API가 즉시 completed 처리하는 stub으로 구현됨
- 실제 노드 간 데이터 흐름 엔진은 미구현 상태
- Epic 18(MCP) 이후에 실제 실행 엔진을 설계/구현해야 하는 기술 부채
- 사용자에게 "실행 완료(stub)" 메시지가 표시되는 것이 혼동을 줄 수 있음

### 2. 테스트 수가 스토리마다 편차가 큼
- 17-4(템플릿 공유): 50건으로 가장 적음 (신규 API 1개 + UI 변경 소폭)
- 17-1(DB 스키마): 121건으로 가장 많음 (3개 테이블 + WS 채널)
- TEA 리스크 기반 분석이 스토리 복잡도에 비례하여 자연스럽게 조절됨
- 다만, 17-4의 clone API 경계값 테스트가 더 필요할 수 있음

### 3. nexus.tsx 페이지 복잡도 증가
- 조직도 캔버스 + 탭 구조 + 워크플로우 목록/편집 + 모바일 반응형까지 한 파일에 집중
- Epic 16의 messenger.tsx(1100줄+) 교훈이 있었지만, nexus.tsx도 점차 복잡해지는 추세
- 워크플로우 관련 컴포넌트는 별도 파일로 분리(WorkflowListPanel, WorkflowEditor 등)하여 관리됨 — messenger.tsx보다 나은 구조

### 4. 기존 마이그레이션 활용으로 스키마 작업 최소화
- 17-1에서 마이그레이션 0019가 이미 존재하여 Drizzle 스키마 정의만 추가
- 새 마이그레이션 SQL 생성 불필요 — 개발 속도 향상에 기여
- 다만 마이그레이션 파일 28개+ 누적은 여전히 관리 과제

### 5. 워크플로우 편집의 UX 한계
- WorkflowEditor에서 노드 타입이 'default' 하나뿐 (향후 커스텀 노드 확장 필요)
- 노드 추가 시 prompt로 label 입력 — 모달 기반 UX로 개선 가능
- 실행 기록이 하단 패널에만 표시 — 실시간 실행 상태 추적 미구현
