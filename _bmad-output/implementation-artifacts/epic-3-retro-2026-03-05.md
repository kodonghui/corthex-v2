## 잘 된 점

1. **Admin 패턴 재사용으로 빠른 구현** — 3-1에서 Admin 앱의 로그인/에러 처리 패턴을 참조하여 개발 속도 크게 향상
2. **서버 API 변경 최소화** — 5개 스토리 중 서버 수정은 3-4의 email alias 1건뿐. 기존 API 활용도 높음
3. **점진적 리팩터링 성공** — settings.tsx가 3-4(기능 추가) → 3-5(탭 구조 분리)로 2단계에 걸쳐 자연스럽게 개선
4. **UX 스펙 추적 일관성** — 모든 스토리에서 UX 스펙 참조 번호(10.1, 10.3, 10.14 등)가 명확히 추적됨
5. **기존 코드 변경 최소화 원칙** — 탭 구조 분리 시 기존 섹션 내부 코드 변경 금지 원칙 적용, 회귀 위험 최소화

---

## 어려웠던 점

1. **Admin/App 코드 이중화** — errorMessages, RateLimitError, 401 redirect 패턴이 양쪽 앱에 복사-붙여넣기. 공유 라이브러리 없이 진행
2. **서버-프론트 네이밍 불일치** — 프론트 'email' vs 서버 'smtp' 문제로 서버에 alias 추가 필요
3. **모바일 반응형 반복 작업** — 사이드바(3-2), 그리드(3-3), 탭 스크롤(3-5)에서 매번 모바일 대응 필요

---

## 핵심 교훈

1. **공통 코드는 @corthex/shared로 올릴 것** — Admin/App 간 에러 매핑, API 클라이언트 패턴이 중복되고 있음. 차이가 벌어지기 전에 공유 패키지로 통합 필요
2. **서버 API 네이밍 컨벤션 통일** — provider 식별자(email vs smtp)가 프론트와 다르면 혼란 발생
3. **프론트 전용 스토리에서도 테스트 추가 필요** — 5개 스토리 중 새 테스트가 1건뿐이라 프론트 회귀 위험 있음
