## 잘 된 점

1. **admin-store 패턴 확립** — Zustand persist 미들웨어로 회사 선택 상태를 localStorage에 유지, 이후 모든 스토리에서 재사용
2. **N+1 쿼리 해결 (2-5)** — 에이전트마다 개별 API 호출 → toolId 기준 단일 API로 변경. 성능 최적화 패턴 확립
3. **토스트 시스템 통합 (2-8)** — 개별 스토리에서 임시 피드백을 넣던 것을 전역 토스트 store로 통일
4. **삭제 보호 패턴 일관 적용** — 회사(소속 직원 확인), 부서(소속 에이전트 확인) 모두 자식 엔티티 수 확인 → 0이 아니면 409
5. **보안 테스트 커버리지** — JWT 분리(admin vs user), 테넌트 격리, rate limit 모두 테스트 포함

---

## 어려웠던 점

1. **하드코딩 companyId 제거 반복** — `companyData?.data?.[0]?.id` 패턴이 거의 모든 페이지에 존재. 2-2~2-8 전체에 걸쳐 admin-store 연동으로 교체하는 작업이 반복됨
2. **토스트 알림 늦은 통합** — 2-2~2-7에서 각자 토스트를 만들다가 2-8에서 통일. 초기에 공통 컴포넌트로 만들었으면 반복 작업 줄일 수 있었음
3. **리뷰에서 반복적으로 테넌트 격리 누락 지적** — reset-password, terminate-session 모두 companyId 격리 누락이 리뷰에서 발견됨

---

## 핵심 교훈

1. **공통 UI 패턴(토스트 등)은 첫 스토리에서 바로 만들 것** — 나중에 통합하면 비일관성과 반복 작업 발생
2. **테넌트 격리는 API 작성 시 체크리스트로 관리** — 리뷰에서 발견되는 것보다 작성 시점에 잡는 게 효율적
3. **하드코딩 대신 store 패턴을 먼저 확립** — admin-store가 2-2에서 만들어진 후 나머지 스토리가 훨씬 수월해짐
