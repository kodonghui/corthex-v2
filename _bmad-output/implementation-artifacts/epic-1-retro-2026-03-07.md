### 1.2 잘한 점 (What Went Well)

**1. "기존 코드 읽고 확장" 전략이 모든 스토리에서 일관되게 적용되었습니다**

모든 7개 스토리의 Dev Notes에 "CRITICAL: 기존 코드 분석" 또는 "CRITICAL: Existing Code" 섹션이 포함되어 있었습니다. 기존 34+ 테이블 스키마, auth/tenant 미들웨어, credential-vault 서비스 등을 먼저 정확히 파악한 후 확장하는 접근이 일관되게 적용되었습니다.

> "기존 `packages/server/src/db/schema.ts`에 **이미 34개 테이블 + 관계가 정의됨**. 반드시 기존 코드를 읽고 확장할 것. 새로 만들지 말 것!"
> -- Story 1-1 Dev Notes

이 패턴은 Epic 0 회고에서 제안된 "갭 분석 먼저" 교훈이 잘 적용된 결과입니다.

**2. 스토리 간 인텔리전스 전달이 효과적이었습니다**

각 스토리의 Dev Notes에 "Previous Story Intelligence" 섹션이 포함되어 이전 스토리에서 배운 점을 다음 스토리에 전달했습니다:
- Story 1-2: Story 1-1의 테이블 패턴, 테스트 기준선 참조
- Story 1-3: Story 1-1의 audit_logs 테이블 활용, enum 추가 패턴 참조
- Story 1-4: Story 1-1/1-3의 스키마, 에러 코드 패턴 참조
- Story 1-5: Story 1-4의 AuditLogService, Story 1-2의 tenant context 참조
- Story 1-7: Stories 1-1~1-6 전체 컴포넌트 참조

이 "누적 학습" 패턴 덕분에 후반 스토리로 갈수록 구현이 빨라졌습니다.

**3. 보안 아키텍처가 다층(multi-layer)으로 구현되었습니다**

단일 보안 레이어가 아닌 3중 격리가 실제로 동작합니다:
- **미들웨어 레벨**: auth -> tenant -> rbac 체이닝 (JWT 검증 -> companyId 격리 -> 역할 접근 제어)
- **쿼리 레벨**: withTenant(), scopedWhere(), scopedInsert() 헬퍼로 모든 DB 쿼리에 companyId 자동 주입
- **감사 레벨**: 모든 중요 액션에 삭제 불가 audit_logs 기록 + RBAC 거부 시 자동 감사 로그

**4. BMAD 5단계 파이프라인이 실질적인 품질 향상을 가져왔습니다**

모든 스토리에서 create-story -> dev -> TEA -> QA -> code-review 순서가 지켜졌습니다:
- TEA가 리스크 기반 테스트를 자동 생성하여 커버리지 확대 (Story 1-5: 41 TEA 테스트, Story 1-6: 74 TEA 테스트)
- 코드 리뷰에서 실질적 이슈 발견 (Story 1-5: POST /api-keys에서 tenant.companyId 대신 body.companyId 사용하는 보안 취약점 수정)

**5. 감사 로그와 크리덴셜 볼트의 연계가 견고합니다**

Story 1-4(감사 로그)와 Story 1-5(크리덴셜 볼트 확장)의 연계가 깔끔합니다:
- 모든 크리덴셜 CRUD 작업에 audit log 자동 기록
- maskCredentialFields()로 NFR12(로그에 크리덴셜 평문 노출 금지) 완벽 준수
- withAuditLog() 래퍼로 비즈니스 로직과 감사 기록의 원자적 처리

### 1.3 개선할 점 (What Could Be Improved)

**1. 테스트 수 분포가 극단적으로 불균형합니다**

Story 1-6(시드 데이터)이 269건으로 전체 581건의 46%를 차지합니다. 시드 데이터의 각 에이전트 필드, Soul 키워드, 한국어 소통 검증 등 세부적인 데이터 검증 테스트가 대량으로 생성되었습니다.

반면 Story 1-1(스키마 확장)과 Story 1-4(감사 로그)는 각각 20건, 18건으로 상대적으로 적습니다. 핵심 인프라 코드에 비해 데이터 시드 검증에 테스트 리소스가 과도하게 집중된 감이 있습니다.

향후 에픽에서는 TEA에서 테스트 수량 상한을 설정하거나, 데이터 검증은 별도 유효성 검사 스키마(Zod 등)로 대체하는 것을 고려해야 합니다.

**2. 여전히 실제 DB 통합 테스트 환경이 없습니다**

Epic 0 회고에서 "통합 테스트 환경 구축"이 MEDIUM 우선순위 액션 아이템이었으나, Story 1-7에서 통합 테스트를 작성했음에도 실제 DB 연결 없이 로직 레벨에서만 검증하고 있습니다. Drizzle 쿼리 조건 생성은 검증하지만, 실제 PostgreSQL에서 companyId 격리가 동작하는지는 확인하지 못합니다.

이 부채가 Epic 2(Dynamic Org Management) 이후 실제 CRUD API가 늘어나면 더 큰 리스크가 됩니다.

**3. 역할 체계 마이그레이션이 부분적입니다**

Story 1-3에서 4역할 RBAC(super_admin, company_admin, ceo, employee) 체계를 구현했지만, 기존 라우트들은 아직 새 역할 체계로 전부 전환되지 않았습니다. departments.ts만 예시로 마이그레이션되었고, 나머지 라우트들은 기존 adminOnly/companyAdminOnly를 사용합니다. 이는 설계 의도(점진적 마이그레이션)이지만, Epic 2에서 반드시 완료해야 합니다.

### 1.4 배운 점 (Lessons Learned)

**1. 스키마 확장은 첫 번째 스토리로 배치하는 것이 효과적입니다**

Story 1-1에서 모든 Phase 1 테이블을 한꺼번에 정의한 덕분에, 후속 스토리들이 이미 존재하는 테이블 위에 서비스/미들웨어를 구축할 수 있었습니다. "스키마 먼저, 로직 나중에" 전략이 Epic 1에서 잘 동작했습니다.

**2. 미들웨어 체이닝 순서(auth -> tenant -> rbac)가 핵심 아키텍처 결정입니다**

이 순서가 각 레이어의 책임을 명확히 분리합니다:
- auth: JWT 검증 + TenantContext 주입 (실패 시 401)
- tenant: companyId 격리 + body 불일치 검증 (실패 시 403)
- rbac: 역할 기반 접근 제어 (실패 시 403 + audit log)

Story 1-7 통합 테스트에서 이 순서가 검증되었고, 모든 후속 에픽에서 이 체이닝을 따라야 합니다.

**3. 감사 로그의 INSERT-ONLY 설계가 단순하면서도 강력합니다**

updatedAt 칼럼 없음 + delete/update 메서드 미제공 + API에 DELETE/PUT 엔드포인트 없음. 이 3중 보호가 서비스/API/스키마 레벨에서 일관되게 적용되어, 별도 DB 트리거 없이도 불변성을 보장합니다.

**4. 코드 리뷰에서 보안 취약점을 잡는 것이 BMAD 파이프라인의 핵심 가치입니다**

Story 1-5 코드 리뷰에서 "POST /api-keys가 body.companyId를 사용 → tenant.companyId 우회 가능" 취약점이 발견되어 즉시 수정되었습니다. 이는 코드 리뷰 단계가 없었다면 프로덕션까지 갈 수 있었던 실질적 보안 이슈입니다.
