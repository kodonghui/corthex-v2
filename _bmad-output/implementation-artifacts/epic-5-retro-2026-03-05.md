## Part 1: Epic 리뷰

### 잘 된 점

1. **WebSocket 이벤트 패턴 성숙** — Epic 4에서 확립한 ws-store 리스너 패턴을 5-1(위임 이벤트)과 5-2(활동 로그 실시간)에 자연스럽게 확장. 패턴 재사용으로 구현 속도 향상.

2. **fire-and-forget + broadcast 조합** — activity-logger.ts에 `.returning()` + `broadcastToChannel()` 추가로, 기존 fire-and-forget 패턴을 깨지 않으면서 실시간 기능 추가. 비파괴적 확장의 좋은 사례.

3. **공유 컴포넌트 체계 강화** — Phase B(Epic 4-6)에서 3개(ConfirmDialog, Modal, Input)를 추가한 후 Phase C에서 4개(Tabs, Toggle, Textarea, Select)를 추가. @corthex/ui가 14 -> 18개 컴포넌트로 성장하면서 인라인 코드 대폭 감소.

4. **스트리밍 아키텍처 일관성** — 5-1에서 compileReport를 `client.messages.stream()`으로 전환하고 `hasStreamedTokens` 플래그로 중복 방지. Epic 4의 "스트리밍과 저장 상태 통합 설계" 교훈을 잘 적용.

5. **URL 상태 동기화 패턴 확립** — 5-2 필터 칩의 `?filter=chat,delegation` URL 동기화를 Set 기반으로 깔끔하게 구현. Epic 4의 "useSearchParams 설계 번복" 교훈 반영.

### 어려웠던 점

1. **기존 API 필드 누락 발견** — 5-2에서 start/end 매칭을 구현하려니 activity-log API가 eventId, phase를 반환하지 않았음. 스토리 계획 시 기존 API 스펙을 더 면밀히 확인 필요.

2. **테스트 환경 DB 연결 문제** — bun test 실행 시 ConnectionRefused로 140건 실패. DB 연결이 필요한 통합 테스트가 로컬에서 돌아가지 않는 상태 지속. (Epic 4에서도 지적된 이슈)

3. **UX 스펙 파일 크기** — ux-design-specification.md가 51,761 토큰으로 너무 커서 전체 로드 불가. Grep으로 필요한 섹션만 검색해야 했음. 문서 샤딩 필요.

### 이전 회고 액션 아이템 추적 (Epic 4 -> Epic 5)

| 액션 아이템 | 상태 | 비고 |
|------------|------|------|
| 테넌트 격리 체크리스트 적용 | ✅ 유지 | 새 API에도 companyId 필터 적용 |

**평가:** UI 관련 항목은 잘 이행했으나, 테스트/인프라 관련 부채는 3개 에픽째 미해결.

### 기술 부채

1. **StreamEvent 타입 서버/프론트 중복** — (Epic 4 이월) OrchestrateEvent 타입도 추가되어 중복 범위 확대
2. **DB 테스트 환경 미구축** — 140건 통합 테스트가 로컬에서 실패 지속
3. **프론트엔드 컴포넌트 테스트 부재** — 18개 UI 컴포넌트에 단위 테스트 없음
4. **UX 스펙 문서 과대** — 51K 토큰, AI 컨텍스트 윈도우 초과. 샤딩 필요
5. **Toggle 컴포넌트 미사용** — Phase C에서 생성했으나 아직 실제 사용처 없음 (향후 알림 설정 등에서 사용 예정)

### 핵심 교훈

1. **기존 API 스펙을 스토리 생성 시 검증할 것** — 5-2에서 필드 누락을 구현 중 발견. 스토리 생성 단계에서 API 응답 필드를 미리 확인했으면 방지 가능.

2. **테스트 인프라 부채는 누적될수록 위험** — 3개 에픽째 미해결. 다음 에픽에서 우선 처리 권장.

3. **문서 크기 제한 관리** — AI 기반 개발에서 대형 문서는 생산성을 떨어뜨림. 문서가 10K 토큰을 넘으면 샤딩 검토.
