- TaskList/TaskUpdate 기반 자기 조직화 — 워커가 알아서 다음 작업 선택

### 2. 의존성 관리
- 31개 스토리 간 복잡한 의존성 그래프를 blockedBy로 정확히 관리

- 병목점(16.4 Credential CRUD)이 해소되자 ~10개 하류 스토리 동시 해제

### 3. E8 경계 일관성
- 모든 새 코드가 engine/ 경계 규칙 준수
- MCP Manager: engine/ 내부 (E12), Built-in Tools: lib/ 외부 (E13)
- credential-crypto.ts(lib/), tool-cache.ts(lib/) vs semantic-cache.ts(engine/) 배치 정확

### 4. 8-Stage MCP Lifecycle (E12)
- RESOLVE→SPAWN→INIT→DISCOVER→MERGE→EXECUTE→RETURN→TEARDOWN
- agent-loop.ts에 RESOLVE 단계로 통합 — 기존 Hook 파이프라인과 자연스럽게 결합
- Lazy spawn + 120s idle teardown으로 리소스 효율 최적화

### 5. Security Audit (Epic 21) Go/No-Go 전체 통과
- Gate 1: Credential 암호화/복호화 round-trip 검증
- Gate 2: Scrubber가 20+ plaintext 패턴 차단
- Gate 3: MCP 8-stage lifecycle teardown leak 없음
- Gate 4: Telemetry start+end 정확한 durationMs 기록
- Gate 5: Multi-tenant 격리 (회사 A 자격증이 회사 B에 보이지 않음)
- Gate 6: save_report 부분 실패 격리 (한 채널 실패가 다른 채널 차단 안 함)

## 교훈

### 1. Swarm 워커의 BMAD 5-Skill 준수 문제 (Anti-Pattern #8)
- **문제**: 워커가 create-story와 QA 스킬을 건너뛰고 바로 코딩에 들어감
- **영향**: 26개 완료된 스토리에서 story 파일 0개 생성됨
- **원인**: Swarm Worker Prompt에 5-skill 순서가 명시적이지 않았음
- **해결**: Pipeline v5.4로 업그레이드 — 워커 프롬프트에 필수 스킬 순서 명시, Orchestrator가 story 파일 존재 검증
- **교훈**: 자동화 워커에게 "당연한" 절차도 명시적으로 써야 함. 암묵적 기대는 무조건 스킵됨

### 2. 리소스 좀비 축적 (Anti-Pattern #9)
- **문제**: tmux 78+ 패인, stale worktrees, 오래된 team 디렉토리 축적
- **원인**: 워커 종료/실패 시 정리 로직 없었음
- **해결**: hooks.json에 SessionStart/Stop cleanup 자동화. `.claude/hooks/cleanup.sh` 생성
- **교훈**: 자원 정리는 "규칙"이 아닌 "자동화"로 강제해야 함. CLAUDE.md에 적어도 안 지켜짐

### 3. 워커 idle/sleep 600s 문제
- **문제**: 워커가 할 일 없으면 30초마다 TaskList 폴링 → 결국 600s timeout
- **영향**: 사용자가 모바일에서 확인했을 때 "뭐하고 있는거야?" 혼란
- **교훈**: idle 워커는 즉시 종료하도록 프롬프트 개선 필요. "All tasks done" → 즉시 종료

### 4. AES-256-GCM 설계의 per-record IV+salt
- **성공**: 각 credential record마다 독립적인 IV와 salt 생성
- **결과**: 같은 값이어도 암호문이 매번 다름 — 패턴 분석 방지
- **교훈**: D23 결정이 정확했음. 보안 아키텍처 결정은 스토리 시작 전에 확정하면 구현 시 혼란 없음

### 5. Promise.allSettled 부분 실패 패턴 (E15)
- **성공**: save_report에서 pdf_email 채널이 실패해도 web_dashboard 채널은 성공
- **교훈**: 외부 서비스 연동은 항상 부분 실패를 가정하고 설계해야 함
