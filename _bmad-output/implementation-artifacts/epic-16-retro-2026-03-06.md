## 잘 된 점

### 1. 점진적 기능 빌드업의 교과서적 실행
- 16-1(채널 관리) > 16-2(실시간 WS) > 16-3(리액션/스레드) > 16-4(검색/알림) > 16-5(PWA) > 16-6(파일) 순서가 완벽한 의존성 체인
- 각 스토리가 이전 스토리의 인프라를 자연스럽게 확장하며, 재구현 없이 기존 코드 재사용
- 특히 WebSocket 인프라(16-2)가 이후 모든 스토리(리액션, 알림, push)의 기반이 됨

### 2. 코드 리뷰에서 보안 취약점 실제 발견
- **16-3 HIGH**: 채널 삭제 시 reactions FK 제약 위반 — 삭제 순서 수정
- **16-4 HIGH**: ILIKE 검색 와일드카드 이스케이프 미처리 — SQL injection 방지
- **16-4 MEDIUM**: `/channels/:id/read` 멤버십 검증 누락 — assertMember 추가
- **16-3 MEDIUM**: 리액션 POST catch가 모든 에러를 409 처리 — unique constraint만 409으로 분리
- 코드 리뷰가 실질적 보안/안정성 이슈를 잡아내는 핵심 안전망으로 입증됨

### 3. 기존 인프라 재활용률 극대화
- WebSocket 7채널 멀티플렉싱(Epic 1-9): messenger 채널 그대로 활용
- 파일 업로드 API(Epic 13): 16-6에서 동일 API 재사용
- AI 에이전트 응답(Epic 4): getClientForUser + claude-sonnet-4-5로 메신저 내 AI 호출
- 알림 시스템(Epic 6): notifications 테이블 + WS push 재사용
- 채팅 모바일 패턴(Story 4-7): showChat 상태 토글 패턴 그대로 적용
- PWA 기본 인프라(Story 1-8): manifest/SW 등록 코드 확장

### 4. 테스트 커버리지 대폭 확대
- Epic 16에서만 608건의 TEA 테스트 추가 (전체 테스트 1854건+)
- 16-5 PWA 스토리에서 172건으로 가장 많은 테스트 (캐시 전략, push, 오프라인 등 복잡한 시나리오)
- TEA 리스크 기반 분석으로 경계값/동시성/멀티테넌트 격리까지 커버

### 5. 인라인 컴포넌트 패턴의 일관성
- messenger.tsx 1100줄+ 단일 파일에 모든 UI 컴포넌트(설정 모달, 스레드 패널, 검색, 리액션, 파일 첨부) 통합
- 파일 분리 없이도 기능별 컴포넌트(ChannelSettingsModal, ThreadPanel, AttachmentRenderer 등)로 논리적 분리
- 6개 스토리가 동일 파일을 순차 수정했지만 충돌 없이 깔끔하게 확장

## 교훈

### 1. messenger.tsx 파일 크기 관리
- 1100줄+ 단일 파일이 되면서 가독성/유지보수 부담 증가
- 인라인 패턴이 개발 속도에는 유리했지만, 향후 리팩토링 고려 필요
- 컴포넌트별 파일 분리 시점 기준: 단일 파일 1500줄 초과 시 검토

### 2. DB 마이그레이션 누적
- Epic 16에서 마이그레이션 3개 추가 (0025: reactions/threads, 0026: search/unread, 0027: push, 0028: file-attach)
- 마이그레이션 파일 28개 누적 — 프로덕션 배포 시 마이그레이션 스쿼시 고려

### 3. Web Push 인프라 복잡도
- VAPID 키 관리, pushSubscriptions 테이블, SW push 이벤트 등 인프라가 상당히 복잡
- 현재 테스트는 유닛 테스트만 — 실제 push 발송 E2E 테스트는 미구현
- push 실패 시 재시도/fallback 로직 부재 (410 Gone만 처리)

### 4. ILIKE 검색 성능 한계
- 현재 `ILIKE '%keyword%'` 방식은 인덱스 활용 불가
- 메시지 수 증가 시 full-text search (tsvector/tsquery) 전환 필요
- 현재 규모에서는 충분하나, 수만 건 이상 시 성능 저하 예상

### 5. 파일 크기 제한 변경의 영향
- 16-6에서 파일 업로드 제한을 10MB > 50MB로 확장
- 이 변경이 채팅 파일 첨부(Epic 13)에도 영향 — 의도된 변경이지만 문서화 필요
- 서버 메모리/디스크 사용량 모니터링 중요

- **AttachmentRenderer**: 이미지 인라인 썸네일 + 문서 카드 UI
