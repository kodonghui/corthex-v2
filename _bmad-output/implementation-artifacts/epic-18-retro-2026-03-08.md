## 잘 된 점

### 1. Gemini 코드 완전 교체 성공
- 18-1에서 기존 Gemini가 작성한 워크플로우 코드를 처음부터 재작성
- 품질이 낮았던 기존 코드 3개 파일(engine.ts, workflows.ts, 테스트 3개) 완전 교체
- DB 스키마(마이그레이션 완료)는 유지하면서 코드만 재작성하는 판단이 정확했음
- 결과: DAG 검증, Zod 스키마, RBAC까지 포함한 견고한 기반 구축

### 2. 모듈 분리 아키텍처
- 18-1의 CRUD(services/workflow/engine.ts)와 18-2의 실행 엔진(lib/workflow/)을 명확히 분리
- DAGSolver, ExecutionContext, WorkflowEngine 각각 독립 클래스로 설계
- validateDag(검증용)과 DAGSolver.resolveTiers(실행용)가 유사 로직이지만 목적이 달라 의도적 분리
- 이 분리 덕분에 18-4에서 buildDagLayers를 클라이언트 측에 독립 함수로 추출 가능

### 3. 서버-클라이언트 API 재활용
- 18-4, 18-5 UI 스토리에서 서버 수정이 전혀 불필요
- 18-1~18-3에서 만든 API가 충분히 완성되어 UI에서 바로 사용 가능
- API 설계 시 프론트엔드 요구사항까지 고려한 결과

### 4. v1 패턴의 성공적 확장
- v1 순차 실행(`{prev_result}`)을 v2 DAG 기반(`{{stepId.property}}`)으로 확장
- v1 `asyncio.gather()` 병렬을 v2 `Promise.all()` 레이어 병렬로 확장
- v1 지수 백오프 retry 패턴 그대로 적용
- v1에 없던 condition 분기(trueBranch/falseBranch) 추가

### 5. BMAD 파이프라인 안정적 작동
- 5개 스토리 모두 create-story → dev → TEA → QA → CR 5단계 완주
- CR에서 발견된 이슈(addToast 시그니처, invalidation key)가 즉시 수정됨
- TEA가 UI-only 스토리(18-5)에서 "테스트 불필요" 판정을 적절히 내림

## 교훈

### 1. addToast API 시그니처 불일치 (18-4에서 13개 오류)
- toast-store의 `addToast({ type, message })` 객체 인자를 `addToast('type', 'message')` 위치 인자로 잘못 호출
- 13곳에서 동일 실수 반복 — 한 곳에서 패턴을 잘못 잡으면 전체에 전파됨
- **교훈**: 기존 store API 시그니처를 먼저 확인하고, 첫 번째 사용처에서 TS 컴파일 확인 후 나머지 작성

### 2. stub 실행기의 기술 부채
- 18-2에서 tool/llm 스텝 실행기가 stub으로 구현됨 (실제 ToolPool/AgentRunner 미연동)
- Epic 17 회고에서도 "워크플로우 실행 엔진 stub" 이슈를 지적했는데, 18-2에서 엔진 자체는 구현했으나 실제 도구 호출은 여전히 stub
- 실제 프로덕션 사용을 위해서는 ToolPool + AgentRunner 통합이 필요한 상태
- 다만, 이 접근은 의도적: 순수 엔진 로직을 먼저 검증하고, 통합은 별도로 수행

### 3. 단일 파일 복잡도 관리 (workflows.tsx ~735줄)
- 18-4에서 WorkflowsPage, WorkflowEditor, StepForm, DagPreview를 모두 한 파일에 구현
- 18-5에서 ExecutionHistory, ExecutionDetail까지 추가되어 파일이 더 커짐
- Epic 17 회고에서 nexus.tsx 복잡도 증가를 지적했는데, 동일 패턴 반복
- **교훈**: Admin 앱 페이지도 컴포넌트 분리가 필요 (Epic 20-S4 노코드 빌더 시 분리 검토)

### 4. TypeScript unknown 타입 JSX 렌더링
- 18-5에서 `step.output`이 `unknown | null` 타입이어서 JSX에서 직접 렌더링 불가
- `typeof` 체크 + `JSON.stringify` 조합으로 해결
- **교훈**: API 응답의 `any`/`unknown` 필드는 UI에서 타입 가드 필수

### 5. react-query invalidation key 일관성
- 18-5 CR에서 발견: executeMut의 invalidation이 `['workflow-executions']`로 워크플로우 ID 미포함
- ExecutionHistory의 queryKey는 `['workflow-executions', workflowId]`로 ID 포함
- 결과: 실행 후 이력이 자동 갱신되지 않는 버그
- **교훈**: react-query queryKey는 데이터를 유일하게 식별하는 모든 파라미터를 포함해야 함
