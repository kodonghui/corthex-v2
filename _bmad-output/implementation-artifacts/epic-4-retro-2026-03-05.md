## 잘 된 점

1. **ws-store 이벤트 리스너 패턴 확립 (4-2)** — addListener/removeListener 추가로 컴포넌트별 WS 이벤트 구독이 가능해짐. 이후 모든 실시간 기능의 기반
2. **tool_calls API 분리 (4-3)** — 메시지 텍스트에 도구 정보를 끼워 넣는 방식 → 별도 API로 분리하여 UI 유연성 확보
3. **공유 UI 컴포넌트 체계 (4-6)** — ConfirmDialog/Modal/Input 통합으로 인라인 구현 반복 제거. Epic 2 교훈("공통 패턴 선행 구현") 실천
4. **cursor 페이지네이션 (4-4)** — offset 대비 대화 삽입/삭제에 안정적인 cursor 방식 채택
5. **비파괴적 리팩터링 (4-6)** — 기능 변경 없이 디자인 시스템 일관성 확보. 회귀 위험 최소화

---

## 어려웠던 점

1. **스트리밍↔저장 상태 통합 (4-2→4-3)** — 스트리밍 중 보이던 도구 카드가 done 후 사라지는 버그. tool_calls API 분리로 해결했지만 디버깅에 시간 소요
2. **컴포넌트 분리 복잡도 (4-1)** — chat.tsx 366줄 단일 파일을 SessionPanel + ChatArea + AgentListModal 3개로 분리하면서 타입/상태/이벤트 흐름 재설계 필요
3. **설계 번복 (4-1→4-4)** — 4-1에서 useSearchParams 불필요로 판단 → 4-4에서 URL 동기화 필요해져 도입. 초기에 요구사항을 더 넓게 봤으면 방지 가능
4. **모바일 hover 불가 (4-1→4-7)** — hover 기반 UI를 만들고 나중에 터치 대응으로 수정. 처음부터 터치 우선 설계했으면 재작업 불필요

---

## 핵심 교훈

1. **스토리 간 의존성을 미리 파악할 것** — 4-1의 useSearchParams 판단이 4-4에서 번복됨. 에픽 전체 스토리를 먼저 훑어보고 설계하면 방지 가능
2. **모바일 우선 설계** — hover 기반 UI를 나중에 터치로 바꾸는 것보다 처음부터 터치 우선으로 만드는 게 효율적
3. **스트리밍과 저장 상태를 초기에 통합 설계** — 실시간 데이터와 DB 데이터의 전환 시점 UI를 미리 계획해야 "깜빡임" 버그 방지
