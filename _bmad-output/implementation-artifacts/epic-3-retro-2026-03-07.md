## 잘 된 점

1. **레이어드 빌드 패턴 확립** -- 어댑터(3-1) -> 라우터(3-2) -> fallback(3-3) -> AgentRunner(3-4) -> 비용(3-5) -> 배치(3-6) -> 통합 테스트(3-7) 순서로 깔끔하게 적층. 각 스토리가 이전 스토리의 코드를 안정적으로 재사용
2. **"기존 코드 확장, 재생성 금지" 원칙 일관 적용** -- 모든 스토리에 "CRITICAL: Existing Code to Reuse (DO NOT recreate)" 섹션이 있어서 중복 코드 방지. cost-tracker.ts, credential-vault.ts 등 Epic 1 코드를 성공적으로 확장
3. **제로 리그레션 달성** -- 7개 스토리 연속 구현에도 기존 테스트 깨짐 없음. 각 스토리 완료 시 관련 테스트 전체 통과 확인 루틴이 효과적
4. **Anti-Pattern Prevention 문서화** -- 스토리마다 10개 내외의 "하지 말 것" 목록 작성. 특히 "vitest/jest 대신 bun:test", "API 키 노출 금지", "fire-and-forget 패턴 유지" 등이 반복 방지에 효과적
5. **크레덴셜 보안 패턴 강화** -- AgentRunner의 6종 크레덴셜 패턴 스크러빙(sk-*, AIza*, Bearer, API_KEY=, SECRET=, PRIVATE KEY) + LLM 에러 메시지 내 키 마스킹(sk-***) 구현
6. **models.yaml 중앙 설정** -- 모델 가격, 계급별 기본 모델, fallback 순서를 하나의 YAML 파일로 관리. 라우터/비용추적/배치 모두 이 설정 참조 -> 일관성 확보

---

## 어려웠던 점

1. **CredentialVault 프로바이더 키 불일치** -- models.yaml은 `google`을 사용하지만 CredentialVault는 `google_ai`를 사용. 3-2에서 발견하여 `toCredentialProvider()` 매핑 함수로 해결했으나, 이런 불일치가 다른 곳에서도 발생할 수 있음
2. **AgentRunner iterations 카운트 버그** (3-4) -- 코드 리뷰에서 도구 호출 횟수를 toolRecords.length로 세던 것을 실제 루프 카운트로 수정. 도구 호출이 여러 개인 경우 카운트가 부풀려지는 문제였음
3. **Batch API 비동기 폴링** (3-6) -- Anthropic/OpenAI Batch API가 모두 비동기 폴링 패턴(10초 간격)이라 테스트 작성이 복잡. mock 타이머 + 즉시 완료 패턴으로 해결
4. **스트리밍 fallback 복잡도** (3-3) -- 스트림 중간 실패 시 non-streaming fallback으로 전환하는 로직이 예상보다 복잡. pre-chunk 실패와 mid-chunk 실패를 분리 처리해야 했음

---

## 핵심 교훈

1. **레이어드 스토리 순서가 핵심** -- 하위 레이어(어댑터)부터 상위(통합 테스트)까지 순서대로 빌드하면 각 스토리가 자연스럽게 이전 코드를 활용. Epic 4(도구 시스템)에도 이 패턴 적용 가능
2. **Anti-Pattern Prevention 목록이 품질 보호** -- "하지 말 것"을 명시하면 코드 리뷰에서 잡을 문제를 작성 시점에 방지. Epic 4 스토리에도 반드시 포함할 것
3. **중앙 설정 파일(models.yaml)의 가치** -- 여러 컴포넌트가 공유하는 설정을 한 곳에서 관리하면 일관성 유지 + 변경 용이. Epic 4의 도구 카탈로그에도 유사 패턴(tools.yaml 또는 ToolPool 레지스트리) 권장
4. **통합 테스트 마지막 스토리 패턴 효과적** -- 개별 단위 테스트 외에 마지막 스토리(3-7)에서 전체 파이프라인을 검증하는 것이 버그 발견에 매우 효과적. 83개 통합 테스트가 컴포넌트 간 인터페이스 계약을 검증
