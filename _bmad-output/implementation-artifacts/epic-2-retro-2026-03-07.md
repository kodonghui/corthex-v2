## Part 1: 에픽 리뷰

### 1.1 에픽 요약

### 1.2 잘한 점 (What Went Well)

**1. OrganizationService 서비스 레이어 패턴이 에픽 전체에서 일관되게 성장했습니다**

Story 2-1에서 `packages/server/src/services/organization.ts`를 신규 생성하고, 이후 모든 백엔드 스토리(2-2, 2-3, 2-4)가 동일 파일에 함수를 누적 추가했습니다. 결과적으로 하나의 서비스 파일이 부서 CRUD, 에이전트 CRUD, cascade 분석/실행, 템플릿 적용까지 모든 조직 관리 비즈니스 로직을 담당하는 단일 진입점이 되었습니다.

> "Do NOT create a separate cascade service file -- add to existing organization.ts"
> -- Story 2-3 Anti-Patterns

이 패턴 덕분에 라우트 핸들러는 얇고(thin), 비즈니스 로직은 서비스에 집중되어 테스트와 유지보수가 용이합니다.

**2. Story 간 Intelligence 전달이 Epic 1보다 더욱 성숙해졌습니다**

모든 스토리에 "Previous Story Intelligence" 섹션이 포함되어 이전 스토리의 구체적 패턴(return 형식, TenantActor 인터페이스, actorType() 헬퍼, throwIfError() 패턴 등)을 명시적으로 전달했습니다. 이는 Epic 1 회고에서 칭찬한 패턴이 더 발전한 결과입니다.

**3. 보안 취약점이 Story 2-2에서 즉시 발견/수정되었습니다**

Agent CRUD 라우트에 **tenantMiddleware가 완전히 누락**되어 있었습니다. 이는 companyId를 body에서 직접 받아 테넌트 격리가 우회될 수 있는 심각한 보안 갭이었습니다. Story 2-2에서 이를 발견하고 즉시 수정했습니다:
- tenantMiddleware 적용
- body.companyId 대신 tenant.companyId 사용
- 쿼리 파라미터 기반 companyId도 tenant context로 교체

**4. Cascade Analysis Engine(2-3)이 아키텍처 설계를 충실히 구현했습니다**

Architecture Decision #5의 CascadeAnalysis 인터페이스를 정확히 구현하고, 3가지 삭제 모드(force/wait_completion/immediate)로 관리자에게 유연한 선택지를 제공합니다. 데이터 보존 원칙(cost_records 영구 보존, department_knowledge 아카이브, 에이전트 soft-deactivation)이 일관되게 적용되었습니다.

**5. 백엔드(2-1~2-4)와 프론트엔드(2-5~2-9) 분리가 깔끔했습니다**

Stories 2-5~2-9(UI 스토리)에서 서버 코드를 전혀 변경하지 않았습니다. 백엔드 API가 이미 완전히 구현되어 있어 프론트엔드는 순수하게 UI 구현에만 집중할 수 있었습니다. 이 "API-first" 접근이 스토리 간 의존성을 최소화했습니다.

**6. 경량 마크다운 렌더러를 외부 의존성 없이 구현했습니다**

Story 2-7의 Soul 편집기에서 외부 마크다운 라이브러리(marked, react-markdown 등) 없이 인하우스 경량 파서를 구현했습니다. heading, bold, italic, list, code block 등 기본 문법을 지원하면서 XSS 방지도 포함. 의존성 최소화 원칙을 잘 지켰습니다.

### 1.3 개선할 점 (What Could Be Improved)

**1. UI 컴포넌트 코드 중복이 발생했습니다**

Story 2-5(Admin org-chart)와 Story 2-9(CEO org)에서 AgentNode, DepartmentSection, AgentDetailPanel, OrgChartSkeleton 등 거의 동일한 컴포넌트를 **복사-수정** 방식으로 두 앱에 각각 구현했습니다. Story 2-9 Dev Notes에서 인정한 대로 "코드량이 적어 중복 비용이 낮다"고 판단했지만, 향후 UI 변경 시 두 곳을 동시에 수정해야 하는 유지보수 부담이 있습니다.

공통 UI 컴포넌트를 `packages/ui`로 추출하는 것을 향후 고려해야 합니다.

**2. userId 처리 방식이 에이전트 생성 시 혼란스러웠습니다**

Story 2-7 Dev Notes에서 지적된 대로, agents 테이블의 `userId` (NOT NULL FK)가 에이전트 생성 시 문제가 됩니다:
- 서버 `createAgentSchema`는 userId를 body에서 필수로 요구
- Admin UI에서 이 userId를 어떻게 제공할지 명확하지 않았음
- 결국 auth-store에서 현재 로그인한 admin의 userId를 사용하는 방식으로 해결

이 부분은 향후 "인간 직원 1명 = OAuth CLI 1개" 아키텍처(Epic 9)에서 재설계가 필요합니다.

**3. 실제 DB 통합 테스트 환경이 여전히 없습니다**

Epic 1 회고에서 HIGH 우선순위로 지정된 "실제 DB 통합 테스트 환경 구축"이 Epic 2에서도 해결되지 않았습니다. 9개 스토리 모두 DB mock 기반 단위 테스트만 사용합니다. CRUD API가 대폭 증가한 상태에서 실제 PostgreSQL 쿼리 동작은 검증되지 않고 있습니다.

**4. 드래그 앤 드롭 대신 Select 방식으로 대체했습니다**

UX 설계에서는 에이전트를 드래그하여 부서 간 이동하는 것이 기획되었으나, Story 2-8에서 dnd-kit 의존성 추가 없이 Select 드롭다운 방식으로 대체했습니다. 기능적으로 동일하지만 UX 경험은 설계 의도와 다릅니다. 향후 UX 향상 스토리에서 드래그 앤 드롭을 추가할 수 있습니다.

### 1.4 배운 점 (Lessons Learned)

**1. "기존 코드 강화" 패턴이 Epic 2의 성공 열쇠였습니다**

모든 9개 스토리에서 "Existing Code (MUST build on, NOT rewrite)" 원칙이 적용되었습니다. 특히:
- Story 2-1: 기존 departments.ts 라우트에 GET /:id, audit logging, uniqueness 검증 추가
- Story 2-2: 기존 agents.ts 라우트에 tenantMiddleware 추가 + 서비스 레이어 전환
- Story 2-5~2-8: 기존 UI 페이지를 카드 그리드 -> 테이블 뷰로 점진적 리팩터링

이 접근 덕분에 기존 동작하는 코드를 깨뜨리지 않으면서 기능을 확장할 수 있었습니다.

**2. "API-first" 개발 순서가 UI 스토리의 속도를 극대화합니다**

Stories 2-1~2-4에서 모든 API를 먼저 완성한 후, Stories 2-5~2-9에서 UI만 구현하는 순서가 효과적이었습니다. UI 스토리들은 서버 코드 변경 없이 순수 프론트엔드 작업만으로 완료되었습니다. 이 패턴을 후속 에픽에서도 유지하는 것이 좋습니다.

**3. Cascade 삭제의 "모드 선택" 패턴은 관리자 경험에서 핵심입니다**

단순 "삭제하시겠습니까?"가 아닌, cascade 영향 분석을 먼저 보여주고 모드(완료 대기/강제 종료)를 선택하게 하는 패턴이 관리자에게 안전한 의사결정을 지원합니다. 이 패턴은 향후 다른 삭제 작업(에이전트 일괄 비활성화, 회사 삭제 등)에도 적용할 수 있습니다.

**4. 테이블 뷰가 카드 그리드보다 관리 UI에 적합합니다**

Stories 2-6, 2-7에서 기존 카드 그리드를 테이블 뷰로 리팩터링했고, 필터/검색/정렬 기능 추가가 테이블에서 훨씬 자연스러웠습니다. Admin 관리 화면은 "데이터 밀도"가 중요하므로 테이블 뷰가 기본 패턴이 되어야 합니다.

**5. 경량 인하우스 구현이 의존성 추가보다 낫습니다**

마크다운 렌더러(2-7)와 부서 이동 UI(2-8 -- dnd 대신 Select)에서 외부 라이브러리 없이 구현했습니다. 번들 크기와 의존성 관리 부담을 줄이면서 필요한 기능만 정확히 구현할 수 있었습니다.

  1. 4-1: ToolPool Registry + Zod Framework

**회고 완료**: Epic 2 Dynamic Organization Management는 9개 스토리, 23 SP, ~580건+ 신규 테스트로 성공적으로 완료되었습니다. 부서/에이전트 CRUD API, cascade 분석 엔진, 조직 템플릿 적용, Admin 관리 UI(조직도/부서/에이전트/Soul 편집/템플릿), CEO 앱 조직도 읽기 전용 뷰가 모두 실제 동작하는 코드로 구현되었습니다. OrganizationService 서비스 레이어와 API-first 개발 패턴이 에픽 전체의 일관성과 품질을 보장했습니다. 이 기반 위에 Epic 4 (Tool System)를 진행합니다.
