## 7-Expert Debate

### PM (John) -- 핵심 루프 정의의 편향

"핵심 루프가 '명령 입력 -> 위임 관찰 -> 보고서 수신'으로 정의됐다. 사령관실 중심이다. 좋다. 하지만 **v2의 핵심 차별점은 '동적 조직 관리'라고 Executive Summary에서 반복 선언했는데, 조직 설계 루프는 '부차 경험 루프'로 밀려났다**."

"이게 문제인 이유: Priority Matrix에서 박과장에게 동적 조직 관리가 '핵심'이다. 김대표에게도 '필수'다. **P0 기능인 동적 조직 관리가 '부차 경험'이면, UX 설계 시 리소스 할당에서 밀린다**."

"제안: 핵심 루프를 2개로 정의하거나, 최소한 '부차 경험'이 아닌 '2차 핵심 루프'로 격상해야 한다. 또는 **핵심 루프가 1개인 이유를 명시적으로 정당화**해야 한다 -- '일일 사용 빈도 기준으로 사령관실이 1순위이므로 핵심 루프로 지정한다'처럼."

### Architect (Winston) -- "3패널 레이아웃"의 갑작스러운 등장

"Platform Strategy에서 **'사령관실 3패널 레이아웃'이라는 용어가 처음 등장**한다 (line 445). Discovery에서는 '채팅형 입력 + 위임 체인 패널 + 보고서 뷰'라고 3개 영역을 나열했지만, '3패널 레이아웃'이라는 구조적 결정은 한 적이 없다. 이건 Wireframe 단계에서 결정할 사항인데, Core Experience 섹션에서 마치 이미 결정된 것처럼 언급했다."

"또한 **핵심 루프 2단계 '위임 관찰'에서 WS 채널이 'command + delegation + tool'로 표기**됐는데, 이건 Discovery에서 이미 지적한 architecture.md 7채널(agent-status, task-progress, cost-update 등)과의 불일치가 여전히 있다. 이 단계에서도 동일 표기를 쓰면 불일치가 문서 전체에 퍼진다."

"John에 동의하지 않는 부분이 있다. **핵심 루프가 1개인 건 맞다**. 사령관실이 매일 쓰는 화면이고, 조직 설계는 주간이다. 핵심 루프 = 매일 반복하는 가장 빈번한 경험이어야 한다. 다만 '부차'라는 표현이 약한 건 동의한다."

### UX Designer (Sally) -- 대기 시간의 UX가 비어있다

"핵심 루프에서 **가장 중요하고 가장 빠져있는 것은 '대기 시간의 경험'**이다. First-Time Success 경로에서 '위임 체인 실시간 관찰 (2~3분)'이라고 했다. **비개발자 CEO가 2~3분 동안 화면을 보고 있어야 한다.** 그 시간 동안 뭘 보는가?"

"위임 체인이 실시간 업데이트된다고 했지만, **업데이트 빈도가 어떻게 되는가?** LLM API 호출은 10~30초가 걸린다. 그 사이에는? 위임 체인 패널에 '종목분석가 분석 중...'만 표시되고 2분 동안 아무 변화가 없으면, 사용자는 '멈춘 건가?' 하고 불안해진다."

"**필요한 것:**"
"1. 위임 체인 각 단계의 예상 소요 시간 표시 (예: '~30초')"
"2. 진행 중 에이전트의 세부 상태 (예: '데이터 수집 중', '분석 중', '초안 작성 중') -- v1의 자율 딥워크 5단계와 매칭"
"3. 부분 결과 먼저 표시 가능성 (첫 에이전트가 완료되면 바로 표시)"
"4. 스켈레톤/로딩 상태 패턴 정의"

"Winston의 3패널 지적에 추가한다 -- **3패널이 아니라, 명령 처리 중과 결과 수신 후의 화면 상태가 달라야 한다**. 처리 중에는 위임 체인이 주인공이고, 완료 후에는 보고서가 주인공이다. 이 전환이 어떻게 일어나는지가 빠져있다."

### Developer (Amelia) -- Phase 구분 없는 자동 수행 목록

"'자동 수행' 리스트에 **Phase 구분이 없다**. '에이전트 메모리 자동 학습 + 자동 주입'은 Phase 2 기능인데, Phase 1 자동 수행 목록에 함께 있다. 이러면 구현 시 '이거 Phase 1에서 해야 되나?'라는 혼란이 생긴다."

"'보고서 작전일지 자동 저장'도 작전일지가 Phase 2인데 자동 수행에 포함. Phase 1에서는 어디에 저장되는가? 통신로그에만? 명확히 해야 한다."

"**또 하나 -- '입력 방식: 키보드 + 마우스 (터치 미지원)'이라고 했는데**, 이건 Platform Strategy에서 중요한 결정이다. 하지만 '터치 미지원'의 의미가 불명확하다. 태블릿 브라우저에서 접속하면? CSS hover 상태에 의존하는 인터랙션이 있으면 터치에서 깨진다. '터치 미지원'이 '터치 테스트 안 함'인지 '터치 기기에서 접속 차단'인지 다르다."

### QA (Quinn) -- First-Time Success 경로의 비현실성

"First-Time Success 경로가 10분이라고 했는데, **현실적인가 스트레스 테스트**하겠다."

"1. '회원가입 (1분)' -- 이메일 인증 포함? 소셜 로그인? 1분은 이메일 입력 + 비밀번호 설정 수준이다. 이메일 인증이 있으면 5분 이상."
"2. '조직 템플릿 선택 (1분)' -- 4개 카드 중 선택이 1분이면 괜찮다. 하지만 **API 키 입력은 어디서 하나?** LLM을 호출하려면 Anthropic/OpenAI API 키가 필요한데, 이 단계가 빠져있다. API 키 없으면 첫 명령이 실패한다."
"3. '위임 체인 실시간 관찰 (2~3분)' -- LLM API가 느리거나 429 에러가 나면? 5분 이상 걸릴 수 있다."

"**API 키 입력 단계가 First-Time Success 경로에 없다.** 이건 심각한 누락이다. 비개발자에게 'API 키를 입력하세요'라고 하면 '그게 뭔데?' 반응이 나온다. 이 마찰을 어떻게 처리할 건지가 전혀 없다."

### Business Analyst (Mary) -- Make-or-Break의 페르소나 편향

"Critical Success Moments가 잘 정의됐다. Aha! 순간 4개, Make-or-Break 5개. 하지만 **전부 김대표(CEO) 관점**이다."

"**박과장의 Make-or-Break가 없다:**"
"- '첫 조직 설계' 성공/실패: 관리자 콘솔에서 부서 만들고 에이전트 배치하는 과정이 10분 이내 완료되지 않으면 → '이거 너무 복잡해' → 이탈"
"- '팀원 초대 후 사용' 성공/실패: 팀원이 워크스페이스에 접속해서 첫 명령을 성공하지 못하면 → '이거 쓸 수 있는 거 맞아?' → 신뢰 상실"

"**이사장의 Make-or-Break도 약하다:**"
"- '첫 자동매매 설정' 성공/실패: 모의거래로 첫 자동매매를 테스트하는 과정이 직관적이지 않으면 → '위험해서 못 쓰겠다' → 전략실 미사용"
"- Make-or-Break 5번 '실거래 주문'만 있는데, 이건 방어 설계이지 성공 순간이 아니다"

"John의 지적에 동의한다. 부차 경험이라고 표현한 것도, Aha! 순간이 CEO 편향인 것도 같은 문제다 -- **이 문서가 무의식적으로 김대표 중심으로 쓰여졌다**."

### Scrum Master (Bob) -- Experience Principles의 중복성

"EP1~EP5가 추가됐는데, **DP1~DP5와 거의 1:1 매핑**이다:"

```
EP1 (One Command) ≈ DP1(조직 은유) + DP2(위임 투명성)
EP2 (Show Progress) ≈ DP2(위임 투명성)
EP3 (Safe to Explore) ≈ DP3(안전한 변경) + DP4(점진적 복잡성)
EP4 (Cost Visible) ≈ DP5(비용 가시성)
EP5 (Hide Complexity) ≈ DP4(점진적 복잡성) + DP1(조직 은유)
```

"**DP와 EP의 차이가 무엇인가?** DP가 '왜(Why)', EP가 '어떻게 느끼는가(How it feels)'라면, 그 차이가 명확하지 않다. EP2 '진행 과정이 보인다'와 DP2 '위임 투명성'은 같은 말을 다르게 표현한 것이다."

"**두 가지 옵션:**"
"1. EP를 제거하고 DP만 유지 (중복 제거)"
"2. EP를 DP와 명확히 구분 -- DP는 '설계 원칙(구현자용)', EP는 '경험 기준(검증용)'. EP는 측정 가능한 기준을 포함해야 한다. 예: EP2 '명령 처리 중 5초 이상 화면 변화 없는 순간이 0이어야 한다'"

"나는 옵션 2를 추천한다. EP에 **측정 가능한 기준**을 추가하면 DP와 확실히 구분된다."

## Expert Disagreements

1. **John vs Winston (핵심 루프 개수)**: John은 동적 조직 관리를 2차 핵심으로 격상해야 한다고 주장. Winston은 빈도 기준으로 핵심 루프 1개가 맞지만 '부차'라는 표현이 약하다고 동의. **절충안: '부차 경험 루프' -> '보조 핵심 루프(Secondary Core Loop)'로 격상 + 핵심 루프가 1개인 이유 명시.**

2. **Sally vs Amelia (대기 경험 깊이)**: Sally는 위임 체인 대기 중 세부 상태(딥워크 5단계) 표시가 필요하다고 주장. Amelia는 Discovery 수준에서 결정할 건 아니고 Wireframe에서 하자고 주장. **Sally 승: Core Experience가 바로 이 대기 경험의 '느낌'을 정의하는 단계. 세부 UI가 아니라 '5초 이상 변화 없는 순간이 없어야 한다'는 원칙 수준에서 정의 가능.**

3. **Bob vs 전체 (EP 존재 가치)**: Bob은 EP가 DP의 중복이라고 지적. 나머지 전문가는 EP의 존재 자체는 필요하되 차별화가 약하다고 동의. **절충안: EP에 측정 가능한 기준을 추가하여 DP와 역할 분리.**

## Round 1 Verdict

**Status: ACTION REQUIRED**

Major 2건, Medium 4건, Minor 3건. 핵심 루프 정의와 Experience Principles는 잘 구성되었으나, (1) API 키 온보딩 마찰 누락, (2) 대기 시간 경험 미정의가 가장 중요한 빈틈이다.

**Action Items:**
1. First-Time Success 경로에 **API 키 입력 단계** 추가 + 비개발자 마찰 해결 전략 (체험 모드? 관리자가 미리 설정? 가이드 비디오?)
2. 위임 체인 대기 경험 상세화: 에이전트 **딥워크 5단계 상태 표시**, 업데이트 최소 빈도, 부분 결과 먼저 표시 전략
3. 동적 조직 관리를 '보조 핵심 루프'로 격상 + 박과장/이사장의 Aha!/Make-or-Break 추가
4. EP1~EP5에 **측정 가능한 기준** 추가하여 DP와 차별화
5. '자동 수행' 목록에 Phase 1 / Phase 2 구분 추가
6. '3패널 레이아웃'을 '3영역 구성'으로 수정 (레이아웃 미결정 반영)
