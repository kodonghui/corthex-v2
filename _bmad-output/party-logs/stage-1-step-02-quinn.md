### Issues Found
1. **[D3 Accuracy]** Embedding provider and dimension conflict. Document treats Gemini 768d as current infrastructure (code-verified correct: `embedding-service.ts` imports `@google/generative-ai`, `schema.ts:1556` uses `vector(768)`). But Brief mandates Voyage AI `voyage-3` (1024d) and CLAUDE.md states "Gemini API 사용 금지." ALL v3 vector schemas (observations, agent_memories 0064, HNSW indexes) reference 768 dimensions. If Voyage AI is the v3 target, every schema reference is wrong, and existing knowledge_docs + semantic_cache tables need re-embedding (dimensional change). This is not a "migration detail" — it is a **fundamental architecture decision** that changes every schema in this document. — **Critical**
2. **[D6 Risk Awareness]** n8n OOM kill causes irrecoverable data loss. Web-verified: in-flight workflow executions are NOT recovered after Docker OOM kill. PostgreSQL can enter recovery state if killed simultaneously. Known n8n memory leak (GitHub #16980) makes OOM kill not a theoretical risk but an expected operational event over time. The document's mitigation (`--memory=4g` + `restart: unless-stopped`) handles restart but NOT execution recovery or data integrity. — **Critical**
3. **[D6 Risk Awareness]** DNS rebinding attack on n8n `127.0.0.1:5678`. Web-verified: DNS CNAME-based rebinding can bypass IP address filters and reach localhost services. Chrome allows local network access from non-secure contexts until May 2026. n8n's `/healthz` endpoint has no authentication. An attacker could enumerate workflows, read execution data, or trigger webhook endpoints via a malicious website visited by any VPS user. Mitigation: n8n must have `X-N8N-API-KEY` required on ALL endpoints (not just management API), or Hono proxy must be the ONLY process that can reach port 5678 (e.g., iptables rules restricting by UID). — **Major**
4. **[D5 Consistency]** UXUI tooling direction fundamentally confused. Line 435: "Google Stitch deprecated from pipeline. Subframe is sole UXUI tool." But MEMORY.md states: "Stitch 2(메인) + Subframe(폐기)." These are opposite statements. CLAUDE.md's UXUI section also references "Stitch MCP가 생성한 HTML = 디자인 기준." The research document has the tooling direction backwards. — **Major**
5. **[D5 Consistency]** Observations table schema inconsistency within the same document. Step 2 SQL (lines 340-354): `task_execution_id UUID`, `observed_at TIMESTAMP`, `reflected BOOLEAN`. Step 3 Drizzle (lines 783-801): `source VARCHAR(50) NOT NULL`, `embedding VECTOR(768)`, `isProcessed BOOLEAN`, `processedAt TIMESTAMP`. Different column names, different nullable semantics, different fields entirely. Two competing schemas for the same table. — **Major**
6. **[D2 Completeness]** n8n `NODE_OPTIONS=--max-old-space-size` interaction with Docker `--memory` not analyzed. Line 244 specifies both `--memory=4g` and `NODE_OPTIONS=--max-old-space-size=4096`. But `max-old-space-size=4096` tells V8 it can use 4GB of heap. Combined with V8 overhead, RSS, and OS buffers inside the container, total memory easily exceeds the 4GB Docker limit. The max-old-space-size should be set BELOW the Docker limit (e.g., 3072 for a 4GB container) to leave room for non-heap memory. — **Major**
7. **[D2 Completeness]** Missing HNSW index memory attribution. Step 2 never states WHERE HNSW indexes execute. The VPS co-residence table (lines 233-242) does not include HNSW memory. Web-verified: Neon manages HNSW indexes on its compute nodes, not on the VPS. This is CORRECT behavior, but the document should explicitly state "HNSW indexes: managed by Neon compute, zero VPS RAM impact" to prevent misallocation during Sprint planning. Step 5 later corrects this (line 1637) but Step 2 is the primary resource analysis. — **Minor**
8. **[D3 Accuracy]** Neon plan name wrong. Document references "Neon Pro" at various points. Web-verified: Neon current plans are Free/Launch/Scale. The $19/month price point corresponds to the Launch plan, not "Pro." — **Minor**
9. **[D2 Completeness]** No n8n Docker security hardening. Missing: `--security-opt=no-new-privileges`, `--cap-drop=ALL`, `--read-only` filesystem (except data volume), Docker user namespace remapping. These are standard container hardening practices for services that process external webhook data. — **Minor**
