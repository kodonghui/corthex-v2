## Deep Dive Analysis (80%)

### PM (John) -- Priority Matrix의 숨겨진 모순

"Priority Matrix가 추가된 건 좋다. 그런데 **실제로 이 매트릭스를 따르면 모순이 발생**한다. '동적 조직 관리'가 P0이고 김대표/박과장 '핵심'인데, 김대표의 멘탈 모델은 '나는 CEO, AI는 내 회사 직원'이다. CEO는 **조직을 만드는 게 아니라 일을 시키는 게 먼저**다. 그런데 P0에서 사령관실과 조직 관리를 동시에 핵심으로 놨다."

"질문: **김대표가 처음 접속했을 때 조직을 먼저 만들어야 하나, 명령을 먼저 내려야 하나?** 조직 템플릿으로 해결한다고 했는데, 그러면 '동적 조직 관리'는 P0이 아니라 온보딩의 일부일 뿐이다. **실제 김대표의 '동적 조직 변경'은 주간/월간 태스크**라고 Core Task Analysis에서 본인이 명시했다. P0 핵심이라는 라벨과 실제 사용 빈도가 충돌한다."

### Architect (Winston) -- Screen Inventory의 기술적 빈틈

"Screen Inventory가 잘 정리됐다. 하지만 **WebSocket 채널 매핑에 불일치**가 있다. 화면 #1 작전현황에 'WS: cost, agent-status'라고 했는데, architecture.md의 7채널은 agent-status, task-progress, cost-update, org-change, debate-stream, sketchvibe-stream, chat이다. **화면 #2 사령관실이 'WS: command, delegation, tool'이라고 했는데, 이 3개는 architecture의 7채널 어디에도 없다.** 'task-progress' 채널이 위임 체인을 담당하는 건지, 별도 채널이 필요한 건지 명확하지 않다."

"또한 **'사내 메신저'가 화면 인벤토리에 없다**. Phase 3이니까 뺐을 수 있지만, Priority Matrix에는 박과장 '핵심'으로 표시되어 있다. 인벤토리에 Phase 3 화면도 포함하거나, 매트릭스에서 '계획됨' 표시를 해야 한다."

### UX Designer (Sally) -- 멘탈 모델이 디자인을 구동하지 않는다

"멘탈 모델 3개가 추가됐는데, **이것이 실제 디자인 결정에 어떻게 연결되는지가 빠져있다**. 예를 들어 김대표의 멘탈 모델이 '사령관실 = 사무실'이라면, 사령관실 UI는 채팅형이 아니라 '사무실' 은유를 사용해야 하는 건 아닌가? 현재 핵심 UI 패턴은 '채팅형 입력'인데, 이건 ChatGPT와 같은 패턴이다."

"**멘탈 모델에서 도출되어야 하는 핵심 디자인 원칙**이 빠졌다. 예를 들어:"
"- '사무실' 은유 -> 사령관실에 에이전트 아바타/상태가 상시 보여야 한다 (빈 채팅 창이 아니라)"
"- '급여 관리' 은유 -> 비용은 에이전트 프로필에서 바로 접근 가능해야 한다"
"- 'HTS' 은유 -> 전략실은 차트 중심 레이아웃이어야 한다"

"John의 지적에 추가하자면 -- **Core Task Analysis의 '상호작용 패턴'이 너무 추상적**이다. '텍스트 입력 -> 실시간 위임 체인 관찰 -> 보고서 읽기 -> 피드백'은 행위 수준이지 **인터랙션 디자인 수준이 아니다**. 실제 UX에서 중요한 것은: 위임 체인을 보고서 옆에 보여줄까 위에 보여줄까? 보고서가 스트리밍으로 나올 때 스크롤은 자동인가? 피드백 버튼은 보고서 끝에만 있나, 플로팅인가?"

### Developer (Amelia) -- 구현 비용 경고

"네비게이션 모델에서 CEO 앱 사이드바에 **13개 메뉴 항목**이 있다. 이 중 Phase 2 기능이 8개다. **Phase 1 출시 시 사이드바의 60%가 비활성/숨김 상태**가 된다. L1/L2/L3 progressive disclosure를 정의했지만, **Phase 1에서 사이드바가 5개 메뉴만 있으면 '텅 빈 제품' 인상**을 줄 수 있다."

"Screen Inventory에서 AGORA 토론(#5)의 UI 패턴이 '라운드 타임라인 뷰 + 에이전트 아바타 + 발언 카드 스트림 + Diff 뷰'인데, **v1 feature spec을 보면 토론은 사령관실 내에서 /토론 명령으로 시작**한다. 별도 화면인가, 사령관실의 서브뷰인가? 화면 인벤토리에서 독립 화면(#5)으로 잡았는데, 네비게이션에서는 '소통' 그룹 하위다. **사령관실에서 /토론 입력 시 AGORA 화면으로 라우팅되는 건지, 사령관실 내 인라인 뷰인지** 명확하지 않다."

"또한 **자동화/워크플로우(#14)가 모호하다**. '사령관실 내 서브뷰 또는 별도 화면'이라고 양쪽을 열어뒀다. Discovery에서 결정하지 않으면 나중에 라우팅 충돌이 생긴다."

### QA (Quinn) -- 화면 간 이동 경로의 누락

"화면 간 이동 경로가 7개 정의됐는데, **명백한 누락**이 있다:"

"1. **에러 상태 경로 없음**: 에이전트가 에러 상태일 때 사용자가 어디서 대응하나? 작전현황에서 에러 알림 -> ??? -> 에이전트 관리에서 재시작? 이 경로가 없다."
"2. **품질 게이트 반려 경로 부분적**: 통신로그 QA 탭에서 반려 확인까지는 있는데, **반려된 보고서를 어디서 다시 보나?** 사령관실로 돌아가야 하나, 작전일지에서 찾아야 하나?"
"3. **관리자 콘솔 -> CEO 앱 전환 경로 없음**: 박과장이 관리자 콘솔에서 조직 변경 후 CEO 앱에서 확인하고 싶을 때 어떻게 전환하나? 별도 URL? 통합 네비게이션?"

"**비정기 태스크 '부서 해체/재편'의 상호작용이 가장 위험**하다. cascade 경고 -> 선택 -> 에이전트 재배치라고 했는데, 실제로 부서 삭제 시: (1) 진행 중 작업 목록 표시, (2) 각 작업의 처리 방법 선택(대기/강제종료), (3) 하위 에이전트 재배치 대상 부서 선택, (4) 메모리/비용 아카이브 확인 -- 이 4단계가 모달 하나로 되는가, 위저드인가? **UX에서 가장 복잡한 플로우인데 한 줄로 처리됐다.**"

### Business Analyst (Mary) -- 온보딩 플로우의 결정적 빈틈

"온보딩 마찰 포인트 6개가 추가됐는데, **가장 중요한 마찰이 빠졌다**: '무료 체험이 있나? 가격이 얼마인가?' 비즈니스 모델 관련 마찰이다. 물론 UX 스펙의 범위가 아닐 수 있지만, **가격 정보/플랜 선택 화면이 화면 인벤토리에 없다**. SaaS B2B 플랫폼인데 가격 관련 화면이 0이다."

"Amelia의 지적에 동의한다. Phase 1에서 사이드바가 5개면 문제다. 그런데 **나는 다른 관점으로 본다 -- CEO 앱과 관리자 콘솔이 같은 사용자(김대표)가 쓸 수 있는데, 두 앱을 왜 분리했는지에 대한 UX 근거가 없다**. 김대표의 멘탈 모델에서 '조직을 만든다'와 '일을 시킨다'는 같은 맥락인데, 앱이 분리되면 컨텍스트 전환 비용이 생긴다. Admin 콘솔 분리는 아키텍처 결정이지 UX 결정이 아닌 것 같은데, **UX 관점에서 이 분리를 정당화하는 논거**가 있어야 한다."

"Quinn이 지적한 admin -> app 전환 경로 부재가 바로 이 문제의 증거다."

### Scrum Master (Bob) -- 스코프와 일관성 점검

"전체적으로 Round 1 대비 극적으로 개선됐다. 하지만 **일관성 문제**가 몇 개 있다:"

"1. **Phase 표기 불일치**: Priority Matrix에서 '품질 게이트'가 'P0/P1'이고, v1 매핑 테이블에서도 'P0/P1'인데 -- P0인가 P1인가? 비서실장 5항목 체크가 P0이고 quality_rules.yaml 자동 검수가 P1이라면, **화면 인벤토리에서는 어디에 속하나?** 품질 게이트 전용 화면이 없고 통신로그 QA 탭에 포함된 것 같은데, QA 탭이 P1 통신로그에 속해있다."

"2. **화면 수 불일치**: Screen Inventory에서 CEO 앱 13개 화면이지만, 네비게이션 모델에서는 **'설정/프로필'이 하단에 있는데 화면 인벤토리에 없다**. 또 '사내 메신저'(Phase 3)도 네비게이션에 없다."

"3. Sally의 지적에 강하게 동의한다 -- **멘탈 모델 -> 디자인 원칙 -> 구체적 UI 결정의 연결 고리**가 끊어져 있다. 멘탈 모델이 '있는' 것만으로는 부족하고, 그것이 왜 특정 UI 패턴을 선택하게 하는지가 명시되어야 한다."

"4. **Design Opportunities #6 '사령관실 명령 인텔리전스'와 온보딩 마찰 포인트 '명령어를 뭐라고 치지?'가 같은 문제를 다른 곳에서 중복 다루고 있다.** 통합하거나 상호 참조가 필요하다."

## Expert Disagreements

1. **John vs 전체 (조직 관리 P0 적절성)**: John은 김대표에게 동적 조직 관리는 주간 태스크이므로 P0 '핵심'이 과하다고 주장. **반론(Winston)**: v2의 핵심 차별점이 동적 조직이므로 P0이 맞지만, '핵심' 대신 '필수'로 구분해야 한다. 핵심 = 매일 쓰는 것, 필수 = 있어야 하지만 매일은 아닌 것.

2. **Sally vs Amelia (사령관실 AGORA 관계)**: Sally는 토론이 사령관실의 '확장 모드'여야 멘탈 모델과 일치한다고 주장. Amelia는 SSE 스트리밍과 복잡한 라운드 UI를 고려하면 별도 화면이 구현에 유리하다고 주장. **절충안**: 사령관실에서 /토론 입력 시 AGORA 화면으로 자동 전환하되, 보고서 결과는 사령관실에 돌아와서 표시.

3. **Mary vs Bob (앱 분리 정당성)**: Mary는 UX 관점에서 앱 분리가 비효율이라고 주장. Bob은 아키텍처가 이미 결정되었으므로 UX는 전환 경로를 매끄럽게 하는 데 집중해야 한다고 주장. **Quinn 추가**: 같은 도메인(corthex.com)의 서브 경로(/app, /admin)라면 실제 사용자 경험은 '전환'이 아니라 '탭 전환' 수준일 수 있다.

## Devil's Advocate (Winston)

"모든 것이 잘 정리된 것 같지만, **근본적 리스크 하나를 짚겠다**: 이 UX 스펙은 22개 기능 x 21개 화면을 나열했다. 하지만 **실제로 이 모든 것을 구현할 개발 리소스가 있는가?** Phase 1만 해도 CEO 앱 5개 화면 + 관리자 콘솔 8개 화면 = 13개 화면이다. 각 화면에 실시간 WebSocket, 복잡한 인터랙션(드래그 앤 드롭, 캔버스, 차트)이 있다."

"UX 스펙이 아무리 좋아도, **Phase 1에서 '최소한 이것만 있으면 작동한다'는 화면 3~5개의 MVP 스크린 세트**를 별도로 정의해야 한다. 지금은 P0/P1 구분은 있지만, P0 안에서도 '절대 최소'가 뭔지 불명확하다."
