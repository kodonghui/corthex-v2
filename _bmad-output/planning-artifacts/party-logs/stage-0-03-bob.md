**Issue 1 — [D6 리스크] 온보딩 강제 메커니즘 미명시 (Priority: HIGH)**

line 205: "v3 온보딩 UX는 이 플로우를 강제해야 한다" — 요구사항 선언은 됐지만 어떻게 강제하는지 없다. v2에서 이 문제를 겪었다면 v3에서 구체적인 게이트 메커니즘이 있어야 한다:
- **옵션 A**: CEO 앱 첫 진입 시 "Admin 설정 미완료" 빈 상태 + Admin 앱으로 유도 배너
- **옵션 B**: Admin 설정 완료율 진행 바 (0/5단계 → CEO 초대 버튼 활성화)
- **옵션 C**: CEO 계정 생성 시 Admin이 직접 초대 → 초대 없이 CEO 앱 접근 불가 (인증 레벨 차단)

세 가지 중 어느 메커니즘을 택할지 Brief에 명시해야 한다. 미명시 시 개발자가 임의 결정 → v2 반복 가능성.

**Issue 2 — [D4 실행가능성] Admin 앱 페이지 수 v2 기준 미표기 (Priority: MEDIUM)**

"Admin 앱 (27개 페이지)"가 v2 기준인지 v3 완료 기준인지 불명확. v3에서 Admin에 추가되는 페이지:
- n8n 관리 페이지 (신규)
- 메모리 설정 페이지 (Reflection 한도 관리, 신규)
- /office 관리 설정 (잠재적 신규)

"27개 페이지(v2 기준)"로 표기하거나, "27+3개 예정"으로 v3 범위를 명시해야 스프린트 계획 시 Admin 앱 scope를 정확히 잡을 수 있다. 현재 표현은 v3 Admin 개발 범위를 과소 표현.
